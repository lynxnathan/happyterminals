# Multi-strategy shading ramps with per-model selection

## Idea

Right now `ShadingRamp` has a single configurable `&[char]` ramp + `light_dir`,
and `run_scene()` hardcodes `ShadingRamp::default()`. Once we have multiple
models loaded (Phase 2.1+), each model may want a different visual style.

User feedback (2026-04-15): "voxcii is better [as default] (but we will improve
on it later, adopt multi-strategies / let the user select per model which
shading ramp to use, etc)"

## Future direction

1. **Built-in ramp library** — ship a curated set of named ramps:
   - `voxcii` (current default): `.,':;!+*=#$@`
   - `dotted`: `·.:;+oO●█`
   - `blocks`: `░▒▓█`
   - `braille-density`: `⠀⠁⠉⠋⠛⠟⠿⡿⣿`
   - `classic-ascii`: ` .:-=+*#%@`
   - User-supplied custom ramps still supported

2. **Per-model ramp binding** — scene DSL accepts a ramp per node:
   ```rust
   .layer(0, |l| l
       .cube().rotation(&r)              // uses scene-default ramp
       .obj("bunny.obj").shading(Ramp::Blocks))   // override per-model
   ```

3. **Multi-strategy shading models** — beyond flat per-face NdotL:
   - Gouraud (per-vertex interpolation)
   - Phong (per-pixel + specular highlight)
   - Toon/cel shading (banded ramp with sharp transitions)
   - Edge detection overlay (silhouette emphasis)

## Why this is parked

- Phase 2.1 needs to land first (OBJ loading + Mesh abstraction)
- Single-ramp default is sufficient for M1 spinning cube
- Phase 2.x will reveal what shading flexibility actually matters in practice

## When to revisit

Surface this when:
- Phase 2.1 is being planned (mesh API exists → per-mesh attributes become natural)
- OR user feedback on the bunny/cow/teapot demos asks for visual variety
- OR before M3 (compositor) since shading-per-layer composes with effects
