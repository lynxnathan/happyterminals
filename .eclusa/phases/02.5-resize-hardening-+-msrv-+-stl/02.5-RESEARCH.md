**Researched:** 2026-04-16

**Confidence:** HIGH

## Summary

Phase 2.5 has three orthogonal workstreams: (1) harden the resize path in `run.rs` to prevent mid-frame rendering with stale dimensions, (2) add STL mesh loading via `stl_io 0.11` to the renderer crate, and (3) align MSRV policy + CI with reality.

**Critical MSRV finding:** The CONTEXT.md locks MSRV at 1.86, but the current dependency graph makes 1.86 physically impossible. `reactive_graph 0.2.13`, `darling 0.23.0`, `instability 0.3.12`, and `time 0.3.47` all require `rustc >= 1.88`. The workspace `Cargo.toml` already declares `rust-version = "1.88"`, `rust-toolchain.toml` pins `channel = "1.88"`, and the existing CI YAML at `.github/workflows/ci.yml` references `"1.86"` (which would fail). The MSRV must be pinned at **1.88**, not 1.86. The CI already exists but needs its toolchain references updated from `"1.86"` to `"1.88"`.

The STL loader is straightforward: `stl_io::read_stl` returns an `IndexedMesh` with `vertices: Vec<Vertex>` (where `Vertex = [f32; 3]`) and `faces: Vec<IndexedTriangle>` (where each face has `normal: Normal` and `vertices: [usize; 3]`). Conversion to the existing `Mesh` struct is a direct field mapping. STL files are always triangles, so no triangulation is needed.

**Primary recommendation:** Fix the MSRV to 1.88 (matching reality), update CI references, harden the resize event path with a skip-frame flag, and add `load_stl` following the exact `load_obj` pattern.

## Architecture Patterns

### Resize Hardening Pattern

**What:** Prevent the rasterizer from seeing inconsistent dimensions during a frame tick where a resize event arrived between tick start and render.

**Current behavior (run.rs lines 78-121):** The `tokio::select!` loop has two arms: tick and event. When a resize event arrives, `input_signals.terminal_size` is updated and the grid resizes on the next tick via `frame.area()`. The vulnerability: if resize events arrive *during* the event arm processing and a tick fires immediately after, the grid dimensions in `frame.area()` may differ from what the renderer's z-buffer expects.

**Recommended hardening (matching CONTEXT.md decision):**

1. Add a `resize_pending: bool` flag outside the loop.
2. In the `Resize` event handler, set `resize_pending = true`.
3. In the tick arm, if `resize_pending` is true: call `grid.resize(frame.area())`, set `resize_pending = false`, and **skip** the render callback for that frame. The next tick renders at the correct size.
4. This applies to all three `run` variants: `run()`, `run_with_input()`, `run_scene()`.

**Why skip instead of render:** `Grid::resize()` creates a fresh `Buffer::empty()`. The Renderer's z-buffer dimensions from the previous frame would be stale. Skipping one frame (16-33ms) is imperceptible; rendering with mismatched dimensions causes panics or visual corruption.

### STL Loader Pattern

**What:** `load_stl` follows the exact same contract as `load_obj`.

**stl_io API mapping:**
```rust
// stl_io types:
// IndexedMesh { vertices: Vec<Vertex>, faces: Vec<IndexedTriangle> }
// Vertex = Vector<f32> where Vector(pub [f32; 3])
// IndexedTriangle { normal: Normal, vertices: [usize; 3] }
// Normal = Vector<f32>

// read_stl signature:
// pub fn read_stl<R: Read + Seek>(read: &mut R) -> Result<IndexedMesh>
```

**Conversion to `Mesh`:**
```rust
// Source: stl_io 0.11 docs.rs + existing mesh.rs patterns
pub fn load_stl<P: AsRef<Path>>(path: P) -> Result<(Mesh, LoadStats), MeshError> {
    let mut file = File::open(path.as_ref())?;  // MeshError::Io via From
    let stl_mesh = stl_io::read_stl(&mut file)
        .map_err(|e| MeshError::StlParse(format!("{e}")))?;

    // DoS guard
    if stl_mesh.faces.len() > MAX_TRIANGLES {
        return Err(MeshError::Parse(format!(
            "mesh exceeds MAX_TRIANGLES ({MAX_TRIANGLES})"
        )));
    }

    // Convert vertices: stl_io Vertex is [f32; 3]
    let vertices: Vec<Vec3> = stl_mesh.vertices
        .iter()
        .map(|v| Vec3::new(v[0], v[1], v[2]))
        .collect();

    let mut stats = LoadStats::default();
    let mut kept_indices = Vec::with_capacity(stl_mesh.faces.len());
    let mut normals = Vec::with_capacity(stl_mesh.faces.len());

    for (i, face) in stl_mesh.faces.iter().enumerate() {
        let [a, b, c] = face.vertices;
        if a >= vertices.len() || b >= vertices.len() || c >= vertices.len() {
            stats.triangles_skipped += 1;
            // push warning...
            continue;
        }

        // Use STL normal directly (per CONTEXT decision)
        let stl_normal = Vec3::new(face.normal[0], face.normal[1], face.normal[2]);
        let normal = if stl_normal.is_finite() && stl_normal.length() > 1e-12 {
            stl_normal.normalize()
        } else {
            // Fallback: recompute flat normal (degenerate STL normal)
            match flat_normal(vertices[a], vertices[b], vertices[c]) {
                Some(n) => n,
                None => {
                    stats.triangles_skipped += 1;
                    // push warning...
                    continue;
                }
            }
        };

        let a_u32 = a as u32;
        let b_u32 = b as u32;
        let c_u32 = c as u32;
        kept_indices.push([a_u32, b_u32, c_u32]);
        normals.push(normal);
        stats.triangles_loaded += 1;
    }

    Ok((Mesh { vertices, indices: kept_indices, normals, shading: None }, stats))
}
```

### Anti-Patterns to Avoid
- **Re-reading `stl_io::read_stl` return as Triangle vec:** Use `IndexedMesh` directly (already indexed). Converting to flat triangles and back wastes memory.
- **Ignoring STL normals:** STL normals are authoritative per the format spec. Only fall back to recomputation when the stored normal is degenerate (zero-length or NaN).
- **Rendering during resize:** Never call `render_fn` in the same tick that processes a resize event. The z-buffer and grid dimensions would be inconsistent.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| STL parsing (binary + ASCII) | Custom binary/ASCII parser | `stl_io::read_stl` | Binary STL has 80-byte header, per-face attribute bytes, endianness; ASCII has whitespace variations |
| Resize event detection | Custom terminal size polling | crossterm `Event::Resize` | Already wired; polling is wasteful and racy |
| Error enum boilerplate | Manual `impl Display + Error` | `thiserror` derive | Already used; consistent with existing `MeshError` |

## Common Pitfalls

### Pitfall 1: MSRV Mismatch with Dependency Graph
**What goes wrong:** CONTEXT.md specifies Rust 1.86 but `reactive_graph 0.2.13`, `darling 0.23`, `instability 0.3.12`, `time 0.3.47` all require 1.88+. Building with `cargo +1.86 build` fails immediately.
**Why it happens:** Dependencies have raised their MSRV since the project's original planning.
**How to avoid:** Pin MSRV at 1.88 (current `rust-toolchain.toml` value). Update CI YAML from `"1.86"` to `"1.88"`. Update REL-03 docs to state floor = 1.88.
**Warning signs:** `cargo +1.86 build --workspace` fails with "rustc X is not supported by the following packages".

### Pitfall 2: stl_io Index Type Mismatch
**What goes wrong:** `stl_io::IndexedTriangle.vertices` uses `[usize; 3]` but `Mesh.indices` uses `[u32; 3]`. Truncation on large meshes.
**Why it happens:** stl_io uses `usize` for safety; Mesh uses `u32` for GPU-friendly density.
**How to avoid:** Use `u32::try_from(idx)` with a `MeshError::Parse` fallback for overflow (same pattern as `load_obj`'s index rebasing).
**Warning signs:** Meshes with > 4B vertices (astronomically unlikely for terminal rendering, but the guard should exist).

### Pitfall 3: Degenerate STL Normals
**What goes wrong:** Some STL exporters write `[0, 0, 0]` as the normal for every face (relying on viewers to recompute). Using the stored normal directly produces incorrect shading (everything dark).
**Why it happens:** The STL spec says normals "should" be present but many tools ignore this.
**How to avoid:** Check `stl_normal.length() > threshold` before using; fall back to `flat_normal()` recomputation for degenerate/zero normals.
**Warning signs:** All faces shade identically (uniform darkness).

### Pitfall 4: Resize Race in tokio::select
**What goes wrong:** `tokio::select!` is fair -- either arm can win. A resize event and tick can interleave such that the tick reads `frame.area()` with new dimensions but the Renderer's z-buffer was sized for old dimensions.
**Why it happens:** The current code resizes the grid inside `terminal.draw()` via `frame.area()`, but the Renderer's internal buffers were sized from the previous frame.
**How to avoid:** The Renderer already handles this gracefully (it re-sizes its buffers when `w != self.last_width || h != self.last_height`). The real risk is `grid.resize()` creating a new `Buffer::empty()` that is cloned at the end of the draw closure -- the clone is fine but skipping one frame avoids any visual glitch.
**Warning signs:** Brief flash of corrupted output on rapid resize (hard to reproduce outside Windows Terminal which batches resize events differently).

### Pitfall 5: stl_io Read + Seek Requirement
**What goes wrong:** `read_stl` requires `Read + Seek` (it peeks at the header to detect ASCII vs binary). `BufReader<File>` implements both, but `stdin()` does not.
**Why it happens:** Auto-detection needs to seek back after reading the header.
**How to avoid:** Always open as `File::open()` (which is `Read + Seek`). No `BufReader` wrapper needed -- `File` is already buffered by the OS.

### Resize Hardening Skeleton (for run.rs)
```rust
// Add before the loop in each run variant:
let mut resize_pending = false;

// In the event arm, Resize handler:
InputEvent::Resize { width, height } => {
    input_signals.terminal_size.set((*width, *height));
    resize_pending = true;
}

// In the tick arm, before calling render_fn:
_ = tick.tick() => {
    let _span = tracing::trace_span!("frame").entered();
    terminal.draw(|frame| {
        grid.resize(frame.area());
        if resize_pending {
            resize_pending = false;
            // Skip render this frame -- grid just resized, next tick renders clean
            let mut out = grid.deref().clone();
            downsample(&mut out, color_mode);
            *frame.buffer_mut() = out;
            return;
        }
        render_fn(&mut grid, &input_signals);
        let mut out = grid.deref().clone();
        downsample(&mut out, color_mode);
        *frame.buffer_mut() = out;
    })?;
}
```
