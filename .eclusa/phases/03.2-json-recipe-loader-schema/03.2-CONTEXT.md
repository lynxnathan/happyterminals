# Phase 3.2: JSON Recipe Loader + Schema - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning
**Mode:** Smart discuss (autonomous)

<domain>
## Phase Boundary

Build a JSON recipe loader in `happyterminals-dsl` that reads a JSON scene file and produces a `SceneIr` identical to the Rust builder. Schema generated via `schemars`, validated via `jsonschema` before loading. Versioned `$version` field. Covers DSL-04, DSL-06, DSL-07.

</domain>

<decisions>
## Implementation Decisions

### JSON Shape
- **Root object:** `{ "$version": "1.0", "camera": {...}, "layers": [...] }`. Mirrors the Rust builder's `.camera()` + `.layer()` shape.
- **Camera:** `{ "type": "orbit", "azimuth": 0.3, "elevation": 0.2, "distance": 5.0, "target": [0, 0, 0] }`. Also `"freelook"` and `"fps"` types.
- **Layer:** `{ "z_order": 0, "children": [{ "type": "cube" }, { "type": "mesh", "path": "bunny.obj" }] }`.
- **Transform:** Optional on any node: `{ "position": [0, 1, 0], "rotation": [0, 0, 0], "scale": [1, 1, 1] }`.
- **Props:** `{ "props": { "key": "value" } }` — string-keyed, JSON value. Mapped to `PropValue` at load time.

### Schema Generation (DSL-06)
- **Crate:** `schemars 1.x` (workspace dep, already pinned). Derive `JsonSchema` on the recipe structs.
- **Validation:** `jsonschema` crate validates the JSON against the generated schema BEFORE deserialization. Errors include path + description.
- **Output:** `RecipeSchema::generate() -> serde_json::Value` returns the JSON Schema document. Users can export it for LLM prompting.

### Versioned Schema (DSL-07)
- **Field:** `"$version": "1.0"` required at root. Loader checks version and returns clear error for unsupported versions.
- **No migration system** — first breaking change is the trigger to design migrations. For now, only version "1.0" is supported.

### Crate Location
- **`happyterminals-dsl`** — the JSON loader lives alongside the Rust builder DSL. New module: `json.rs`. Depends on `serde`, `serde_json`, `schemars`, `jsonschema`.
- **Public API:** `pub fn load_recipe(json: &str) -> Result<SceneIr, RecipeError>` + `pub fn recipe_schema() -> serde_json::Value`.

### Round-Trip Property Test (DSL-04)
- **Strategy:** Build a SceneIr via the Rust builder, serialize to JSON via the recipe format, deserialize back, and compare. The comparison is structural (same nodes, kinds, transforms, props) not pointer-identity.
- **Proptest:** Generate random valid recipe JSON, load it, verify it produces a valid SceneIr.

### Claude's Discretion
- Exact schemars/jsonschema version (use whatever is current and compatible).
- Whether to use `#[derive(Serialize, Deserialize, JsonSchema)]` on intermediate structs or hand-write the schema.
- Error type naming (`RecipeError` vs `JsonError` vs extending `SceneError`).
- Whether effects/pipeline entries are supported in JSON (probably defer to Phase 3.3 sandbox — JSON in 3.2 covers scene structure only, not effects).

</decisions>

<specifics>
## Specific Ideas

- The schema should be self-describing enough that an LLM given only the schema can write a valid recipe. This means enum variants should be clearly documented in the schema (oneOf with descriptions).
- A sample recipe file (`examples/recipes/hello.json`) ships with the library for documentation.

</specifics>

<deferred>
## Deferred Ideas

- **Effects in JSON recipes** — Phase 3.3 adds the sandbox that makes effect names safe. JSON recipes in 3.2 describe scene structure only.
- **Signal binding in JSON** — reactive signal references in JSON recipes are M4 (Python) scope.
- **Schema migrations** — first breaking change triggers design. Not in 3.2.

</deferred>
