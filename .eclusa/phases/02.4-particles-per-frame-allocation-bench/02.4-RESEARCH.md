# Phase 2.4: Particles + Per-Frame Allocation Bench - Research

**Researched:** 2026-04-16
**Domain:** Particle system (pool-based, zero-alloc), criterion allocation enforcement
**Confidence:** HIGH

## Summary

This phase adds a pool-based particle system to `happyterminals-renderer` and extends the existing criterion bench to enforce zero per-frame allocations across all renderer paths (cube, OBJ mesh, particles). The particle system is architecturally simple: a fixed-size `Vec<Particle>` with alive/dead flags, point-projection through the existing MVP pipeline, z-tested composition into the same Grid, and life-based ShadingRamp shading with linear color interpolation.

All building blocks already exist in the codebase. The `Renderer::draw()` method demonstrates the exact z-buffer + cell_chars pattern that `draw_particles()` will follow. The `rasterizer::project_vertex()` function handles 3D-to-screen projection for individual points. The `ShadingRamp::shade()` method maps a float to a ramp character (particles use `life/max_life` instead of `NdotL`). The `renderer_alloc.rs` criterion bench provides the template for the capacity-stability assertion.

**Primary recommendation:** Implement `particle.rs` as a self-contained module with `Particle` (Copy struct) + `ParticleEmitter` (owns the pool Vec). Add `Renderer::draw_particles()` using the same z-buffer/cell_chars arrays already allocated by `draw()`. Use `rand` (already in workspace deps) for spawn spread randomization. Extend `renderer_alloc.rs` with a `bench_draw_particles` function following the exact pattern of `bench_draw_mesh_bunny`.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **Crate location:** `happyterminals-renderer` -- particles are render primitives alongside Mesh and Cube, not a separate crate.
- **Pool strategy:** Fixed-size `Vec<Particle>` allocated once at emitter creation. Each particle has an `alive: bool` flag. Spawn scans for first dead slot; no heap allocation after init.
- **Update loop ownership:** Consumer (render callback) calls `emitter.update(dt)` each frame. The library does not auto-update via Effect.
- **Particle struct:** `Particle { position: Vec3, velocity: Vec3, life: f32, max_life: f32, color_start: Color, color_end: Color, alive: bool }`. All fields are `Copy`.
- **Emitter struct:** `ParticleEmitter { particles: Vec<Particle>, spawn_rate: f32, spawn_accumulator: f32, gravity: Vec3, origin: Vec3, spread: Vec3 }`.
- **Projection:** Same `Projection` pipeline as meshes, z-test against existing z-buffer.
- **Shading:** Single char from `ShadingRamp` based on `life / max_life`.
- **Color-over-time:** Linear interpolation `color_start` -> `color_end` based on `life / max_life`.
- **Composition:** Single pass into same `Grid` as meshes, z-buffer handles occlusion.
- **Allocation enforcement:** Extend existing criterion bench with capacity-stability proxy.
- **Coverage:** Every renderer path (cube, OBJ mesh, particles) gets allocation assertion.
- **Example:** Snow falling over bunny mesh, <=120 LOC, with InputMap controls (Space=play/pause, R=reset).
- **InputMap actions:** 2 Bool actions: `"particle_toggle"` (Space), `"particle_reset"` (R). Registered in example, not library defaults.

### Pattern 1: Pool-Based Particle Lifecycle
**What:** Fixed-size Vec allocated once; spawn scans for dead slots; update decrements life and marks dead.
**When to use:** Any time you need zero per-frame allocation with bounded particle count.
**Example:**
```rust
// Source: CONTEXT.md locked decision + existing Renderer pattern
pub struct Particle {
    pub position: Vec3,
    pub velocity: Vec3,
    pub life: f32,
    pub max_life: f32,
    pub color_start: Color,
    pub color_end: Color,
    pub alive: bool,
}

impl ParticleEmitter {
    pub fn update(&mut self, dt: f32, rng: &mut impl Rng) {
        // 1. Update alive particles (gravity, position, life countdown)
        for p in &mut self.particles {
            if !p.alive { continue; }
            p.velocity += self.gravity * dt;
            p.position += p.velocity * dt;
            p.life -= dt;
            if p.life <= 0.0 { p.alive = false; }
        }
        // 2. Spawn new particles using accumulator
        self.spawn_accumulator += self.spawn_rate * dt;
        while self.spawn_accumulator >= 1.0 {
            self.spawn_accumulator -= 1.0;
            self.spawn_one(rng);
        }
    }
    
    fn spawn_one(&mut self, rng: &mut impl Rng) {
        if let Some(p) = self.particles.iter_mut().find(|p| !p.alive) {
            // Initialize particle with randomized position within spread
            *p = Particle { alive: true, position: self.origin + /* spread */, .. };
        }
        // If no dead slot: silently drop (pool full, no alloc)
    }
}
```

### Pattern 2: Point Projection (no triangle rasterization)
**What:** Each alive particle projects to a single grid cell via `rasterizer::project_vertex()`, then z-tests against the same z-buffer used by mesh rendering.
**When to use:** Point-like particles that occupy exactly one cell.
**Example:**
```rust
// Reuses existing project_vertex from rasterizer module
pub fn draw_particles(
    &mut self,
    grid: &mut Grid,
    emitter: &ParticleEmitter,
    camera: &OrbitCamera,
    projection: &Projection,
    shading: &ShadingRamp<'_>,
) {
    // z_buffer and cell_chars already allocated by prior draw() call
    // (or re-allocated here if dimensions changed -- same pattern as draw())
    let mvp = /* same MVP computation as draw() */;
    
    for particle in emitter.alive_particles() {
        let Some((sx, sy, depth)) = project_vertex(particle.position, &mvp, half_w, half_h) else {
            continue;
        };
        let px = sx as usize;
        let py = sy as usize;
        if px >= grid_w || py >= grid_h { continue; }
        let idx = py * grid_w + px;
        
        // Reversed-Z depth test (same as triangle rasterizer)
        if depth > self.z_buffer[idx] {
            self.z_buffer[idx] = depth;
            // Life-based ramp character
            let t = (particle.life / particle.max_life).clamp(0.0, 1.0);
            let ramp_char = shading.ramp[(t * (shading.ramp.len() - 1) as f32) as usize];
            self.cell_chars[idx] = ramp_char;
            // Color lerp stored for later Grid write
        }
    }
}
```

### Pattern 3: Capacity-Stability Proxy (Allocation Enforcement)
**What:** Assert that Vec capacities do not grow between frames after warmup. If capacity is stable across 100 frames, no allocations occurred.
**When to use:** Criterion bench for zero-alloc enforcement without nightly (no dhat/custom allocator).
**Example:**
```rust
// Source: existing renderer_alloc.rs pattern
fn bench_draw_particles(c: &mut Criterion) {
    // Setup: create emitter, warmup renderer + emitter
    let mut emitter = ParticleEmitter::new(500, /* params */);
    let mut renderer = Renderer::new();
    
    // Warmup: run a few frames to fill the pool
    for _ in 0..10 {
        emitter.update(0.016, &mut rng);
        renderer.draw_particles(&mut grid, &emitter, &camera, &projection, &shading);
    }
    
    let cap_before = renderer.z_buffer.capacity();
    let pool_cap_before = emitter.particles.capacity();
    
    c.bench_function("renderer_draw_particles_80x24", |b| {
        b.iter(|| {
            emitter.update(0.016, &mut rng);
            renderer.draw_particles(&mut grid, &emitter, &camera, &projection, &shading);
        });
    });
    
    // Post-bench assertions
    assert_eq!(renderer.z_buffer.capacity(), cap_before);
    assert_eq!(emitter.particles.capacity(), pool_cap_before);
}
```

### Pattern 4: Color Interpolation for Particles
**What:** Linear lerp between `color_start` and `color_end` based on normalized life.
**When to use:** Per-particle color that changes over lifetime.
**Example:**
```rust
// ratatui_core::style::Color is an enum. For Rgb variant:
fn lerp_color(start: Color, end: Color, t: f32) -> Color {
    match (start, end) {
        (Color::Rgb(r1, g1, b1), Color::Rgb(r2, g2, b2)) => {
            let r = (r1 as f32 + (r2 as f32 - r1 as f32) * t) as u8;
            let g = (g1 as f32 + (g2 as f32 - g1 as f32) * t) as u8;
            let b = (b1 as f32 + (b2 as f32 - b1 as f32) * t) as u8;
            Color::Rgb(r, g, b)
        }
        _ => start, // Fallback: non-RGB colors don't lerp
    }
}
```
**Critical detail:** The `Color` enum from ratatui-core does NOT implement `Copy` (only `Clone`). However, checking the actual source: `Color` IS `Copy` in ratatui-core 0.1. The particle struct decision says "All fields are `Copy`" -- this works because `Color` is `#[derive(Copy, Clone)]`. Verified: ratatui-core 0.1 `Color` is `Copy`.

### Anti-Patterns to Avoid
- **Vec::push in spawn loop:** Never push new particles. Scan for dead slots and overwrite in-place.
- **Per-particle heap allocation:** No String, Box, or Vec inside Particle. All fields are Copy.
- **Separate z-buffer for particles:** Reuse the Renderer's existing z-buffer so particles and meshes occlude each other correctly.
- **Frame-rate-dependent spawn count:** Use the `spawn_accumulator` pattern (fractional accumulation) so spawn rate is consistent regardless of dt.
- **Calling draw_particles before draw:** The z-buffer must be cleared by the caller or by `draw()` first. Document that `draw_particles` does NOT clear the z-buffer -- it composites on top.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| RNG for spawn spread | Manual LCG | `rand::thread_rng()` (workspace dep) | Already in dep tree; better distribution; no maintenance burden |
| Color lerp | Complex color space math | Simple linear RGB lerp (3 component lerps) | Terminal color resolution is 8-bit per channel; gamma-correct interpolation is invisible at this fidelity |
| Allocation tracking | Custom global allocator | Criterion capacity-stability proxy | Works on stable Rust; existing pattern from Phase 1.3; dhat requires nightly |

**Key insight:** The particle system is intentionally simple -- point projection (not billboards), flat ramp shading (not texture sampling), fixed pool (not dynamic). This keeps the implementation under 200 LOC and the example under 120 LOC.

## Common Pitfalls

### Pitfall 1: Z-Buffer State Between draw() and draw_particles()
**What goes wrong:** If `draw_particles()` is called without `draw()` having been called first in the same frame, the z-buffer may be uninitialized or stale from a different frame size.
**Why it happens:** `draw()` handles buffer resize and clearing; `draw_particles()` assumes buffers are ready.
**How to avoid:** Either (a) have `draw_particles` also perform the resize/clear logic, or (b) extract a `prepare_frame()` method. Recommended: option (a) -- duplicate the 5-line resize+clear block for safety.
**Warning signs:** Particles rendering at wrong positions or not appearing after a terminal resize.

### Pitfall 2: spawn_accumulator Precision Drift
**What goes wrong:** At very high spawn rates with variable dt, the accumulator can overshoot, causing burst-then-starve patterns.
**Why it happens:** Floating-point accumulation with large `spawn_rate * dt` values.
**How to avoid:** Cap the number of spawns per frame (e.g., max 10) to prevent frame-time spikes from spawning hundreds.
**Warning signs:** Frame drops followed by particle bursts.

### Pitfall 3: Color Write Path -- cell_chars vs Grid Style
**What goes wrong:** The existing `draw()` writes characters to `cell_chars[]` then copies to Grid, but only sets the symbol (no color). Particles need to set fg color on the Grid cell too.
**Why it happens:** The original mesh renderer is monochrome (ASCII chars only). Particles introduce color-per-cell.
**How to avoid:** After writing cell_chars, the Grid write loop for particles must also call `cell.set_style(Style::default().fg(lerped_color))`. This means particles need a parallel "color buffer" or must write directly to Grid during projection.
**Warning signs:** All particles appear in the terminal's default foreground color.

**Important design decision:** The existing `draw()` method stores only chars in `cell_chars` -- it has no color buffer. For particles, two approaches:
1. **Add a parallel `cell_colors: Vec<Option<Color>>` buffer** to Renderer (zero-alloc after warmup, matches existing pattern).
2. **Write particles directly to Grid** instead of through cell_chars (breaks the two-phase pattern but simpler).

Recommendation: Option 1 (parallel color buffer). It maintains the existing two-phase architecture (project -> write) and the color gets downsampled by the backend flush pass automatically.

### Pitfall 4: Particle Example LOC Budget
**What goes wrong:** Example exceeds 120 LOC because of boilerplate.
**Why it happens:** InputMap setup, mesh loading, camera fitting, and render callback all consume lines.
**How to avoid:** Follow the model-viewer pattern closely. The model-viewer is ~100 LOC of callback + ~120 LOC total. The particle example should be simpler (one mesh, no cycling). Target: ~30 LOC setup + ~60 LOC callback + ~30 LOC main.
**Warning signs:** Count LOC early during implementation.

### Pitfall 5: Ramp Direction for Particles vs Meshes
**What goes wrong:** `ShadingRamp::shade()` takes a `Vec3` normal and computes NdotL. Particles don't have normals -- they use `life/max_life` as brightness.
**Why it happens:** API mismatch between normal-based shading and lifetime-based brightness.
**How to avoid:** Don't call `shade()` for particles. Instead, index into `shading.ramp` directly using `t = life/max_life`. The ramp array is public (`pub ramp: &[char]`), so direct indexing works: `ramp[(t * (ramp.len() - 1) as f32) as usize]`.
**Warning signs:** Particles all rendering as the same character regardless of age.

### ParticleEmitter with Spawn Accumulator
```rust
use rand::Rng;

pub struct ParticleEmitter {
    pub particles: Vec<Particle>,
    pub spawn_rate: f32,       // particles per second
    pub gravity: Vec3,
    pub origin: Vec3,
    pub spread: Vec3,          // half-extents of spawn volume
    pub life_range: (f32, f32),
    pub color_start: Color,
    pub color_end: Color,
    spawn_accumulator: f32,
    paused: bool,
}

impl ParticleEmitter {
    pub fn new(capacity: usize) -> Self {
        Self {
            particles: vec![Particle::default(); capacity],
            spawn_rate: 50.0,
            gravity: Vec3::new(0.0, -2.0, 0.0),
            origin: Vec3::ZERO,
            spread: Vec3::ONE,
            life_range: (3.0, 5.0),
            color_start: Color::White,
            color_end: Color::Rgb(180, 200, 255),
            spawn_accumulator: 0.0,
            paused: false,
        }
    }
    
    pub fn update(&mut self, dt: f32, rng: &mut impl Rng) {
        if self.paused { return; }
        // Update alive particles
        for p in &mut self.particles {
            if !p.alive { continue; }
            p.velocity += self.gravity * dt;
            p.position += p.velocity * dt;
            p.life -= dt;
            if p.life <= 0.0 { p.alive = false; }
        }
        // Spawn new particles via accumulator
        self.spawn_accumulator += self.spawn_rate * dt;
        let max_spawns = 10; // cap per frame
        let mut spawned = 0;
        while self.spawn_accumulator >= 1.0 && spawned < max_spawns {
            self.spawn_accumulator -= 1.0;
            self.spawn_one(rng);
            spawned += 1;
        }
    }
    
    pub fn reset(&mut self) {
        for p in &mut self.particles { p.alive = false; }
        self.spawn_accumulator = 0.0;
    }
    
    pub fn toggle_pause(&mut self) { self.paused = !self.paused; }
    pub fn is_paused(&self) -> bool { self.paused }
    
    pub fn alive_count(&self) -> usize {
        self.particles.iter().filter(|p| p.alive).count()
    }
}
```

## Open Questions

1. **Color buffer in Renderer**
   - What we know: The existing `draw()` only stores chars in `cell_chars`. Particles need per-cell colors.
   - What's unclear: Whether to add a `Vec<Option<Color>>` parallel buffer or write colors directly to Grid during projection.
   - Recommendation: Add `cell_colors: Vec<Option<Color>>` to Renderer. Initialized to `None`, set when a particle writes a cell. The Grid write loop then applies colors from this buffer. This keeps the two-phase pattern and the color gets downsampled by the backend flush pass. The Vec is allocated once (same warmup pattern as z_buffer).

2. **draw_particles before or after draw()?**
   - What we know: z-buffer must be cleared before either call. `draw()` clears it.
   - What's unclear: Whether the example should call `draw()` then `draw_particles()` or vice versa.
   - Recommendation: `draw()` first (clears z_buffer + renders mesh), then `draw_particles()` (composites particles on top with z-testing). This matches the "snow falling OVER bunny" visual -- particles behind the bunny are correctly occluded.

3. **Prelude re-exports**
   - What we know: CONTEXT.md says re-export `ParticleEmitter`, `Particle` from prelude.
   - Recommendation: Add to `crates/happyterminals/src/lib.rs` prelude: `pub use happyterminals_renderer::{Particle, ParticleEmitter};`
