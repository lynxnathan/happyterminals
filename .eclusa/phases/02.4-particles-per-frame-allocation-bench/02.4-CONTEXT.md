<decisions>
## Implementation Decisions

### Particle System Architecture
- **Crate location:** `happyterminals-renderer` — particles are render primitives alongside Mesh and Cube, not a separate crate.
- **Pool strategy:** Fixed-size `Vec<Particle>` allocated once at emitter creation. Each particle has an `alive: bool` flag. Spawn scans for first dead slot; no heap allocation after init. Matches zero-alloc discipline from Phase 1.3 criterion bench.
- **Update loop ownership:** Consumer (render callback) calls `emitter.update(dt)` each frame. The library does not auto-update via Effect — the consumer owns the tick. Same pattern as camera math in the model-viewer.
- **Particle struct:** `Particle { position: Vec3, velocity: Vec3, life: f32, max_life: f32, color_start: Color, color_end: Color, alive: bool }`. All fields are `Copy`. No Box, String, or heap-allocated payload.
- **Emitter struct:** `ParticleEmitter { particles: Vec<Particle>, spawn_rate: f32, spawn_accumulator: f32, gravity: Vec3, origin: Vec3, spread: Vec3 }`. The `spawn_accumulator` fractionally tracks when to spawn the next particle (avoids frame-rate-dependent spawn counts).

### Rendering + Composition
- **Projection:** Each alive particle projects to a grid cell via the same `Projection` pipeline as meshes. Z-test against the existing z-buffer decides visibility (particles can be occluded by mesh surfaces).
- **Shading:** Single char from `ShadingRamp` based on `life / max_life` — brighter when young, dims as it dies. Reuses existing ramp infrastructure, no new shading types.
- **Color-over-time:** Linear interpolation between `color_start` and `color_end` based on `life / max_life`. Zero allocations (pure math on Copy types). Passes through the Phase 2.2 color-mode downsample at flush time.
- **Composition:** Particles render into the SAME `Grid` as meshes in a single pass. The renderer's z-buffer handles occlusion naturally — no two-pass or overlay layer needed. `Renderer::draw_particles(&self, grid, emitter, camera, projection, shading)` is a new method alongside `draw()`.

### Allocation Enforcement
- **Strategy:** Extend existing criterion bench (`renderer_alloc.rs`) with capacity-stability proxy — the Phase 1.3 pattern. Measure `Vec` capacities before and after 100 frames; assert stable (no growth = no allocations). No new deps (dhat is nightly-only, custom global allocator is overkill for this).
- **Coverage:** Every renderer path gets an allocation assertion: cube, OBJ mesh (bunny), particles. The bench fails CI if any path regresses.

### Example Design
- **Effect:** Snow falling over the bunny mesh. Visually clear gravity-only physics, good for demo screenshots.
- **Complexity:** ≤ 120 LOC. Loads bunny, creates snow emitter above the bounding sphere, orbit camera via InputMap, play/pause/reset via keys.
- **InputMap integration:** 3 Bool actions: `"particle_toggle"` (Space — play/pause), `"particle_reset"` (R — reset all particles). Registered and bound in the example, not in library defaults.

### Claude's Discretion
- Exact spawn-rate tuning for the snow effect (visually pleasing density).
- Whether `draw_particles` takes `&ParticleEmitter` or `&[Particle]` slice — implementation detail.
- RNG strategy for spawn spread (use `fastrand` or manual LCG to avoid `rand` dep).
- Whether to add a `ParticleEmitter::with_capacity(n)` builder or just expose the struct fields.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `crates/happyterminals-renderer/src/lib.rs` — `Renderer::draw()` with z-buffer rasterizer, projection pipeline, shading ramp. Particle rendering extends this with a simpler point-projection path (no triangle rasterization).
- `crates/happyterminals-renderer/src/camera.rs` — `OrbitCamera`, `Camera` trait with `view_matrix()`. Particles use the same camera.
- `crates/happyterminals-renderer/benches/renderer_alloc.rs` — existing criterion bench with `bench_draw_cube` and `bench_draw_mesh_bunny`. Extend with `bench_draw_particles`.
- `crates/happyterminals-input/` — InputMap, `run_with_input`, action registration. Particle example uses this for play/pause/reset.
- `crates/happyterminals-renderer/src/shading.rs` — `ShadingRamp` with `shade()` method. Particles pass `life/max_life` as the "brightness" parameter.

### Established Patterns
- `Renderer::draw(grid, mesh, camera, projection, shading)` — the existing render call signature. `draw_particles` follows the same pattern.
- Zero-alloc per-frame: no `Vec::push` in hot paths, pre-allocate, reuse buffers. Validated by criterion bench.
- Examples follow the `run_with_input(callback, FrameSpec, InputMap)` pattern from Phase 2.3.

### Integration Points
- **New files:** `crates/happyterminals-renderer/src/particle.rs` — `Particle`, `ParticleEmitter`, update/spawn logic.
- **Modified:** `crates/happyterminals-renderer/src/lib.rs` — add `draw_particles()` method to `Renderer`.
- **Modified:** `crates/happyterminals-renderer/benches/renderer_alloc.rs` — add particle allocation bench.
- **New:** `crates/happyterminals/examples/particles/main.rs` — snow-over-bunny example.
- **Prelude:** re-export `ParticleEmitter`, `Particle` from `happyterminals::prelude`.

</code_context>

<specifics>
## Specific Ideas

- Snow effect: emitter origin at `bounding_sphere.center + Vec3::Y * radius * 1.5`. Spawn spread: random XZ within radius, small random Y variance. Gravity `Vec3::new(0.0, -2.0, 0.0)`. Lifetime 3-5 seconds (randomized). Start color white (`Rgb(255,255,255)`), end color light blue (`Rgb(180,200,255)`).
- Particle count: 500 pool size for the snow example. At 30fps with 3-5s lifetime, ~150-250 alive at steady state.
- The bench should also verify that the particle pool itself doesn't grow: `emitter.particles.capacity()` before == after 100 frames.

</specifics>

<deferred>
## Deferred Ideas

- **GPU-accelerated particles** — not applicable to TTY; parked unless WASM target surfaces.
- **Particle collision with mesh** — requires spatial acceleration structure. Not in scope for M2.
- **Multiple emitter types** (point, line, sphere, mesh-surface)** — only point emitter in Phase 2.4. Others in M3 if needed.
- **Particle trails / ribbons** — render complexity, defer to M3 compositor work.
- **Custom particle shaders** — the ShadingRamp is the "shader" for now. Custom per-particle ramps could be M3.

</deferred>
