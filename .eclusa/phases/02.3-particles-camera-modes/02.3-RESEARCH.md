# Phase 2.3: Input Action System + Camera Modes - Research

**Researched:** 2026-04-16
**Domain:** Input action architecture, camera systems, reactive signal integration, crossterm mouse/keyboard events
**Confidence:** HIGH

## Summary

Phase 2.3 introduces a new `happyterminals-input` crate implementing a Godot/Unreal-inspired input action system where **actions are reactive signals** -- a novel approach no existing game engine uses. The system translates raw crossterm events into named, typed actions (`Bool`, `Axis1D`, `Axis2D`) through a binding layer with context stacking, modifiers, and edge detection. Camera modes (orbit enhanced with mouse drag, free-look, FPS) are the first consumers.

The existing codebase is well-prepared: `Signal<T>` and `Effect` primitives work, `TerminalGuard` already enables `EnableMouseCapture`, crossterm 0.29 provides `MouseEventKind::Drag(button)` with per-button tracking, and `CameraConfig` is already an extensible enum designed for exactly this expansion. The main architectural challenge is the reactive integration -- action signals must be `!Send + !Sync` (same-thread, matching `Signal<T>` semantics) but dispatched from the event arm of `run.rs`'s `tokio::select!` loop, which runs on the same thread (current_thread runtime). This is a clean fit.

**Primary recommendation:** Build `happyterminals-input` as a standalone crate with zero async dependency. The `InputMap` is a synchronous state machine: `dispatch(&mut self, event: &CrosstermEvent)` walks the context stack, resolves bindings, applies modifiers, and calls `signal.set()` directly. No channels, no polling. The event loop in `run.rs` calls `input_map.dispatch(ev)` where it currently does per-variant matching.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **Input action system architecture**: Actions are typed (`ActionValue::Bool/Axis1D/Axis2D`), identified by string keys. `ActionState` has `JustPressed/Held(Duration)/Released/Inactive` edges. `Binding` supports `Key/MouseButton/Scroll/Drag`. `InputContext` stacking with push/pop. `InputMap` as runtime registry. `Modifier` pipeline (Negate/Scale/Deadzone/Swizzle).

- **Mouse drag state machine**: `Idle -> Pressed(button, start_pos) -> Dragging(button, delta) -> Released`. Left-drag = orbit, middle-drag = pan, scroll = zoom. 3D viewport convention.

### Pattern 1: Actions as Reactive Signals (Novel Architecture)

**What:** Each registered action owns a `Signal<ActionState>` (Bool), `Signal<f32>` (Axis1D), or `Signal<Vec2>` (Axis2D). When `InputMap::dispatch()` resolves a binding, it calls `signal.set()` directly, which triggers the reactive graph to propagate to all subscribers (Effects, Memos).

**When to use:** Always -- this is the core architectural decision.

**Example:**
```rust
// Action registration (InputMap owns the signals)
pub struct ActionEntry {
    pub value_type: ActionValueType,
    pub state: Signal<ActionState>,
    pub axis1d: Option<Signal<f32>>,
    pub axis2d: Option<Signal<glam::Vec2>>,
}

// Camera subscribes via Effect (consumer side)
let orbit_action = input_map.action_signal_2d("orbit").clone();
let zoom_action = input_map.action_signal_1d("zoom").clone();

Effect::new(move || {
    let delta = orbit_action.get();
    // delta is Vec2 -- x = azimuth change, y = elevation change
    camera_azimuth.update(|a| *a += delta.x * sensitivity);
    camera_elevation.update(|e| *e += delta.y * sensitivity);
});
```

**Key insight:** This is genuinely novel. Godot/Unreal poll `Input.is_action_pressed()` per frame. Here, the reactive graph dispatches only when input actually changes, and only to subscribers. A key press with zero subscribers is a no-op.

### Pattern 2: Context Stack Dispatch

**What:** `InputMap` holds a `Vec<InputContext>` stack. `dispatch()` walks from top to bottom; first context with a matching binding consumes the event (no propagation).

**When to use:** All event dispatch.

**Example:**
```rust
pub fn dispatch(&mut self, event: &CrosstermEvent) {
    // First, update drag state machine (mouse events may become Drag actions)
    if let CrosstermEvent::Mouse(mouse) = event {
        self.drag_machine.update(mouse);
    }

    // Walk context stack top-down
    for context in self.contexts.iter().rev() {
        if let Some(action_key) = context.resolve(event, &self.drag_machine) {
            self.fire_action(action_key, event);
            return; // consumed
        }
    }
}
```

### Pattern 3: Mouse Drag State Machine

**What:** Tracks button-down -> drag -> release transitions. Crossterm 0.29 provides `MouseEventKind::Drag(MouseButton)` natively, so the state machine primarily tracks the start position (for deadzone) and computes deltas.

**When to use:** Drag-to-orbit, drag-to-pan, drag-to-look.

**Example:**
```rust
pub enum DragState {
    Idle,
    Pressed {
        button: MouseButton,
        start_col: u16,
        start_row: u16,
    },
    Dragging {
        button: MouseButton,
        last_col: u16,
        last_row: u16,
    },
}

impl DragStateMachine {
    pub fn update(&mut self, mouse: &MouseEvent) -> Option<DragOutput> {
        match (&self.state, mouse.kind) {
            (DragState::Idle, MouseEventKind::Down(btn)) => {
                self.state = DragState::Pressed {
                    button: btn,
                    start_col: mouse.column,
                    start_row: mouse.row,
                };
                None
            }
            (DragState::Pressed { button, start_col, start_row },
             MouseEventKind::Drag(btn)) if btn == *button => {
                let dx = mouse.column as f32 - *start_col as f32;
                let dy = mouse.row as f32 - *start_row as f32;
                self.state = DragState::Dragging {
                    button: *button,
                    last_col: mouse.column,
                    last_row: mouse.row,
                };
                Some(DragOutput { button: *button, delta: Vec2::new(dx, dy) })
            }
            (DragState::Dragging { button, last_col, last_row },
             MouseEventKind::Drag(btn)) if btn == *button => {
                let dx = mouse.column as f32 - *last_col as f32;
                let dy = mouse.row as f32 - *last_row as f32;
                self.state = DragState::Dragging {
                    button: *button,
                    last_col: mouse.column,
                    last_row: mouse.row,
                };
                Some(DragOutput { button: *button, delta: Vec2::new(dx, dy) })
            }
            (_, MouseEventKind::Up(_)) => {
                self.state = DragState::Idle;
                None // could emit Released if needed
            }
            _ => None,
        }
    }
}
```

### Pattern 4: Camera Trait with InputActions

**What:** A `Camera` trait decouples camera math from the concrete camera type. `update()` takes a reference to action signals (or a snapshot struct), and `view_matrix()` returns the Mat4.

**Example:**
```rust
pub trait Camera {
    fn view_matrix(&self) -> Mat4;
    fn update(&mut self, dt: Duration, actions: &CameraActions);
}

/// Snapshot of relevant action values for camera update.
pub struct CameraActions {
    pub orbit_delta: Vec2,     // from "orbit" Axis2D action
    pub pan_delta: Vec2,       // from "pan" Axis2D action
    pub zoom_delta: f32,       // from "zoom" Axis1D action
    pub move_forward: bool,    // from "move_forward" Bool action
    pub move_back: bool,
    pub move_left: bool,
    pub move_right: bool,
}
```

### Anti-Patterns to Avoid
- **Polling actions per frame:** The entire point is reactive dispatch. Never `if input.is_pressed("W")` in a render loop. Use `Effect::new` or read signals with `.untracked()` when the renderer needs the current value.
- **Propagating events past the consuming context:** Once a context claims an event, it is gone. No bubble-up.
- **Storing raw crossterm types in the public API:** The input crate should expose its own `Binding::Key(KeyCode)` etc., re-exporting crossterm types. Consumers should not need `use crossterm::event::*`.
- **Mutable camera in the render hot path:** Camera `update()` happens in the tick callback (before render), not during `terminal.draw()`. The render path gets an immutable camera reference.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Mouse event tracking | Custom CSI parser | crossterm 0.29 `MouseEventKind::Drag(button)` | Already handles terminal protocol differences; tracks button during drag natively |
| View matrix math | Manual matrix construction | `glam::Mat4::look_at_rh` | Numerically stable, handles edge cases (gimbal near poles) |
| Focus change detection | ANSI escape polling | crossterm `EnableFocusChange` + `FocusGained`/`FocusLost` events | Standard terminal escape sequence, already abstracted |
| Reactive propagation | Manual subscriber lists | `Signal<T>.set()` + `Effect::new()` from happyterminals-core | Already built, proven with 257 tests passing |

**Key insight:** The heavy lifting (terminal event abstraction, reactive graph, linear algebra) is already solved. This phase is pure architecture -- designing the InputMap/Action/Binding/Context layer and wiring it to existing systems.

## Common Pitfalls

### Pitfall 1: Drag Delta Accumulation vs. Frame-Rate
**What goes wrong:** Mouse drag deltas accumulate between frames. If the event loop processes 3 drag events before the next render tick, the camera jumps by the sum of all 3 deltas.
**Why it happens:** crossterm events arrive asynchronously; the tick interval is 33ms (30fps) but mouse events can arrive at 1-5ms intervals.
**How to avoid:** Two approaches: (a) accumulate deltas per frame and reset on tick, or (b) use the most recent absolute position and compute delta from last-tick position. Approach (a) is better for smooth orbit because it preserves total displacement. The `InputMap` should accumulate Vec2 deltas and the tick callback should read-and-reset.
**Warning signs:** Jerky orbit that "catches up" in bursts, or lost rotation when moving slowly.

### Pitfall 2: Signal Threading Mismatch
**What goes wrong:** `Signal<T>` is `!Send + !Sync`. If the input map is created on a different thread than the signal, `signal.set()` panics.
**Why it happens:** `tokio::main(flavor = "current_thread")` keeps everything on one thread, but if someone uses `multi_thread` or spawns tasks, signals break.
**How to avoid:** Document that `InputMap` must be created on the same thread as the reactive runtime (the render thread). Use `SignalSetter` if cross-thread dispatch is ever needed (future extension, not Phase 2.3). Current architecture is single-thread -- this is fine.
**Warning signs:** Panic with `reactive_graph` arena access from wrong thread.

### Pitfall 3: TerminalGuard Focus Change Restore
**What goes wrong:** Adding `EnableFocusChange` to the guard's acquire without adding `DisableFocusChange` to restore leaks focus-change mode to the shell after exit.
**Why it happens:** Easy to add the enable but forget the symmetric disable.
**How to avoid:** Add `DisableFocusChange` to `TerminalGuard::restore()` alongside `DisableMouseCapture`. The `execute!` macro in restore already handles multiple commands.
**Warning signs:** Terminal echoing `^[[I` / `^[[O` escape sequences after the program exits.

### Pitfall 4: Context Stack Ordering Ambiguity
**What goes wrong:** Multiple contexts claim the same key (e.g., WASD for pan in default, WASD for translate in freelook). If the stack order is wrong, the wrong action fires.
**Why it happens:** User pushes context but forgets to pop when switching back.
**How to avoid:** Context push/pop should be symmetric. Camera mode switch: pop old context, push new context. The `"default"` context is never popped (enforced by InputMap). Consider a `replace_top()` convenience method.
**Warning signs:** Keys doing the wrong thing after switching camera modes.

### Pitfall 5: Elevation Gimbal Lock at Poles
**What goes wrong:** OrbitCamera elevation at +/- PI/2 causes view_matrix to degenerate (up vector aligns with look direction).
**Why it happens:** `look_at_rh` with Y-up fails when eye is directly above/below target.
**How to avoid:** Clamp elevation to `(-PI/2 + epsilon, PI/2 - epsilon)` where epsilon is ~0.01. Same applies to FpsCamera pitch.
**Warning signs:** Camera snaps or flickers when looking straight up/down.

### Pitfall 6: Deadzone Jitter on Sensitive Terminals
**What goes wrong:** Some terminals (especially over SSH or tmux) report micro-movements as drag events, causing unwanted orbit drift.
**Why it happens:** Terminal mouse reporting has integer cell granularity; micro-movements round to the same cell, but crossterm still emits Drag events.
**How to avoid:** `Modifier::Deadzone(threshold)` on drag bindings. Default threshold of 1-2 cells. Apply before the delta reaches the action signal.
**Warning signs:** Model slowly rotating when mouse is stationary but button held.

### Pitfall 7: Event Consumption and Quit
**What goes wrong:** If the InputContext stack consumes Ctrl-C as an action (e.g., mapped to "quit" action), the hardcoded `is_quit_event()` check in `run.rs` never fires.
**Why it happens:** Two systems trying to handle the same event: the old `is_quit_event` and the new InputMap.
**How to avoid:** Remove `is_quit_event()` from `run.rs` and make quit purely action-driven. The quit action's Effect calls a shutdown signal. Or: check quit BEFORE dispatching to InputMap. The latter is simpler for Phase 2.3.
**Warning signs:** Program won't exit, or exits on Ctrl-C but not on 'Q' (or vice versa).

## Code Examples

### FreeLookCamera Implementation
```rust
// Source: Standard FPS-camera math (glam::Mat4::look_at_rh)
use glam::{Mat4, Vec3};

#[derive(Debug, Clone)]
pub struct FreeLookCamera {
    pub position: Vec3,
    pub yaw: f32,   // radians, 0 = looking along -Z
    pub pitch: f32,  // radians, clamped
    pub speed: f32,
}

impl FreeLookCamera {
    pub fn view_matrix(&self) -> Mat4 {
        let forward = self.forward();
        let target = self.position + forward;
        Mat4::look_at_rh(self.position, target, Vec3::Y)
    }

    pub fn forward(&self) -> Vec3 {
        Vec3::new(
            self.yaw.sin() * self.pitch.cos(),
            self.pitch.sin(),
            -self.yaw.cos() * self.pitch.cos(),
        )
    }

    pub fn right(&self) -> Vec3 {
        self.forward().cross(Vec3::Y).normalize()
    }
}

impl Default for FreeLookCamera {
    fn default() -> Self {
        Self {
            position: Vec3::new(0.0, 0.0, 5.0),
            yaw: 0.0,
            pitch: 0.0,
            speed: 5.0,
        }
    }
}
```

### FpsCamera (Ground-Constrained)
```rust
#[derive(Debug, Clone)]
pub struct FpsCamera {
    pub position: Vec3,
    pub yaw: f32,
    pub pitch: f32,
    pub speed: f32,
    pub ground_y: f32,
}

impl FpsCamera {
    pub fn view_matrix(&self) -> Mat4 {
        let forward = Vec3::new(self.yaw.sin(), 0.0, -self.yaw.cos()).normalize();
        let up_offset = Vec3::new(0.0, self.pitch.sin(), 0.0);
        let target = self.position + forward + up_offset;
        Mat4::look_at_rh(self.position, target, Vec3::Y)
    }

    pub fn translate_xz(&mut self, forward: f32, strafe: f32, dt: f32) {
        let fwd = Vec3::new(self.yaw.sin(), 0.0, -self.yaw.cos());
        let right = fwd.cross(Vec3::Y).normalize();
        self.position += (fwd * forward + right * strafe) * self.speed * dt;
        self.position.y = self.ground_y; // lock to ground
    }
}
```

### InputMap Dispatch (Recommended Architecture)
```rust
pub struct InputMap {
    contexts: Vec<InputContext>,       // stack, index 0 = bottom ("default")
    actions: HashMap<String, ActionEntry>,
    drag_machine: DragStateMachine,
}

impl InputMap {
    pub fn dispatch(&mut self, event: &CrosstermEvent) {
        // Phase 1: Update drag state machine for mouse events
        let drag_output = match event {
            CrosstermEvent::Mouse(mouse) => self.drag_machine.update(mouse),
            _ => None,
        };

        // Phase 2: Walk context stack top-down for regular events
        for ctx in self.contexts.iter().rev() {
            if let Some(fired) = ctx.try_resolve(event, drag_output.as_ref()) {
                self.apply_action(&fired);
                return;
            }
        }
    }

    fn apply_action(&self, fired: &FiredAction) {
        if let Some(entry) = self.actions.get(&fired.action_key) {
            // Apply modifier chain
            let value = fired.modifiers.iter().fold(
                fired.raw_value.clone(),
                |v, m| m.apply(v),
            );
            // Set the signal -- reactive graph propagates from here
            match value {
                ActionValue::Bool(b) => entry.state.set(if b {
                    ActionState::JustPressed
                } else {
                    ActionState::Released
                }),
                ActionValue::Axis1D(v) => {
                    if let Some(sig) = &entry.axis1d { sig.set(v); }
                }
                ActionValue::Axis2D(v) => {
                    if let Some(sig) = &entry.axis2d { sig.set(v); }
                }
            }
        }
    }

    pub fn push_context(&mut self, context: InputContext) {
        self.contexts.push(context);
    }

    pub fn pop_context(&mut self, name: &str) -> Option<InputContext> {
        if let Some(pos) = self.contexts.iter().rposition(|c| c.name == name) {
            if pos == 0 { return None; } // never pop default
            Some(self.contexts.remove(pos))
        } else {
            None
        }
    }

    pub fn rebind(&mut self, context_name: &str, action: &str, binding: Binding) {
        if let Some(ctx) = self.contexts.iter_mut().find(|c| c.name == context_name) {
            ctx.rebind(action, binding);
        }
    }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Polling `Input.is_action_pressed()` per frame (Godot, Unreal) | Actions as reactive signals (this project) | Phase 2.3 (novel) | Only subscribers run; no polling waste; fine-grained reactivity |
| Raw key matching in event loop | Named action system with bindings | Godot 4.0 (2023), Unreal since UE4 | Decouples logic from specific keys; enables rebinding |
| Single global input handler | Context stack (push/pop) | Common in game engines since ~2015 | Mode-specific bindings without if/else chains |

**Deprecated/outdated:**
- `crossterm 0.27` had limited drag tracking; 0.29 provides full `Drag(MouseButton)` natively.
- Older input systems used `HashMap<KeyCode, Callback>` -- no value typing, no edge detection, no modifiers.

## Open Questions

1. **Accumulated deltas vs. per-event deltas for Axis2D actions**
   - What we know: Multiple drag events can arrive between render ticks. Both approaches (accumulate with reset, or latest-only) have tradeoffs.
   - What's unclear: Which produces smoother orbit in practice on various terminal emulators.
   - Recommendation: Start with accumulation (sum deltas, reset on tick). Planner should include a per-event variant as a fallback if accumulation feels laggy. The tick callback reads `orbit_delta.untracked()` then sets it to `Vec2::ZERO`.

2. **ActionState edge detection timing**
   - What we know: `JustPressed` should fire once, then transition to `Held(duration)`. But when does the transition happen? On the next dispatch? On the next tick?
   - What's unclear: Whether the reactive graph needs to see `JustPressed -> Held` as two distinct signal changes.
   - Recommendation: `JustPressed` fires on key-down dispatch. A tick-level update sets `Held(elapsed)` for any currently-active actions. `Released` fires on key-up dispatch. This requires a `tick_update()` method on InputMap called per frame.

3. **InputMap ownership model**
   - What we know: InputMap must live where `Signal<T>` is valid (same thread as reactive runtime).
   - What's unclear: Whether `InputMap` should be passed into `run()`/`run_scene()` as a parameter, or created inside the loop.
   - Recommendation: Add `InputMap` as an optional field on `FrameSpec` or as a new parameter to `run_scene()`. For backward compat, `run()` and `run_scene()` work without an InputMap (legacy path). New `run_with_input()` takes an InputMap.
