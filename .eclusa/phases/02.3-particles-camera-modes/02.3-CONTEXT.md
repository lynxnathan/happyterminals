<decisions>
## Implementation Decisions

### Input Action System Architecture (user vision: "think Godot/Unreal input mode")

**Core type model:**
- `Action` — named, identified by string key (`"orbit"`, `"pan"`, `"zoom"`, `"cycle_model"`, `"quit"`). Each action has an `ActionValue` type.
- `ActionValue` — `Bool(bool)` / `Axis1D(f32)` / `Axis2D(Vec2)`. Actions are typed at creation.
- `ActionState` — `JustPressed` / `Held(Duration)` / `Released` / `Inactive`. Edge detection built-in. `JustPressed` fires once on transition from inactive → active; `Held` tracks duration; `Released` fires once on transition from active → inactive.
- `Binding` — `Key(KeyCode)` / `MouseButton(MouseButton)` / `Scroll(ScrollDirection)` / `Drag { button: MouseButton, axis: DragAxis }`. Multiple bindings per action.
- `InputContext` — named group of action-to-binding mappings. Contexts are **pushed/popped** on a stack; the topmost context that matches an event consumes it (event does not propagate down). Examples: `"orbit_mode"`, `"freelook_mode"`, `"menu_mode"`.
- `InputMap` — the runtime registry. Holds context stack + all registered actions. `InputMap::rebind(context, action, binding)` for runtime rebinding. Persistence (save to file) is a future phase.
- `Modifier` — `Negate` / `Scale(f32)` / `Deadzone(f32)` / `Swizzle`. Applied to bindings before the action signal updates. Deadzone is critical for mouse drag (small movements → zero) and future analog input.

**The reactive twist — actions ARE signals:**
- Every registered action gets a `Signal<ActionState>` (for Bool) or `Signal<f32>` (for Axis1D) or `Signal<Vec2>` (for Axis2D).
- Camera controllers and other consumers read actions via the reactive graph — `Effect::new(|| { let delta = orbit_action.get(); ... })`.
- When raw crossterm events arrive, the `InputMap` resolves bindings → updates the action signals → reactive graph propagates to all subscribers. **No polling. No per-frame input read loop.** The reactive graph IS the dispatch.
- This means: a key press with no subscribers is a no-op (fine-grained — only touched cells re-render). This is the SolidJS advantage applied to input.

**Mouse drag state machine:**
- Drag requires state tracking: `Idle` → `Pressed(button, start_pos)` → `Dragging(button, current_delta)` → `Released`.
- `Drag { button, axis }` binding maps to `Axis2D` action value (delta from last frame, not absolute position).
- Deadzone modifier prevents micro-jitter on stationary mouse.
- Mouse drag → orbit: left-button drag maps azimuth/elevation delta. Mouse drag → pan: middle-button or Shift+left maps X/Y translation delta.

**Input context stack (full, not scaffolded):**
- Contexts pushed/popped at runtime: `input_map.push_context("freelook_mode")`, `input_map.pop_context("freelook_mode")`.
- Event matching: walk the stack top-down; first context with a matching binding consumes the event.
- Default context at bottom of stack: `"default"` with orbit/pan/zoom/quit bindings. Never popped.
- Camera mode switching pushes a new context on top that overrides relevant bindings.
- Example: FreeLook mode pushes `"freelook"` context where WASD → translate instead of default WASD → pan.

### Camera Modes

**Orbit camera (existing, enhanced):**
- Currently: azimuth/elevation driven by time-based rotation signal in spinning-cube, arrow-key cycling in model-viewer.
- After Phase 2.3: orbit driven by `Drag { button: Left, axis: XY }` action. Scroll wheel maps to `Axis1D` zoom (adjusts `OrbitCamera.distance`). Arrow keys available for model cycling (no conflict — different InputContext or different action).

**FreeLook camera (new):**
- WASD → translate (forward/back/strafe left/right) relative to camera heading.
- Mouse drag → look direction (yaw/pitch).
- No ground-plane constraint (free flight through scene).
- New struct: `FreeLookCamera { position: Vec3, yaw: f32, pitch: f32, speed: f32 }` with `view_matrix()`.

**FPS camera (new):**
- Same as FreeLook but **constrained to ground plane** (Y position locked or gravity-influenced).
- WASD → translate on XZ plane.
- Mouse drag → look direction (yaw/pitch, pitch clamped to avoid gimbal).
- New struct: `FpsCamera { position: Vec3, yaw: f32, pitch: f32, speed: f32, ground_y: f32 }`.

**`Camera` trait:**
- Introduce `trait Camera { fn view_matrix(&self) -> Mat4; fn update(&mut self, actions: &InputActions); }`.
- `OrbitCamera`, `FreeLookCamera`, `FpsCamera` all implement.
- `CameraConfig` enum in scene crate gets `FreeLook(FreeLookCamera)`, `Fps(FpsCamera)` variants (already extensible by design from Phase 1.4).
- Camera mode switching: push/pop InputContext + swap CameraConfig variant on the Scene.

### Model-Viewer Upgrade

- Replace hardcoded arrow-key handling with proper InputMap + Action registration.
- Default context: `"viewer_default"` — Left/Right → `CycleModel` action, mouse drag → `Orbit`, scroll → `Zoom`, WASD → `Pan`, Ctrl-C → `Quit`.
- Bunny-orientation fix: with user-controlled orbit rotation via mouse drag, user can rotate the bunny to see it from any angle. No per-model rotation offset needed — the control surface solves it.
- Model name overlay stays at row 0 (from Phase 2.1).
- Camera auto-fit from `bounding_sphere()` still applied per model switch (resets orbit distance but preserves user's azimuth/elevation — don't reset rotation on model switch).

### Crate Location

- New crate: `happyterminals-input` — lives alongside `-core`, `-renderer`, etc. in the workspace.
- Depends on: `happyterminals-core` (Signal, Effect), `crossterm` (event types, re-exported for binding construction).
- Consumed by: `happyterminals-backend-ratatui` (event → InputMap dispatch), `happyterminals-scene` (CameraConfig update), examples.
- Prelude re-exports: `InputMap`, `Action`, `Binding`, `InputContext`, `ActionValue`, `Modifier`.

- Internal event dispatch architecture (channel vs direct call vs reactive subscription topology) — researcher + planner decide.
- Exact modifier application order — researcher investigates best practices from Godot/Unreal source.
- Whether `InputMap` owns the action Signals or the consumer creates them and registers with InputMap — implementation detail.
- Test strategy for mouse drag state machine (mock events vs integration test with synthetic crossterm EventStream).
- How to handle simultaneous conflicting bindings (e.g., two contexts both claim `KeyCode::W`) — standard stack precedence should suffice, but edge cases are planner's call.

## Existing Code Insights

### Reusable Assets
- `crates/happyterminals-renderer/src/camera.rs` — `OrbitCamera { azimuth, elevation, distance, target }` pure-data struct with `view_matrix()`. Does NOT hold Signals — pure input from the render loop. Phase 2.3 extends with `FreeLookCamera`, `FpsCamera`.
- `crates/happyterminals-scene/src/camera.rs` — `CameraConfig` enum with `Orbit(OrbitCamera)` variant. Already designed for extensibility with additional camera variants.
- `crates/happyterminals-backend-ratatui/src/run.rs` — `crossterm::event::Event` already consumed in `tokio::select!` loop. Currently only handles `Event::Key` and `Event::Resize`. Mouse events (`Event::Mouse`) already arrive via crossterm but are ignored. The InputMap dispatch hooks in here.
- `crates/happyterminals-core/src/signal.rs` — `Signal<T>`, `SignalSetter` — the reactive primitives that action signals wrap.
- `crates/happyterminals-core/src/effect.rs` — `Effect::new(|| ...)` — how camera controllers subscribe to action signal changes.
- `crates/happyterminals/examples/model-viewer/main.rs` — 96-LOC example with hardcoded arrow-key handling. Phase 2.3 replaces this with InputMap + Action registration.

### Established Patterns
- Signals are `T: Send + Sync` — action signals will need `ActionState: Send + Sync` (trivially true for Copy types).
- `SignalSetter` channel for cross-thread writes — input events arrive on the event thread, action signals update via setter, reactive graph propagates on the render thread. Same pattern as existing signal flow.
- `FrameSpec` is the user-facing configuration entry point. `InputMap` could be passed alongside or built independently.
- `TerminalGuard` enables crossterm features (mouse capture, focus events, bracketed paste). Mouse capture must be enabled for mouse events to arrive.

### Integration Points
- **`run.rs` event loop:** the `Event::Key` / `Event::Mouse` / `Event::FocusGained|Lost` match arms become `input_map.dispatch(event)` — a single call replaces the current per-key handling.
- **TerminalGuard:** must enable `crossterm::event::EnableMouseCapture` and `crossterm::event::EnableFocusChange` at setup (currently only enables raw mode + alt screen). This is the TTY-level plumbing for "other things people connect to TTYs."
- **Scene / DSL:** camera controller reads actions, not raw keys. The DSL's `.camera(OrbitCamera { ... })` builder stays; what changes is HOW the camera is driven per-frame.
- **Prelude:** add input types alongside existing re-exports.

<specifics>
## Specific Ideas

- **User vision analogy:** "pensa no input mode do Godot ou Unreal" — the input system should feel like a game engine's, not a TUI library's. Named actions, multi-binding, contexts, modifiers. The distinguishing feature vs Godot/Unreal: actions are reactive signals, not polled.
- **Mouse drag → orbit is the killer UX for the model viewer.** Left-drag = orbit, scroll = zoom, middle-drag or Shift+left-drag = pan. This is the 3D viewport convention (Blender, Unity scene view, VS Code 3D preview). Follow convention.
- **Bunny-orientation fix comes for free:** once user can orbit via mouse drag, they can look at the bunny from any angle without needing per-model rotation offsets.
- **TerminalGuard needs to enable mouse capture + focus events.** crossterm's `EnableMouseCapture` makes the terminal emit mouse events (CSI-style). `EnableFocusChange` makes the terminal emit focus in/out. Both are standard escape sequences supported by all modern terminals (Win Terminal, iTerm2, Kitty, GNOME Terminal, Alacritty). SSH + tmux pass them through.
- **Default InputContext bindings (shipped with the library, not just examples):** orbit (left-drag), pan (middle-drag or Shift+left-drag), zoom (scroll), quit (Ctrl-C or Q). Users override by pushing their own context on top.
- **Phase 2.3 replaces the hardcoded model-viewer arrow keys** with proper `Action::CycleModel` bound to Left/Right in the `"viewer"` InputContext. The old spinning-cube example stays simple (time-driven rotation, no input beyond Ctrl-C) — it's an M1 artifact, not upgraded here.

</specifics>

## Deferred Ideas

- **Particle system** — moved to Phase 2.4 (render-side, orthogonal to input layer). Particles will consume the input system via signal-driven emitter controls (play/pause/reset actions), but the emitter/lifetime/gravity core is render work.
- **Gamepad / serial input** — the `Binding` enum can be extended with `Gamepad(GamepadButton)` / `GamepadAxis(...)` later without breaking the action model. Not in Phase 2.3 — crossterm doesn't expose gamepad; would need `gilrs` or similar.
- **Input persistence (save/load keybindings to file)** — natural Phase 3.x work alongside JSON recipes. `InputMap::to_json()` / `from_json()`.
- **Touch input / gesture recognition** — not applicable to TTY; parked unless WASM target surfaces.
- **Macro recording / replay** — record action stream, replay for demo/test automation. Cool but post-v1.
- **Input visualization** (show active bindings overlay, debug which context consumed an event) — dev-tool, not user-facing. Maybe Phase 3.x.
- **Resize hardening + MSRV + STL** — moved from Phase 2.4 to Phase 2.5 due to scope restructure.
