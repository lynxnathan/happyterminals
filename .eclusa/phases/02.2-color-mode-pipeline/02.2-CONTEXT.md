# Phase 2.2: Color-Mode Pipeline - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning
**Mode:** Smart discuss (autonomous)

<domain>
## Phase Boundary

Make happyterminals color output safely degrade across the full terminal capability spectrum (truecolor → 256 → 16 → monochrome) with explicit user control via `NO_COLOR` and `--force-color`, so the library never outputs a garbled color sequence on a terminal that can't parse it. Covers REND-08. Addresses PITFALL §8 (color regression across terminals).

</domain>

<decisions>
## Implementation Decisions

### Detection + Override
- **Env var probe cascade (priority order):** `NO_COLOR` (hard off, any non-empty value) → `FrameSpec.color_mode` programmatic override → `$COLORTERM=truecolor|24bit` (truecolor signal) → `$TERM` inspection fallback. Follows no-color.org + bixense.com/clicolors conventions.
- **No external dep** — hand-roll the cascade in `happyterminals-backend-ratatui`. `supports-color` crate considered and rejected: the cascade is ~30 LOC; adding a dep for trivial env probing violates the "no speculative deps" discipline from Phase 0.
- **API surface:** `FrameSpec.color_mode: Option<ColorMode>` where `ColorMode = TrueColor | Palette256 | Ansi16 | Mono`. `None` = runtime detection; `Some(x)` = hard override. Mirrors the existing FrameSpec pattern from Phase 1.1.
- **Detection timing:** eager at `run()` entry, cached in `TerminalGuard`. Detected mode passed into the render loop as a `ColorMode` value, constant for the lifetime of the run. No per-frame env reads.

### Downsampling Algorithm
- **RGB → 256:** Euclidean distance in sRGB space against the 256-color palette (6×6×6 color cube + 24 greys + 16 system). No `palette` or Oklab dep — the 256 palette is too coarse for perceptual differences to matter in ASCII art.
- **256 → 16:** `const fn` precomputed lookup table (256-entry array) built at compile time from the standard ANSI 16-color quantization. Zero runtime cost per cell.
- **Monochrome fallback:** drop `fg`/`bg` colors entirely; preserve `Modifier` (bold/italic/underline/reverse). Rationale: the shading ramp from Phase 1.3 already encodes brightness via char density, so color-stripped output still communicates shape. Luminance → bold-dim mapping rejected as over-clever.
- **Color pass locus:** **flush-time, lazy downsampling**. Grid stores raw `Color::Rgb(...)` values regardless of detected mode; the backend ratatui bridge applies the downsample as a final pass before `ratatui::Terminal::flush()`. Keeps Grid representation stable across mode changes and avoids lossy early commitment.

### Test + Docs Strategy
- **Snapshot matrix:** `insta` snapshots of flushed ANSI bytes for 4 color modes × 2 reference scenes (static cube + static bunny). 8 fixtures total. Pattern matches Phase 1.5's cross-terminal verification approach.
- **New example:** `crates/happyterminals/examples/color-test/main.rs`. Accepts `--force-color={truecolor,256,16,none}` CLI arg; renders the same scene across modes for visible comparison. ≤ 80 LOC target (color-test is more mechanical than model-viewer; should be leaner).
- **Docs location:** inline in `crates/happyterminals/README.md` under §"Terminal Color Support". Include tmux guidance: `set -g default-terminal "tmux-256color"` + `set -ga terminal-overrides ",*256col*:Tc"`. Cross-link from color-test example README.
- **tmux Tc flag:** documented in the README section; we do not auto-probe tmux-specific env vars (users configure tmux themselves; we respect `$COLORTERM` which tmux passes through when `Tc` is set).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ratatui_core::style::Color` (re-exported via `happyterminals_core::cell::Color`) — already models `Rgb(u8,u8,u8)`, `Indexed(u8)` for 256, and the 16 named ANSI variants (`Black`, `Red`, …, `White`, `LightRed`, …). No new color enum needed in our crate.
- `crates/happyterminals-core/src/grid.rs` — `Grid` newtype over `ratatui::Buffer`; per-cell `set_fg`/`set_bg`/`set_style` already write `Color` values. Phase 2.2 does NOT touch Grid writes — only the flush path downstream.
- `crates/happyterminals-backend-ratatui/src/run.rs` — `TerminalGuard` RAII from Phase 1.1 is where eager detection belongs. `run()` already has a pre-loop setup block.
- `crates/happyterminals-renderer/src/shading.rs` — `ShadingRamp` is the brightness-via-char-density mechanism that makes monochrome viable. No changes needed; it complements the color pipeline.
- `insta` workspace dep already wired into renderer's `[dev-dependencies]`; extend to backend-ratatui dev-deps for the snapshot matrix.

### Established Patterns
- `FrameSpec` is the user-facing configuration struct passed into `run_scene`. Adding `color_mode: Option<ColorMode>` is the idiomatic extension point. Default is `None` (auto-detect).
- `TerminalGuard` caches per-run state (cursor visibility, raw mode, alt-screen) — extending it with a cached `ColorMode` is a natural fit.
- Lint baseline: `unsafe_code=forbid`, `unwrap_used=deny`, `expect_used=deny`, `dbg_macro=deny` across all library crates. Env var probing via `std::env::var` returns `Result`; propagate with `?` or `.ok()`.
- `[workspace.dependencies]` pins no color or env libs currently. Phase 2.2 adds none.

### Integration Points
- **New module:** `crates/happyterminals-backend-ratatui/src/color.rs` — hosts `ColorMode` enum, detection cascade, and `fn downsample(buffer: &mut Buffer, mode: ColorMode)` final-pass transform.
- **FrameSpec extension:** `crates/happyterminals-backend-ratatui/src/lib.rs` (wherever FrameSpec lives) — add `color_mode: Option<ColorMode>` field with default `None`.
- **Flush hook:** the existing flush path in `run.rs` — insert the downsample call between scene render and `Terminal::flush()`. Single line of code at the integration point.
- **Prelude re-export:** `crates/happyterminals/src/lib.rs` — add `pub use happyterminals_backend_ratatui::color::ColorMode;` so end-users can `use happyterminals::prelude::*; FrameSpec { color_mode: Some(ColorMode::Ansi16), .. }`.

</code_context>

<specifics>
## Specific Ideas

- `--force-color=none` in the CLI arg = `ColorMode::Mono` (semantic alias for the `NO_COLOR=1` env pathway; keeps the CLI idiomatic).
- tmux docs section should include the common gotcha: `$TMUX` + `$TERM=screen-256color` without `Tc` override → falsely detected as 256-color when underlying terminal is truecolor-capable. The workaround (`Tc` in terminal-overrides) is the sole recommended config.
- Color-test example outputs a small readable header at row 0 indicating the active mode: "color-test — mode: Palette256 (auto-detected from $COLORTERM)". Useful when visually comparing screenshots.
- The 256 palette quantization should match `xterm` standard (6×6×6 color cube, levels `[0, 95, 135, 175, 215, 255]` per channel). Document this in a docstring comment on the const table so future maintainers don't second-guess.
- Snapshot fixtures should be named `{scene}_{mode}.snap` (e.g., `cube_truecolor.snap`, `bunny_ansi16.snap`). Matches `insta`'s conventions.

</specifics>

<deferred>
## Deferred Ideas

- **Per-cell color mode overrides** — e.g., forcing a specific cell to truecolor even in Ansi16 mode. Not needed; violates the "one mode per run" simplicity. If ever wanted, surfaces in M3 compositor work.
- **Oklab / perceptually-uniform color distance** — deferred unless visible banding becomes a reported issue. Current Euclidean sRGB is adequate.
- **Automatic tmux probe / `$TMUX` handling** — users own their tmux config. Documenting the `Tc` gotcha is sufficient.
- **Theme / palette customization** — swap in a custom 256 palette instead of xterm standard. Not a Phase 2.2 concern; sits naturally in a future M3 "theming" phase if user demand surfaces.
- **Auto-fallback chain** — "if terminal supports 256 but user forced truecolor, warn and downgrade." Rejected: if user forces a mode, we respect it literally. No silent overrides.
- **Background color strategies (transparency / preserve terminal bg)** — out of scope here; the Grid already models bg as `Option<Color>`-equivalent via `Color::Reset`.

</deferred>
