## User Constraints (from CONTEXT.md)

### Locked Decisions

**Detection + Override**
- Env var probe cascade (priority order): `NO_COLOR` (hard off, any non-empty value) → `FrameSpec.color_mode` programmatic override → `$COLORTERM=truecolor|24bit` (truecolor signal) → `$TERM` inspection fallback. Follows no-color.org + bixense.com/clicolors conventions.
- **No external dep** — hand-roll the cascade in `happyterminals-backend-ratatui`. `supports-color` crate considered and rejected: the cascade is ~30 LOC; adding a dep for trivial env probing violates the "no speculative deps" discipline from Phase 0.
- **API surface:** `FrameSpec.color_mode: Option<ColorMode>` where `ColorMode = TrueColor | Palette256 | Ansi16 | Mono`. `None` = runtime detection; `Some(x)` = hard override. Mirrors the existing FrameSpec pattern from Phase 1.1.
- **Detection timing:** eager at `run()` entry, cached in `TerminalGuard`. Detected mode passed into the render loop as a `ColorMode` value, constant for the lifetime of the run. No per-frame env reads.

**Downsampling Algorithm**
- **RGB → 256:** Euclidean distance in sRGB space against the 256-color palette (6×6×6 color cube + 24 greys + 16 system). No `palette` or Oklab dep — the 256 palette is too coarse for perceptual differences to matter in ASCII art.
- **256 → 16:** `const fn` precomputed lookup table (256-entry array) built at compile time from the standard ANSI 16-color quantization. Zero runtime cost per cell.
- **Monochrome fallback:** drop `fg`/`bg` colors entirely; preserve `Modifier` (bold/italic/underline/reverse). Rationale: the shading ramp from Phase 1.3 already encodes brightness via char density, so color-stripped output still communicates shape. Luminance → bold-dim mapping rejected as over-clever.
- **Color pass locus:** **flush-time, lazy downsampling**. Grid stores raw `Color::Rgb(...)` values regardless of detected mode; the backend ratatui bridge applies the downsample as a final pass before `ratatui::Terminal::flush()`. Keeps Grid representation stable across mode changes and avoids lossy early commitment.

**Test + Docs Strategy**
- Snapshot matrix: `insta` snapshots of flushed ANSI bytes for 4 color modes × 2 reference scenes (static cube + static bunny). 8 fixtures total.
- New example: `crates/happyterminals/examples/color-test/main.rs`. Accepts `--force-color={truecolor,256,16,none}` CLI arg. ≤ 80 LOC target.
- Docs location: inline in `crates/happyterminals/README.md` under §"Terminal Color Support". Include tmux guidance: `set -g default-terminal "tmux-256color"` + `set -ga terminal-overrides ",*256col*:Tc"`.
- tmux Tc flag: documented in the README section; we do not auto-probe tmux-specific env vars.

**Integration Points**
- New module: `crates/happyterminals-backend-ratatui/src/color.rs` — hosts `ColorMode` enum, detection cascade, and `fn downsample(buffer: &mut Buffer, mode: ColorMode)`.
- FrameSpec extension: `crates/happyterminals-backend-ratatui/src/frame_spec.rs` — add `color_mode: Option<ColorMode>` field with default `None`.
- Flush hook: existing flush path in `run.rs` — insert the downsample call between scene render and `*frame.buffer_mut() = grid.deref().clone()`.
- Prelude re-export: `crates/happyterminals/src/lib.rs` — add `pub use happyterminals_backend_ratatui::ColorMode;`.

## Summary

Phase 2.2 is fully spec-driven: the no-color.org standard and the xterm 6×6×6-cube spec both give us unambiguous canonical values, so this is an **implementation phase, not an exploration phase**. Every locked decision in CONTEXT.md has a corresponding primary source (no-color.org, termstandard/colors, `ratatui_core::buffer::Cell` public fields, ITU T.416 for ANSI 16). Risk surface is low: no external deps, no async concerns, no cross-crate plumbing beyond a single `FrameSpec` field and a single new module.

The three concrete technical unknowns that were worth researching — (a) `ratatui_core::buffer::Cell` field mutability, (b) `NO_COLOR` empty-string semantics, (c) canonical xterm palette math — all resolved to simple, well-documented answers. Cell fields `fg`, `bg`, `modifier` are `pub` AND have setter methods, so `downsample(buffer: &mut Buffer, mode: ColorMode)` can iterate `buffer.content_mut()` and mutate in place without clone. `NO_COLOR` spec says "present and not an empty string (regardless of value)" disables color — so `NO_COLOR=""` is OFF = treat as unset. The xterm 256 palette is a product table over levels `[0, 95, 135, 175, 215, 255]` for the cube and a closed-form arithmetic for the greys.

**Primary recommendation:** implement as three small self-contained pieces: (1) `ColorMode` enum + `detect_color_mode(override: Option<ColorMode>) -> ColorMode` cascade function, (2) `const PALETTE_256: [(u8,u8,u8); 256]` built at compile time + `fn nearest_256(rgb) -> u8` with squared-Euclidean (no sqrt), (3) `const PALETTE_16_LUT: [u8; 256]` mapping each 256-index to its nearest 16-system-color. The downsample pass is a single `match mode` over `buffer.content_mut()`. Everything else (FrameSpec field, TerminalGuard cache, run.rs hook, prelude re-export, color-test example, README section, 8 insta fixtures) is wiring.

### Pattern 1: Env Var Cascade (detection)

**What:** Pure function that reads env once and returns a `ColorMode`.
**When to use:** Called exactly once at `run()` entry.
**Example:**
```rust
// Source: no-color.org (HIGH), termstandard/colors (HIGH)
pub fn detect_color_mode(override_mode: Option<ColorMode>) -> ColorMode {
    // 1. NO_COLOR is the highest-priority disable (per no-color.org spec).
    //    "Present and not an empty string, regardless of value" → disable.
    if let Ok(v) = std::env::var("NO_COLOR") {
        if !v.is_empty() {
            return ColorMode::Mono;
        }
    }

    // 2. Explicit programmatic override from FrameSpec.
    if let Some(mode) = override_mode {
        return mode;
    }

    // 3. COLORTERM truecolor signal (de facto standard per Alacritty,
    //    Kitty, iTerm2, recent VTE-based terminals). Case-insensitive
    //    comparison is conservative; the spec uses lowercase but some
    //    shells upper-case env defaults.
    if let Ok(v) = std::env::var("COLORTERM") {
        let lower = v.to_ascii_lowercase();
        if lower == "truecolor" || lower == "24bit" {
            return ColorMode::TrueColor;
        }
    }

    // 4. TERM fallback — conservative mapping.
    match std::env::var("TERM").as_deref() {
        Ok("dumb") => ColorMode::Mono,
        Ok(t) if t.contains("256color") => ColorMode::Palette256,
        Ok("xterm-kitty" | "alacritty") => ColorMode::TrueColor,  // both set TERM + support truecolor
        Ok(_) => ColorMode::Ansi16,   // any other TERM → assume 16-color baseline
        Err(_) => ColorMode::Mono,    // no TERM at all → piped / non-tty → plain text
    }
}
```
*Note:* `VarError::NotUnicode` is the rare case where bytes aren't UTF-8; we treat it as "unset" by using `.ok()` on the `Result` and falling through. Non-unicode env values are pathological on Linux and nonexistent on Windows (env is UTF-16 there).

### Pattern 2: Const 256-Palette Table

**What:** Compile-time array of `(u8,u8,u8)` RGB triples for xterm color indices 0..255.
**When to use:** Built once in the binary; `nearest_256` does squared-Euclidean distance against it.
**Example:**
```rust
// Source: Wikipedia ANSI escape code spec (HIGH),
//         https://en.wikipedia.org/wiki/ANSI_escape_code#8-bit
//         https://www.ditig.com/256-colors-cheat-sheet (verification)

/// Standard xterm 256-color palette.
/// - Indices 0..16: ANSI system colors (xterm variant)
/// - Indices 16..232: 6×6×6 RGB cube, ordered as 16 + 36*r + 6*g + b
///   with channel levels [0, 95, 135, 175, 215, 255]
/// - Indices 232..256: 24 greyscale steps, level = 8 + 10*(i - 232)
///   (so greys span 8, 18, 28, ..., 238)
pub const PALETTE_256: [(u8, u8, u8); 256] = build_palette();

const CUBE_LEVELS: [u8; 6] = [0, 95, 135, 175, 215, 255];

const SYSTEM_16: [(u8, u8, u8); 16] = [
    (0x00, 0x00, 0x00), (0x80, 0x00, 0x00), (0x00, 0x80, 0x00), (0x80, 0x80, 0x00),
    (0x00, 0x00, 0x80), (0x80, 0x00, 0x80), (0x00, 0x80, 0x80), (0xC0, 0xC0, 0xC0),
    (0x80, 0x80, 0x80), (0xFF, 0x00, 0x00), (0x00, 0xFF, 0x00), (0xFF, 0xFF, 0x00),
    (0x00, 0x00, 0xFF), (0xFF, 0x00, 0xFF), (0x00, 0xFF, 0xFF), (0xFF, 0xFF, 0xFF),
];

const fn build_palette() -> [(u8, u8, u8); 256] {
    let mut out = [(0u8, 0u8, 0u8); 256];
    // System colors
    let mut i = 0;
    while i < 16 { out[i] = SYSTEM_16[i]; i += 1; }
    // 6x6x6 cube
    let mut r = 0;
    while r < 6 {
        let mut g = 0;
        while g < 6 {
            let mut b = 0;
            while b < 6 {
                out[16 + 36*r + 6*g + b] =
                    (CUBE_LEVELS[r], CUBE_LEVELS[g], CUBE_LEVELS[b]);
                b += 1;
            }
            g += 1;
        }
        r += 1;
    }
    // Greyscale 232..256
    let mut k = 0;
    while k < 24 {
        let v = 8 + 10 * (k as u8);
        out[232 + k] = (v, v, v);
        k += 1;
    }
    out
}

/// Squared Euclidean distance in sRGB.
/// We return squared (not sqrt) — ordering is preserved and avoids f64.
#[inline]
const fn sq_dist(a: (u8,u8,u8), b: (u8,u8,u8)) -> u32 {
    let dr = a.0 as i32 - b.0 as i32;
    let dg = a.1 as i32 - b.1 as i32;
    let db = a.2 as i32 - b.2 as i32;
    (dr*dr + dg*dg + db*db) as u32
}

pub fn nearest_256(rgb: (u8, u8, u8)) -> u8 {
    let mut best_i = 0u8;
    let mut best_d = u32::MAX;
    let mut i = 0u16;  // 0..256
    while i < 256 {
        let d = sq_dist(rgb, PALETTE_256[i as usize]);
        if d < best_d { best_d = d; best_i = i as u8; }
        i += 1;
    }
    best_i
}
```

### Pattern 3: Const 256→16 Lookup

**What:** `[u8; 256]` indexed by 256-palette-index, yielding the nearest 16-system-color index.
**When to use:** Runtime hot path: `PALETTE_16_LUT[nearest_256(rgb) as usize]`.
**Example:**
```rust
pub const PALETTE_16_LUT: [u8; 256] = build_16_lut();

const fn build_16_lut() -> [u8; 256] {
    let mut out = [0u8; 256];
    let mut i = 0;
    while i < 256 {
        // For each 256-palette entry, find the nearest SYSTEM_16.
        let target = PALETTE_256[i];
        let mut best_j = 0u8;
        let mut best_d = u32::MAX;
        let mut j = 0;
        while j < 16 {
            let d = sq_dist(target, SYSTEM_16[j]);
            if d < best_d { best_d = d; best_j = j as u8; }
            j += 1;
        }
        out[i] = best_j;
        i += 1;
    }
    out
}
```
Both `build_palette()` and `build_16_lut()` are `const fn` and resolve at compile time. No runtime cost beyond the single array lookups.

### Pattern 4: Flush-Time Downsample Pass

**What:** Walk every cell in the `ratatui::Buffer`, transform fg/bg according to mode.
**When to use:** Once per frame, between scene render and `Terminal::draw` completion.
**Example:**
```rust
use ratatui_core::buffer::Buffer;
use ratatui_core::style::Color;

pub fn downsample(buffer: &mut Buffer, mode: ColorMode) {
    if matches!(mode, ColorMode::TrueColor) { return; }   // fast path
    // buffer.content is pub; iterate all cells. In ratatui 0.30 Cell.fg/bg are pub.
    for cell in buffer.content.iter_mut() {
        cell.fg = map_color(cell.fg, mode);
        cell.bg = map_color(cell.bg, mode);
        // modifier preserved
    }
}

fn map_color(c: Color, mode: ColorMode) -> Color {
    match mode {
        ColorMode::TrueColor => c,
        ColorMode::Mono => match c {
            Color::Reset => Color::Reset,      // keep Reset — it has its own meaning
            _ => Color::Reset,                 // everything else → default terminal color
        },
        ColorMode::Palette256 => match c {
            Color::Rgb(r, g, b) => Color::Indexed(nearest_256((r, g, b))),
            other => other,                    // Indexed / named 16 / Reset pass through
        },
        ColorMode::Ansi16 => match c {
            Color::Rgb(r, g, b) => {
                let idx16 = PALETTE_16_LUT[nearest_256((r, g, b)) as usize];
                named_from_u8(idx16)
            }
            Color::Indexed(i) => named_from_u8(PALETTE_16_LUT[i as usize]),
            other => other,                    // already 16-color-safe or Reset
        },
    }
}

fn named_from_u8(i: u8) -> Color {
    match i {
        0 => Color::Black,        1 => Color::Red,         2 => Color::Green,
        3 => Color::Yellow,       4 => Color::Blue,        5 => Color::Magenta,
        6 => Color::Cyan,         7 => Color::Gray,        8 => Color::DarkGray,
        9 => Color::LightRed,    10 => Color::LightGreen, 11 => Color::LightYellow,
        12 => Color::LightBlue, 13 => Color::LightMagenta,14 => Color::LightCyan,
        _ => Color::White,
    }
}
```

### Pattern 5: Run-Loop Hook

Existing `run.rs` flush path:
```rust
terminal.draw(|frame| {
    grid.resize(frame.area());
    render_fn(&mut grid, &input_signals);
    *frame.buffer_mut() = grid.deref().clone();
})?;
```
becomes:
```rust
let color_mode = detect_color_mode(spec.color_mode);  // once, before loop
// ...
terminal.draw(|frame| {
    grid.resize(frame.area());
    render_fn(&mut grid, &input_signals);
    let mut out = grid.deref().clone();
    downsample(&mut out, color_mode);                 // flush-time transform
    *frame.buffer_mut() = out;
})?;
```
The clone already exists (ratatui owns its own buffer internally); we downsample the clone, not the Grid, preserving "Grid stays truecolor".

### Anti-Patterns to Avoid

- **Probing env inside the frame loop:** reading `std::env::var` per frame is both a syscall and a race hazard against `set_var`. Detection is eager-once at `run()` entry — cache, pass by value.
- **Cloning the buffer twice:** `terminal.draw` already gives you a `&mut Buffer` via `frame.buffer_mut()`. The grid→ratatui-buffer copy currently clones once; adding a second clone just to downsample would double memory churn. Downsample the single clone in place (see Pattern 5).
- **Running nearest_256 per-cell with sqrt:** squared distance preserves ordering; skip the sqrt.
- **Using `supports-color` crate:** CONTEXT rejected. 30 LOC of std is less review surface than a new dep.
- **Emitting a warning if user forces truecolor in a 16-color terminal:** CONTEXT rejected. Respect overrides literally.

## Common Pitfalls

### Pitfall 1: `NO_COLOR` empty-string semantics

**What goes wrong:** Treating `NO_COLOR=""` as "disable color" produces unexpected behavior — some shells unset-by-setting-empty and the user did NOT opt out of color.
**Why it happens:** The no-color.org spec says "present and **not an empty string**, regardless of value". An empty string is the one exception.
**How to avoid:** `if let Ok(v) = env::var("NO_COLOR") { if !v.is_empty() { return Mono; } }`. Note: `env::var` returns `Err(NotPresent)` if unset AND `Ok("")` if set-to-empty — these must be distinguished.
**Warning signs:** Tests that set `NO_COLOR=""` and expect color on.
**Source:** [no-color.org](https://no-color.org/) (HIGH) — canonical spec.

### Pitfall 2: `COLORTERM=1` false positive

**What goes wrong:** Some setups set `COLORTERM=1` as a sloppy "yes, colors supported" signal. Our cascade checks for `truecolor|24bit`, so `1` falls through — but if we'd done `!v.is_empty()` we'd falsely claim truecolor.
**Why it happens:** COLORTERM pre-dates the termstandard/colors spec; many dotfiles have legacy non-spec values.
**How to avoid:** Exact match against `truecolor` or `24bit` (case-insensitive). Anything else → fall through to `$TERM`.
**Warning signs:** Screenshots from Windows Terminal 1.x showing garbled 24-bit escapes in apps that "detect" truecolor naively.
**Source:** [termstandard/colors](https://github.com/termstandard/colors) (HIGH).

### Pitfall 3: tmux without `Tc` — the silent strip

**What goes wrong:** User runs `tmux` with default config. Inside tmux, `TERM=screen-256color`, `COLORTERM` may or may not be forwarded. If we detect `Palette256` and emit indexed escapes, fine. But if user has `COLORTERM=truecolor` in their shell's `.bashrc` AND tmux doesn't have `Tc` set, tmux strips the 24-bit escapes down to approximations — user sees garbled colors and blames us.
**Why it happens:** tmux multiplexes many panes over one underlying terminal; it needs an explicit opt-in (`terminal-overrides ",*256col*:Tc"`) to pass truecolor through rather than quantize it.
**How to avoid:** Document the fix in README; don't try to auto-detect tmux. If user forces truecolor inside broken-tmux, it's on them. Our responsibility: (a) honor their override, (b) tell them how to fix tmux.
**Warning signs:** Bug reports saying "colors look random inside tmux".
**Source:** [tmux FAQ](https://github.com/tmux/tmux/wiki/FAQ) (HIGH), [Sunaku's tmux 24-bit guide](https://sunaku.github.io/tmux-24bit-color.html) (HIGH).

### Pitfall 4: Snapshot non-determinism from env leaking between tests

**What goes wrong:** Test A sets `NO_COLOR=1`, then Test B runs in the same process (cargo test runs tests in parallel by default, but threads share env) and reads the leaked value.
**Why it happens:** `std::env::set_var` is process-global. Parallel test threads see each other's sets.
**How to avoid:** Either (a) test `detect_color_mode` as a pure function that takes env values as params (best — dependency-inject env reads via a small `EnvProvider` trait with a default `RealEnv` impl), or (b) serialize env-touching tests with `#[cfg(test)]` `std::sync::Mutex` and `serial_test`/`temp_env`. Option (a) is simpler and avoids a new dep.
**Warning signs:** Tests pass locally, fail in CI with `--jobs 4`.
**Source:** Standard Rust testing hazard; `std::env::set_var` docs call it out explicitly.

### Pitfall 5: Snapshot non-determinism from ratatui's diff escape ordering

**What goes wrong:** `insta` snapshot captures raw ANSI bytes, but ratatui's diff may emit `\e[38;2;r;g;bm` before or after `\e[48;2;r;g;bm` depending on previous-cell state. Minor reorderings cause snapshot churn.
**Why it happens:** ratatui's flush walks the buffer and emits SGR resets/changes based on the delta from the previous cell.
**How to avoid:** Snapshot against a **clean `TestBackend`** (ratatui's in-memory backend) initialized identically for each case. The TestBackend buffer-string representation is deterministic given identical input. See `ratatui::backend::TestBackend::buffer()` which returns the final Buffer state, which `insta::assert_debug_snapshot!` serializes deterministically. Alternative: capture the raw output via `Terminal::draw` into a `Vec<u8>` backed by `CursorBackend`, but snapshotting the final `Buffer` state is easier and equally diagnostic.
**Warning signs:** First-run snapshot accepted; CI run of the same code differs by escape-sequence order.
**Source:** [ratatui TestBackend docs](https://docs.rs/ratatui/latest/ratatui/backend/struct.TestBackend.html) (HIGH).

### Pitfall 6: Logging in the downsample hot path (PITFALL §29)

**What goes wrong:** Temptation to log "downsampled N cells in mode X" inside the flush loop. Floods stdout where the TUI draws. Corrupts the screen.
**Why it happens:** Debugging the pipeline during development.
**How to avoid:** All logging in this phase is `tracing::trace!` only, gated as today; and logs MUST route to stderr or a file, never stdout (project rule). If you need to log the detected mode once, do it once at `run()` entry with `tracing::debug!`, not per-frame.
**Warning signs:** Garbled output in examples when RUST_LOG is set.
**Source:** `.eclusa/research/PITFALLS.md` §29.

### Pitfall 7: `NO_COLOR` stripping `Color::Reset`

**What goes wrong:** In `Mono` mode, naively converting every `Color` to `Color::Reset` sounds right — but some cells legitimately already hold `Color::Reset` (meaning "default terminal color"). Converting Reset→Reset is a no-op and correct. The subtle trap is mapping `Color::Rgb` → `Color::Black` (a solid color) instead of `Color::Reset`. On a light-theme terminal the user wants their default foreground, not forced black.
**Why it happens:** Writer thinks "strip color means set to the default" and picks `Color::Black`.
**How to avoid:** `Mono` maps all chromatic colors to `Color::Reset`. Let the terminal choose. Modifiers (bold/underline/reverse) are preserved (spec-defined mono output).
**Warning signs:** Mono mode shows black text on a light terminal background instead of honoring user's default fg.

### Pitfall 8: MSRV drift on `const fn` features

**What goes wrong:** The const-palette builder uses `while` + mutation, which requires fairly recent `const fn` capabilities. If our MSRV is too old the const builder won't compile.
**Why it happens:** Rust's `const fn` feature set has grown across versions.
**How to avoid:** Workspace MSRV is Rust 1.88 (verified from `Cargo.toml` `[workspace.package].rust-version = "1.88"`). `while` in const context has been stable since Rust 1.46; mutable bindings + indexed array writes in const are stable since 1.83. All comfortably below 1.88.
**Warning signs:** `error[E0658]: ... is only allowed in a const context` on CI build.
**Source:** Rust reference, const evaluation stability list.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `$TERM` string parsing ("xterm-24bit") for truecolor | `$COLORTERM=truecolor` env signal | ~2016 (Alacritty, Kitty conventions) | $TERM is for capability DB lookup; $COLORTERM is the truecolor signal. |
| Unconditional 24-bit SGR emission | NO_COLOR-respecting cascade | no-color.org published 2019 | Users expect to disable color; honoring NO_COLOR is table-stakes. |
| Oklab perceptual distance | sRGB Euclidean (for low-color-count targets) | Always acceptable for ≤256 palette | Oklab matters for photorealistic quantization; overkill for ANSI-art. |
| `tput colors` terminfo probe | Env-var cascade | ~2018+ | Terminfo is slow (fork+exec); env vars are instant and cross-platform. |

**Deprecated/outdated:**
- Relying on `TERM=xterm-truecolor` or `TERM=xterm-24bit`: non-standard; never widely adopted. `$COLORTERM` won.
- Probing terminfo at startup: slow + not cross-platform (Windows has no terminfo DB by default). Env cascade is the modern default.

## Open Questions

1. **Does `TerminalGuard` hold the detected `ColorMode` or does `run()` pass it directly to the render loop?**
   - What we know: both are viable; CONTEXT says "cached in `TerminalGuard`" which nudges option A.
   - What's unclear: stylistic preference only.
   - Recommendation: hold it on `TerminalGuard` as `pub color_mode: ColorMode` (simple field), so future code that accesses the guard (panic hook, etc.) can inspect the active mode cheaply.

2. **Should the color-test example produce a stable image for screenshots, or animate?**
   - What we know: CONTEXT says "≤80 LOC target, more mechanical than model-viewer".
   - What's unclear: animating makes visual comparison harder across modes; static is easier.
   - Recommendation: static scene rendered once with a readable header ("mode: Palette256 (auto-detected from $COLORTERM)"). User quits with Ctrl-C. No rotation / no timer. Lean.

3. **Are `ratatui::Buffer.content` iteration semantics stable in 0.30?**
   - What we know: Docs show `buffer[(x, y)].set_style(...)` works; `cell_mut(Position)` is a public API.
   - What's unclear: whether `content` field itself is `pub` in 0.30 (it was in earlier versions).
   - Recommendation: use `Buffer::area()` + nested loop with `cell_mut(Position::new(x, y))` as the stable API. Slightly more verbose than `.content.iter_mut()` but API-stable. Verify both during implementation and pick whichever compiles clean.
