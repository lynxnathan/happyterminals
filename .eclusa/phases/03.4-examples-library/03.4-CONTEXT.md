# Phase 3.4: Examples Library - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning
**Mode:** Auto (recommended defaults selected; hero example = text reveal over 3D)

<domain>
## Phase Boundary

Ship at least 5 runnable examples that collectively demonstrate: 3D rendering, particles, scene transitions, JSON recipe loading, and text/effect composition. Every example carries a header doc-comment (DEMO-05). Every example runs via `cargo run --example <name> -p happyterminals` in a fresh checkout with no extra setup. Covers REL-04, DEMO-05.

This phase is the **demo reel** — the first impression for every future user who lands in the repo. At least one example must land the spatial/3D-in-terminal thesis viscerally (per validated pre-repo reactions: bunny + physics-y particles, friend's DIRTY menu). Coverage matters; a hero moment matters more.

</domain>

<decisions>
## Implementation Decisions

### Example Lineup

- **D-01:** The five headline examples for v1 release are: `model-viewer` (mesh + cameras), `particles` (physics-y particles over a mesh), `transitions` (scene A → B with named effects), `json-loader` (NEW — load a recipe, render it), `text-reveal` (NEW — tachyonfx text effects composed over a 3D scene, the hero).
- **D-02:** `spinning-cube` stays as the M1 onboarding artifact — the ~40-LOC proof-of-stack — but is NOT counted in the 5 headline examples. `color-test` and `static_grid` stay as dev/test utilities, not demo-reel material, and their headers get a one-line "developer utility — not a demo" marker.
- **D-03:** Build order: **text-reveal first** (the hero), then **json-loader** (the shareable), then header polish across all existing examples. Front-loading the hero means the phase ships its strongest signal first.

### Hero Example: text-reveal

- **D-04:** `text-reveal` composes tachyonfx text effects (e.g., `HslShift`, `Sweep`, or `FadeFromFg`) **over a live 3D scene** — not standalone text on a blank grid. The bar is "an unexpected combination": text reveals *through* a rotating mesh or *over* a dissolving cube. This is what makes the "GPU-quality effects on text" story land.
- **D-05:** Scene baseline for text-reveal: start with the bunny mesh (validated reaction source) rotating slowly, overlay reveal-animated text (title + tagline + a ">  press [space] to continue" cue) via tachyonfx, let the text stick around after the reveal completes. Use the existing pipeline + effects machinery — no new rendering primitives.
- **D-06:** Interactive hook: spacebar cycles the text reveal (so the user can replay the effect without restarting) and Tab could swap reveal effect types. Controls documented in header.

### JSON Loader Example: json-loader

- **D-07:** `json-loader` loads `hello.json` (the sample recipe shipped in Phase 3.2) via `load_recipe_sandboxed()` with a `SandboxConfig` whose asset_root points to `examples/models/`. Demonstrates both the happy path (recipe → scene) AND the sandbox (one commented-out line showing a rejected `../../etc/passwd` path, with the error printed to stderr on run — educational).
- **D-08:** No live-editing or file-watching — keep it simple. A second headline recipe can ship (e.g., `transitions.json` loading a 2-scene transition recipe) if natural, but NOT required. The point is "JSON ↔ SceneIr round-trip renders correctly," not "build a JSON editor."

### Model-Viewer, Particles, Transitions (existing)

- **D-09:** Existing code stays; only header doc-comments get revised for consistency with DEMO-05. Do NOT refactor working examples to "harmonize" — that's scope creep.
- **D-10:** Add to `particles` header a note about the zero-per-frame-allocation property (Phase 2.4) — this is a brag that reinforces the engineering discipline story.

### Header Doc-Comment Style (DEMO-05)

- **D-11:** Each example's `main.rs` starts with a `//!` doc-comment block containing, in order:
  1. One-line purpose (what this demonstrates)
  2. "Features exercised:" bulleted list (≤5 bullets, names from the crate surface)
  3. "Controls:" block if interactive (key/mouse → action)
  4. "Run with:" line showing the exact cargo command
  5. Optional "Why this exists:" 1–2 sentences when the example has a narrative worth stating (text-reveal definitely, json-loader yes, spinning-cube yes)
- **D-12:** model-viewer's existing header is the reference shape — mirror its structure across the others.

### Runnability Gate

- **D-13:** The phase's Definition of Done includes `cargo run --example <name> -p happyterminals` succeeding in CI for each of the 5 headline names — `model-viewer`, `particles`, `transitions`, `json-loader`, `text-reveal`. Automated smoke in a headless tty or via a short-duration timeout. Fail-fast if any example panics on startup.

### Claude's Discretion

- Specific tachyonfx effect selection for `text-reveal` (any of HslShift, Sweep, FadeFromFg, Dissolve-on-text — pick what reads best visually in a test run).
- Recipe content in `hello.json` / whether to ship a second recipe for `json-loader`.
- Exact key bindings for interactive controls (following the existing viewer-context pattern from Phase 2.3).
- Whether to surface `transitions.json` as a second recipe demo inside `json-loader` or defer.

</decisions>

<specifics>
## Specific Ideas

- The hero shot for `text-reveal` should be *literally frameable* — the moment text materializes over a rotating bunny is the screenshot that goes on the crates.io README. Design the example so pausing it at ~t=1s produces a good-looking frame.
- `json-loader`'s header comment should include 3–4 lines of example JSON inline so a reader of the source file understands the format without opening the recipe file.
- Every example's `main.rs` should be readable top-to-bottom in under 5 minutes. If an example grows past ~150 LOC, factor helpers into a sibling file rather than expanding main.
- Validation pre-repo: bunny + particle-physics-effects was the unexpected hit. text-reveal + bunny is the v2.0 equivalent — compose the two already-validated primitives.

</specifics>
