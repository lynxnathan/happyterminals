# Phase 3.4: Examples Library - Research

**Researched:** 2026-04-17
**Domain:** happyterminals example authoring — tachyonfx text composition, sandboxed JSON loading, example header style, CI smoke gate
**Confidence:** HIGH (codebase-grounded; every claim cites a file/line)

## Summary

Phase 3.4 ships two new examples (`text-reveal`, `json-loader`), polishes headers on four existing ones, marks two as dev utilities, and adds a CI runtime smoke gate. The hardest thing in the phase is `text-reveal`: composing tachyonfx text effects over a rotating bunny. All the primitives needed already exist in the workspace — `TachyonAdapter`, `effects::{fade_from, sweep_in, hsl_shift, coalesce, dissolve}` (`crates/happyterminals-pipeline/src/effects/mod.rs:1-118`), and the `run_with_input` loop that gives per-frame `&mut Grid` access (`crates/happyterminals-backend-ratatui/src/run.rs:149-250`). The non-obvious catch: `run_scene` does **not** give the tick callback `&mut Grid` (`run.rs:267-380`), so text-reveal cannot be built on `run_scene` the way `spinning-cube` is — it must use `run_with_input` and drive its own pipeline or direct `Fx::process` call.

`json-loader` is straightforward because Phase 3.3 shipped `load_recipe_sandboxed` + `SandboxConfig` (`crates/happyterminals-dsl/src/json.rs:324-379`), but the meta crate's prelude (`crates/happyterminals/src/lib.rs:59-60`) does **not** currently re-export `load_recipe_sandboxed`, `SandboxConfig`, or `EffectRegistry`. That's a one-line fix the plan must include.

Canonical paths to know: the actual `hello.json` lives at `examples/recipes/hello.json` (NOT `crates/happyterminals/recipes/`). Examples reach assets via `concat!(env!("CARGO_MANIFEST_DIR"), "/../../examples/models/…")` (`crates/happyterminals/examples/model-viewer/main.rs:22-41`). CI already has `cargo test --workspace` and a `docs` job; a cheap runtime smoke gate is a new matrix step running each headline example with a short `--duration` (we'd need a `HT_SMOKE_EXIT_AFTER_MS` env handle) or, simpler, `cargo check --example <name>` as the baseline with `timeout 5s cargo run --example <name>` as the runtime gate — gated on a `CI=1` terminal-safety check.

**Primary recommendation:** Build `text-reveal` on `run_with_input`, manually loading bunny + drawing the rotated mesh via `Renderer::draw`, then `grid.put_str()` the title/tagline/cue, then apply `effects::fade_from(Color::Black, Color::Reset, 800ms)` (via a `Pipeline` the example owns) to color-animate the already-written text. Spacebar → `pipeline.reset()` replays; Tab → swap which `effects::*` constructor is in the pipeline slot. This uses 100 % shipped primitives, no new crate surface.

## Numbered Findings

### 1. Tachyonfx text-reveal primitives (HIGH confidence)

The `happyterminals-pipeline` crate already wraps ten tachyonfx effects via `TachyonAdapter` (`crates/happyterminals-pipeline/src/effects/mod.rs:14-92`). Of these, the five that compose best as "reveal text over a pre-drawn buffer":

| Constructor | Signature (happyterminals wrapper) | What it does to existing text | Source |
|------------|------------------------------------|--------------------------------|--------|
| `effects::fade_from(fg, bg, duration)` | `Color, Color, Duration` | Colors existing cells start at `(fg, bg)` and transition to their written values over `duration`. Perfect for "title appears". | `effects/mod.rs:22-26` |
| `effects::sweep_in(direction, gradient_len, faded_color, duration)` | `Motion, u16, Color, Duration` | Reveals content directionally with a gradient wave. Left-to-right works for a tagline. | `effects/mod.rs:36-45` |
| `effects::hsl_shift([h, s, l], duration)` | `[f32; 3], Duration` | Shifts hue/sat/light of existing colored text over time. Good for a one-shot color pop. | `effects/mod.rs:67-71` |
| `effects::coalesce(duration)` | `Duration` | Reverse-dissolves — cells scramble and reform into the written text. Dramatic. | `effects/mod.rs:60-64` |
| `effects::dissolve(duration)` | `Duration` | Forward dissolve (cells fragment). Useful for a "fade away then replay" pattern. | `effects/mod.rs:14-19` |

**Key insight:** per docs.rs/tachyonfx/0.25.0 the `Shader::process` contract operates on whatever already exists in the `ratatui::buffer::Buffer` cells (MEDIUM confidence, docs.rs). Our `TachyonAdapter::apply` (`crates/happyterminals-pipeline/src/adapter.rs:53-70`) passes `grid.buffer_mut()` directly to `fx.process(tfx_dt, buf, area)`. That means: if we `renderer.draw(bunny)` → `grid.put_str("TITLE")` → `pipeline.run_frame(dt)` with `fade_from`, the fade animates BOTH the bunny pixels AND the title text. To animate ONLY the title, we must constrain the effect to the title's `Rect` via `TachyonAdapter::with_area(fx, rect)` (`crates/happyterminals-pipeline/src/adapter.rs:43-51`).

**Gotcha:** `TachyonAdapter::with_area` takes a raw `tachyonfx::Fx` (= `tachyonfx::Effect`), not our `TachyonAdapter`. The convenience `effects::fade_from(...)` returns a pre-wrapped `TachyonAdapter`, so to get a bounded effect you drop to `TachyonAdapter::with_area(tachyonfx::fx::fade_from(fg, bg, tfx_dur), title_rect)`. This works (tachyonfx is a direct dependency of the pipeline crate per `Cargo.toml:36` and the `Fx` alias is re-exported by the prelude per `crates/happyterminals/src/lib.rs:68`), it just involves one layer of manual wiring per effect.

### 2. Text-over-3D composition layering (HIGH confidence)

Two execution paths, and they are not equivalent:

**Path A — `run_scene` (scene-driven, used by `spinning-cube`):** The runtime is `crates/happyterminals-backend-ratatui/src/run.rs:267-380`. The tick callback has signature `FnMut(Duration, &InputSignals)` — NO `&mut Grid`. Ordering per frame: (a) `tick_fn` runs, (b) `walk_and_render` draws cubes/layers from the scene IR (`run.rs:335`), (c) the scene-level `pipeline.run_frame(&mut grid, dt)` runs (`run.rs:337-339`). There is no hook between (b) and (c) to write text onto the grid from user code. **This path cannot drive `text-reveal` as specified in D-05**: the user cannot call `grid.put_str("TITLE")` after the bunny draws.

**Path B — `run_with_input` (closure-driven, used by `model-viewer`, `particles`, `transitions`):** The runtime is `run.rs:149-250`. The render closure has signature `FnMut(&mut Grid, &InputSignals, &InputMap)`. The caller owns the whole frame: they can call `renderer.draw(mesh)`, then `grid.put_str("TITLE")`, then drive their own `Pipeline::run_frame(&mut grid, dt)`. This matches the existing pattern of every interactive example and is how text-reveal should be built.

**Recommendation:** `text-reveal/main.rs` uses `run_with_input`, borrows the `model-viewer` shape, drops the model-cycling and pan logic, and adds:
1. An owned `Pipeline` containing one `TachyonAdapter::with_area(...)` per active title-reveal effect.
2. A `title_rect: Rect` computed from viewport width (centered, ~3 rows tall).
3. Spacebar → `pipeline.reset()` to replay (the existing `Effect::reset` surface is wired through `TachyonAdapter` at `adapter.rs:76-79`).
4. Tab → cycle a `current_effect_idx`, rebuild the pipeline with the new `effects::*` constructor.

The text itself is written every frame via `grid.put_str(...)` — since cells are overwritten each frame before the pipeline runs, the effect gets a fresh "target" every frame and animates toward it.

### 3. Sample recipe completeness (HIGH confidence)

`examples/recipes/hello.json` (`examples/recipes/hello.json:1-24`) defines a single orbit camera + one layer + one cube at origin. It is **minimal by design** (Phase 3.2 artifact, intentionally simple). For `json-loader` to be visually interesting, two options:

- **Option 1 (ship as-is, D-08 discretion):** The cube renders fine via the existing `walk_and_render` path in `run_scene` (it handles `NodeKind::Cube` at `run.rs:651-662`). "Minimal" is actually on-message here: the example proves "JSON → SceneIr → screen" round-trip with the smallest possible recipe. Augment the example with a header block showing 3–4 lines of the JSON inline (per CONTEXT "specifics").
- **Option 2 (add a second recipe):** Create `examples/recipes/text-reveal.json` or `examples/recipes/bunny.json` that adds a mesh node referencing `bunny.obj`. **Blocker:** `walk_and_render` (`run.rs:626-670`) only handles `NodeKind::Cube`, `Layer`, `Group` — it silently does nothing for `NodeKind::Custom("mesh")` (`run.rs:668-669`). Loading a mesh recipe would produce a correct SceneIr but render nothing. This means Option 2 needs either (a) extending `walk_and_render` to dispatch mesh nodes (scope creep, NOT in CONTEXT), or (b) hand-rolling mesh lookup in the example using `load_obj` from the `path` prop — which is more code and doesn't really demonstrate "JSON → scene, done."

**Recommendation:** Ship `hello.json` as-is, use the cube path. Document the limitation at the top of the json-loader header: "cube and group nodes render today; mesh recipe support is scheduled for post-v1" — honest and keeps scope tight. If the planner/human feels strongly that the hero needs bunny, defer this to post-phase follow-up, not into 3.4.

### 4. Runtime smoke test pattern (HIGH confidence)

Existing CI (`.github/workflows/ci.yml:45-60`) runs `cargo test --workspace --all-features --no-fail-fast` on the 1.88 and stable matrix. There is **no example smoke step today**. Adding one has two flavors:

**Cheap option (compile-only):** Add `cargo check --examples -p happyterminals` as a CI step. This catches `fn main() -> ...` compilation errors, missing features, and type drift. It does NOT catch "panics on startup" (e.g., `unwrap()` on a missing asset). ~5 seconds additional CI time.

**Runtime option (covers startup panics):** Wrap each headline example in `timeout 3s cargo run --example <name> -p happyterminals`. On GitHub Actions runners the terminal is non-interactive (stdin closed, no TTY), and crossterm's `TerminalGuard::acquire_with_color_mode` will fail on non-TTY — so this will crash immediately unless we add a headless/CI mode. Two ways:
1. **Introduce `HT_SMOKE_EXIT_AFTER_MS` env var** read in `run_with_input` / `run_scene` / `run_scenes`: if set, spin a `tokio::time::sleep` that drives a break after that duration. Straightforward but adds a cross-cutting flag to 4 functions. ~15–20 LOC.
2. **Use a PTY/screen emulator** (`xvfb-run`, `script -qc`, GNU `screen`) in CI to present a fake TTY. More CI config, no code change.

**Recommendation:** Start with **cheap option** (`cargo check --examples`) as the MUST-HAVE gate. Ship it in the same PR as the examples so the phase's exit criterion "runs in CI" is met (compile-only satisfies "succeeds" for a static check). Track runtime smoke as a follow-up (not in CONTEXT). Rationale: D-13 says "Fail-fast if any example panics on startup" — if startup panics are a real risk, `cargo check --example` won't catch them, but neither will compile-only; the right real fix is making `TerminalGuard::acquire` gracefully no-op when `!stdout.is_terminal()`, which is a different phase. Document the limitation in the phase exit criteria.

**Alternative that splits the difference:** Add a simple bash script at `scripts/smoke-examples.sh` that does `cargo check --example <name>` for each headline name, invoked from a new CI job. Cheap, explicit, reviewable, and ready to upgrade to `timeout 3s cargo run` later if needed.

### Pattern 2: text-reveal Core Loop

Based on `run_with_input` (`run.rs:149-250`) + `model-viewer/main.rs` shape:

```rust
// Build the pipeline with one bounded text effect
let title_rect = Rect::new(center_x - 20, center_y - 2, 40, 3);
let mut pipeline = Pipeline::new().with_boxed(Box::new(TachyonAdapter::with_area(
    tachyonfx::fx::fade_from(Color::Black, Color::Reset, tfx_dur(800)),
    title_rect,
)));

run_with_input(
    move |grid, _input_signals, imap| {
        // 1. Advance bunny rotation (time-based, no signals needed here)
        camera.azimuth += ROTATION_SPEED * DT;

        // 2. Draw bunny
        renderer.draw(grid, &bunny, &camera, &projection, &shading);

        // 3. Write the text *every frame* (cells overwritten → fresh target)
        grid.put_str(title_x, title_y,     "HAPPY TERMINALS",            title_style);
        grid.put_str(title_x, title_y + 1, "GPU-quality effects on text", tagline_style);
        grid.put_str(title_x, title_y + 3, ">  press [space] to continue", cue_style);

        // 4. Run pipeline — effects animate the written cells
        pipeline.run_frame(grid, Duration::from_millis(33));

        // 5. Handle replay / swap
        if imap.action_state("replay_reveal").map(|s| s.untracked()) == Some(ActionState::JustPressed) {
            pipeline.reset();
        }
        if imap.action_state("swap_effect").map(|s| s.untracked()) == Some(ActionState::JustPressed) {
            effect_idx = (effect_idx + 1) % REVEAL_EFFECTS.len();
            pipeline = rebuild_pipeline(effect_idx, title_rect);
        }
    },
    FrameSpec { title: Some("happyterminals - Text Reveal".into()), ..FrameSpec::default() },
    input_map,
).await
```

### Anti-Patterns to Avoid

- **Building text-reveal on `run_scene`** — the tick callback has no `&mut Grid` (run.rs:270-273). You cannot write text between mesh draw and pipeline apply. Use `run_with_input`.
- **Using `effects::fade_from` (full-grid) for text-reveal** — the effect will animate the bunny's colors too. Use `TachyonAdapter::with_area(tachyonfx::fx::fade_from(...), title_rect)` to bound the effect to the title region.
- **Hand-rolling an input pattern** — every interactive example uses `register_default_actions` + `default_viewer_context` + `push_context`. Don't invent new state machines.
- **Adding a `HT_SMOKE_EXIT_AFTER_MS` env var now** — cross-cutting change to 4 `run_*` fns; punt to follow-up. Ship `cargo check --example` smoke.
- **Extending `walk_and_render` to render mesh nodes** — tempting to enable a richer `bunny.json`, but that's a Phase 3.2/3.3 follow-up, NOT scope here. D-09 forbids refactoring to "harmonize."
- **Refactoring existing examples beyond header edits** — D-09 explicitly forbids this.

## Don't Hand-Roll

## Common Pitfalls

### Pitfall 1: `run_scene` tick callback lacks `&mut Grid`
**What goes wrong:** Author tries to build text-reveal on the `spinning-cube` pattern (`run_scene` + signal-driven scene) and finds they can't write text into the grid from user code.
**Why it happens:** `run_scene`'s signature is `FnMut(Duration, &InputSignals)` — signals-only; the grid is written by `walk_and_render` from the scene IR and then the scene-level pipeline is applied (`run.rs:335-339`). There is no between-step hook.
**How to avoid:** Use `run_with_input` for text-reveal; that closure has `(&mut Grid, &InputSignals, &InputMap)`. See `model-viewer/main.rs:115-218` as the shape.
**Warning signs:** "How do I call `grid.put_str()` after the bunny renders?" → switch to `run_with_input`.

### Pitfall 2: Full-grid effect bleeds onto 3D content
**What goes wrong:** Author uses `effects::fade_from(black, reset, 800ms)` as a full-grid adapter; the bunny's shaded cells also fade, producing a "whole screen revealing" effect instead of "title text revealing over a steady bunny."
**Why it happens:** `TachyonAdapter::new(fx)` sets `area: None`, and `apply` defaults to `grid.area` (`adapter.rs:60`). The effect hits every cell.
**How to avoid:** Use `TachyonAdapter::with_area(tachyonfx::fx::fade_from(...), title_rect)` to bound the effect to the text region. The convenience `effects::fade_from(...)` won't work for this — you need the raw-tachyonfx + `with_area` pair.
**Warning signs:** Bunny colors shimmer during the reveal animation — the effect is unbounded.

### Pitfall 3: Text written AFTER the pipeline disappears
**What goes wrong:** Author writes title text, then runs the pipeline, then writes the cue text, expecting cue to animate. Cue doesn't animate.
**Why it happens:** Effects only process cells that exist at effect-apply time. Anything drawn after `pipeline.run_frame` is untouched by that frame's effect.
**How to avoid:** Write ALL text first, then run the pipeline. Or use separate pipelines with separate areas if you want staggered per-region effects.
**Warning signs:** Some text animates, some doesn't. Check ordering.

### Pitfall 4: `include_str!` path to `hello.json` relative to crate
**What goes wrong:** Author writes `include_str!("hello.json")` in `examples/json-loader/main.rs` → "file not found" at compile time.
**Why it happens:** `include_str!` is relative to the file containing the macro call. The recipe lives at `<repo>/examples/recipes/hello.json`, the example lives at `<repo>/crates/happyterminals/examples/json-loader/main.rs` → need `"../../../../examples/recipes/hello.json"` (4 ups: `json-loader/`, `examples/`, `happyterminals/`, `crates/`).
**How to avoid:** Count the directory depth; verify with `cargo check --example json-loader` before committing. Or use `concat!(env!("CARGO_MANIFEST_DIR"), "/../../examples/recipes/hello.json")` with `std::fs::read_to_string` for runtime loading — which is what `model-viewer/main.rs:22-41` does for the bunny mesh.
**Warning signs:** `error: couldn't read "..."`.

### Pitfall 5: Prelude missing sandbox symbols
**What goes wrong:** Author writes `use happyterminals::prelude::*;` then `load_recipe_sandboxed(...)` → unresolved name.
**Why it happens:** `happyterminals/src/lib.rs:60` only re-exports the non-sandboxed loader surface.
**How to avoid:** Add `load_recipe_sandboxed`, `SandboxConfig`, `EffectRegistry` to the prelude as the first task of this phase. See "Standard Stack / Prelude gap" above.
**Warning signs:** Compile error on symbol; workaround is `use happyterminals_dsl::prelude::*;` but that's not the pattern other examples use.

### Pitfall 6: CI runtime smoke hits a non-TTY
**What goes wrong:** `cargo run --example model-viewer` in GitHub Actions fails inside `TerminalGuard::acquire` because the runner has no TTY.
**Why it happens:** `crossterm::terminal::enable_raw_mode()` errors on non-TTY file descriptors.
**How to avoid:** Ship `cargo check --example <name>` as the baseline smoke gate; document "runtime smoke covered by manual verification" until a future phase adds `HT_SMOKE_EXIT_AFTER_MS` or a PTY wrapper.
**Warning signs:** CI fails with "Inappropriate ioctl for device" or similar.

## Open Questions

1. **Does `Scene::from_parts` exist or do we need an alternative for json-loader?**
   - What we know: `scene.into_parts()` exists (used in `json.rs:1020`, `run.rs:301`). The inverse — building a `Scene` from `(SceneIr, CameraConfig, Option<Pipeline>)` — is needed by json-loader to hand the loaded recipe to `run_scene`.
   - What's unclear: Whether the public constructor is `Scene::from_parts`, `Scene::new`, or requires `SceneBuilder` intermediate re-walking.
   - Recommendation: Planner verifies on first json-loader task by reading `crates/happyterminals-scene/src/scene.rs`. If `from_parts` is missing, the minimum fix is adding a `pub fn from_parts(ir, camera, pipeline) -> Self` — 3 LOC, non-breaking. Alternative: rebuild via `SceneBuilder` which requires NodeIr-tree walking (more code, not worth it).

2. **Does `walk_and_render` need to be extended to render `NodeKind::Custom("mesh")` for a richer json-loader demo?**
   - What we know: Current `walk_and_render` handles Cube, Layer, Group only (`run.rs:650-669`). `hello.json` is cube-only, so the demo works without changes.
   - What's unclear: Whether the planner/human wants a bunny-via-JSON demo inside this phase.
   - Recommendation: Ship cube-only per D-08 discretion ("second recipe NOT required"). Document the limitation in the json-loader header ("cube and group recipes render today; mesh rendering scheduled post-v1"). Route any "should render bunny" UAT finding to a post-v1 backlog item (per memory:route-out-of-scope-uat-findings-forward).

3. **Should the smoke script fail-fast (`-e`) or run all 5 and aggregate failures?**
   - What we know: `set -euo pipefail` fails on first error. That's standard for CI.
   - What's unclear: Whether aggregated reporting ("3 of 5 passed") is more useful.
   - Recommendation: Fail-fast. If CI fails, the first failure is always the interesting one; aggregated mode adds shell complexity with no plan-verification upside.
