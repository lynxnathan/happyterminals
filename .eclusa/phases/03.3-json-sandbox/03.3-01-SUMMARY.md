## Accomplishments
- `EffectRegistry` with three built-in names (`dissolve`, `slide-left`, `fade-to-black`) and a `register()` hook; unknown names surface as `RecipeError::UnknownEffect`
- `sanitize_path` rejects `..`, `.`, absolute paths (POSIX and Windows), Windows drive letters, backslash-hidden traversal, and empty strings — all before any file I/O

- Prelude re-exports `load_recipe_sandboxed`, `SandboxConfig`, and `EffectRegistry` so downstream code gets the safe API with a single `use happyterminals_dsl::prelude::*`

## Decisions Made

- **Pure-string path validation.** `canonicalize()` was considered and rejected: it requires the file to exist and follows symlinks, both of which work against us here. A pre-I/O string check keeps the sandbox honest.
- **Byte-level ANSI scanner over regex.** The sandbox is a security primitive that must not bring in surprise dependencies. A ~50-line state machine handles CSI/OSC/simple escapes without `regex` or `vte`.
- **Two-loader API (load_recipe + load_recipe_sandboxed).** Trusted recipes produced by `scene_ir_to_recipe` round-trip through `load_recipe` unchanged; the sandbox overhead is only paid by untrusted input. This preserved all 21 existing tests and the hello.json integration unchanged.
- **Strip ANSI from mesh paths before sandboxing.** If a recipe stored `"\u001b[2J../etc/passwd"` the validator would see the ESC bytes and miss the traversal. Running `strip_ansi` first exposes the `..` to `sanitize_path` — verified by the `sandbox_strips_ansi_from_mesh_path` test.
- **Deep sanitization of prop JSON values.** Recipes can store arbitrary JSON in props; strings nested inside arrays and objects are sanitized via `sandbox_json_value`.
- **EffectRegistry plumbed even without an effect-name field.** The registry lives on `SandboxConfig` today so the next recipe revision (e.g., scene-level transition effect names) slots in without redesigning the sandbox.

## Deviations from Plan

None — plan executed exactly as written.

Two minor extras were added that align with the plan's intent (covered by Rule 2, auto-add missing critical functionality):

- **Backslash separator rejection in `sanitize_path`.** The plan said "Split on `/` and reject if ANY component is `..` or `.`", but a recipe like `"subdir\..\bunny.obj"` would have slipped through a forward-slash-only split on POSIX. I split on both `/` and `\` and added the `sandbox_path_rejects_backslash_traversal` test.
- **Deep sanitization of nested JSON in props.** The plan said "strips ANSI from all string prop values". I treated that as including strings nested inside arrays/objects (a single-layer strip would leak escapes in `tags: ["\u001b[31mred\u001b[0m"]`). Added `sandbox_strips_ansi_from_nested_prop_array` to lock the behavior in.

Both are correctness extensions of the stated behavior rather than new features.

## Issues Encountered

- **Raw ESC bytes in `format!`-built JSON strings fail `serde_json`.** My initial `sandbox_strips_ansi_from_mesh_path` test used `"\u{001b}[2J..."` in a Rust string and inlined it via `format!`, which sent raw `0x1B` bytes into `serde_json::from_str` — `serde_json` correctly rejects unescaped control characters. Fixed by writing the JSON as a raw string literal with `\u001b` (the JSON Unicode escape), which the parser decodes to ESC. One iteration, one failing test converted to passing.
- **Pre-existing workspace clippy failures.** `cargo clippy -p happyterminals-dsl --all-targets` reports 60+ errors from `unwrap_used = "deny"` applied to test code that was written before this plan. These are pre-existing tech debt, out of scope for 03.3-01 per the Eclusa scope-boundary rule. My new test code follows the same convention as the existing tests in `json.rs` (`.unwrap()`, `.unwrap_err()`) so it doesn't change the overall count materially. One `unwrap_err()` I added in `sandbox.rs` was rewritten to a `let Err(err) = ... else { panic!() }` pattern during authoring.
