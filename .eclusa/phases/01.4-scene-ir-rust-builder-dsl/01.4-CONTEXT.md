## Phase Boundary

**Goal:** Build `happyterminals-scene` (SceneIr, SceneGraph, layered z-order, Camera-on-Scene) and `happyterminals-dsl` (Rust builder producing SceneIr, react-three-fiber-shaped tree with signal-bindable props). This is the first external-facing API — the shape users see when they learn the library. The spatial composition model that makes terminal UIs navigable.

**In scope:**
- `happyterminals-scene`: SceneIr, SceneNode, SceneGraph, Layer with z-order, Camera ownership
- `happyterminals-dsl`: Rust builder DSL producing SceneIr, chainable props, signal/memo/value props
- Fine-grained prop subscription (signal-level reactivity on scene node props)
- TransitionManager scaffold (type + owner-disposal, full transitions in M3)
- Scene validation at build time (Result<Scene, SceneError>)
- Hello world ≤ 25 lines
- Round-trip snapshot test + build-time error test

<decisions>
## Implementation Decisions (locked)

### SceneIr structure (SCENE-01, SCENE-02)
- **D-01:** SceneIr is a tree of SceneNodes. Each node has: an ID (u64 or string), a kind (Cube, Layer, Group, Custom), transform (position/rotation/scale as Vec3), props (type-erased map), children (Vec<SceneNode>), and an optional Pipeline.
- **D-02:** [auto] Layers are SceneNodes with an explicit z-order field (i16). The SceneGraph sorts layers by z-order for render ordering. This matches the game engine compositor model — layers are spatial planes in the terminal.
- **D-03:** Camera is owned by the Scene (not global). One camera per scene. The builder's `.camera(orbit_camera)` sets it. Renderer reads camera from SceneIr.
- **D-04:** TransitionManager scaffold: a `TransitionManager` struct exists with `transition_to(new_scene)` that disposes the old scene's Owner (cleaning up signals/effects). Actual transition effects deferred to M3.

### Prop binding model (SCENE-03, SCENE-05)
- **D-05:** Props can be `PropValue::Static(T)`, `PropValue::Signal(Signal<T>)`, or `PropValue::Memo(Memo<T>)`. The `PropValue` enum is generic over T using type erasure (Box<dyn Any + Send + Sync>).
- **D-06:** [auto] Fine-grained subscription: when a scene renders, each node's signal/memo props are read inside an Effect. When a signal changes, only that node's Effect re-runs, invalidating only that node's render state — not the whole scene. This is the SolidJS model adapted to scene nodes.
- **D-07:** Invalid signal bindings (referencing a signal from a disposed Owner) fail at `build()` time with `SceneError::InvalidBinding { node_id, prop_name, reason }`.

### Builder DSL (DSL-01, DSL-02)
- **D-08:** R3F-shaped tree builder with chainable methods:
  ```rust
  scene()
      .camera(orbit_camera)
      .layer(0, |l| l
          .cube()
              .rotation(&rotation_signal)
              .position(vec3(0., 0., 0.))
      )
      .effect(fx::vignette(0.3))
      .build()?
  ```
- **D-09:** [auto] Each node type (cube, layer, group) is a method on the parent builder. Props are methods on the node builder. Signal props accept `&Signal<T>` references. Plain values accept `impl Into<T>`. This gives the "components-as-methods, props-as-chainable-calls" feel of R3F.
- **D-10:** Hello world ≤ 25 lines including imports. The prelude re-exports everything needed: `scene()`, `run()`, node builders, `Signal`, `vec3`, `FrameSpec`, etc.

### Scene → Renderer bridge (SCENE-01)
- **D-11:** [auto] `run()` in the backend crate accepts a `Scene` (built from the DSL). Each frame tick: iterate SceneIr nodes, for each Cube node call `Renderer::draw()` with the node's transform, for each Layer respect z-order, apply Pipeline effects after all nodes render. The bridge is thin — SceneIr is a data structure the runtime walks.
- **D-12:** In Phase 1.5, `run()` signature changes from `FnMut(&mut Grid, &InputSignals)` to accepting a `Scene`. Phase 1.4 provides the Scene type; 1.5 wires it into run().

### Quality (DSL-03)
- **D-13:** All errors are Result-based. `SceneError` enum covers: InvalidBinding, MissingCamera, EmptyScene, DuplicateId. `clippy::unwrap_used` and `clippy::expect_used` denied in both crates.

### Claude's Discretion
- Exact SceneNode ID type (u64 vs String vs newtype)
- Whether to use a `NodeBuilder<T>` generic or concrete builders per node type
- Internal representation of the prop map (HashMap<String, Box<dyn Any>> vs typed struct per node kind)
- Whether `scene()` returns a `SceneBuilder` or is a function that takes a closure
- How the TransitionManager scaffold handles Owner disposal timing
- Whether to add a `text()` node type for in-scene text (useful for the hello-world demo)

</decisions>
