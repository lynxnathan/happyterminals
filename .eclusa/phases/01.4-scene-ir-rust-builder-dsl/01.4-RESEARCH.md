# Phase 01.4: Scene IR + Rust Builder DSL - Research

**Researched:** 2026-04-15
**Domain:** Scene graph IR, type-erased reactive props, Rust builder DSL
**Confidence:** HIGH

## Summary

Phase 1.4 builds two new crates (`happyterminals-scene` and `happyterminals-dsl`) from Phase 0 stubs. The scene crate defines SceneIr -- a tree of SceneNodes with type-erased, signal-bindable props, layered z-order, and camera ownership. The DSL crate provides an R3F-shaped builder that produces SceneIr with chainable methods and validates at build time via `Result<Scene, SceneError>`.

The core technical challenges are: (1) designing a `PropValue` enum that type-erases `Signal<T>`, `Memo<T>`, and plain `T` while remaining `Send + Sync`-compatible for future JSON/Python frontends, (2) building a nested tree builder that feels like react-three-fiber's JSX but in Rust method chains, and (3) wiring fine-grained reactivity so that each node's signal props are read inside an Effect for per-node invalidation. All reactive primitives (Signal, Memo, Effect, Owner) already exist and are battle-tested (133 passing tests).

**Primary recommendation:** Use a tree-of-owned-children `SceneNode` struct with `HashMap<String, PropValue>` for the prop map, `NodeId(u64)` newtype with atomic counter for IDs, and a consuming-self builder pattern (`fn position(mut self, v: impl Into<Vec3>) -> Self`) for the DSL. Keep `bon` out of this -- the DSL shape is too custom for derive macros. SlotMap is not needed; the tree is short-lived (built once, walked each frame) and owned children are simpler.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- D-01: SceneIr is a tree of SceneNodes. Each node has: an ID (u64 or string), a kind (Cube, Layer, Group, Custom), transform (position/rotation/scale as Vec3), props (type-erased map), children (Vec<SceneNode>), and an optional Pipeline.
- D-02: Layers are SceneNodes with an explicit z-order field (i16). The SceneGraph sorts layers by z-order for render ordering.
- D-03: Camera is owned by the Scene (not global). One camera per scene. The builder's `.camera(orbit_camera)` sets it.
- D-04: TransitionManager scaffold: a TransitionManager struct exists with `transition_to(new_scene)` that disposes the old scene's Owner. Actual transition effects deferred to M3.
- D-05: Props can be PropValue::Static(T), PropValue::Signal(Signal<T>), or PropValue::Memo(Memo<T>). Type-erased via Box<dyn Any + Send + Sync>.
- D-06: Fine-grained subscription: each node's signal/memo props read inside an Effect. Signal change invalidates only that node.
- D-07: Invalid signal bindings fail at build() time with SceneError::InvalidBinding { node_id, prop_name, reason }.
- D-08: R3F-shaped tree builder with chainable methods (see CONTEXT.md for exact syntax example).
- D-09: Each node type is a method on the parent builder. Props are methods on the node builder. Signal props accept &Signal<T>, plain values accept impl Into<T>.
- D-10: Hello world <= 25 lines including imports. Prelude re-exports everything needed.
- D-11: run() accepts a Scene. Each frame: iterate SceneIr nodes, for Cube call Renderer::draw(), for Layer respect z-order, apply Pipeline effects.
- D-12: Phase 1.5 wires Scene into run(). Phase 1.4 provides the Scene type only.
- D-13: All errors are Result-based. SceneError enum covers InvalidBinding, MissingCamera, EmptyScene, DuplicateId. clippy::unwrap_used and clippy::expect_used denied.

### Pattern 1: NodeId Newtype with Atomic Counter

**What:** A `NodeId(u64)` newtype with a thread-local atomic counter for auto-generation, plus an optional debug name.

**When to use:** Every SceneNode gets a unique ID. Users can optionally name nodes for debugging.

**Example:**
```rust
use std::sync::atomic::{AtomicU64, Ordering};

static NEXT_ID: AtomicU64 = AtomicU64::new(1);

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct NodeId(u64);

impl NodeId {
    pub fn next() -> Self {
        Self(NEXT_ID.fetch_add(1, Ordering::Relaxed))
    }
}
```

**Rationale:** u64 is faster than String for HashMap keys and comparison. The atomic counter ensures uniqueness across the process. The `DuplicateId` error in SceneError catches user-assigned collisions if we later allow manual IDs.

### Pattern 2: PropValue Type-Erasure

**What:** An enum that wraps Static values, Signal references, and Memo references behind `Box<dyn Any + Send + Sync>`.

**When to use:** Every node prop that can be either a constant or a reactive binding.

**Critical design insight:** Signal<T> and Memo<T> are `!Send + !Sync` (they use Rc, RefCell, thread-local storage). But D-05 says `Box<dyn Any + Send + Sync>`. This is a contradiction that must be resolved.

**Resolution:** The PropValue enum should NOT require Send + Sync on the inner type. The type-erasure is for heterogeneous storage (different T types in one map), not for cross-thread transfer. SceneIr itself is `!Send + !Sync` because it holds Signal/Memo references. The `Send + Sync` requirement from ARCHITECTURE.md was for the JSON/Python IR path where signals are referenced by name (BindIr::Signal(SignalRef)), not by value. For the Rust builder path, props hold actual Signal<T> references which are !Send.

**Example:**
```rust
use std::any::Any;

/// A scene node property value -- either a static value or a reactive binding.
pub enum PropValue {
    /// A constant value, set once at build time.
    Static(Box<dyn Any>),
    /// A signal binding -- read inside an Effect for fine-grained updates.
    Reactive(Box<dyn AnyReactive>),
}

/// Trait for type-erased reactive reads.
pub(crate) trait AnyReactive {
    /// Read the current value as a type-erased Box<dyn Any>.
    /// When called inside an Effect, registers a dependency.
    fn get_any(&self) -> Box<dyn Any>;

    /// Read the current value without subscribing.
    fn get_any_untracked(&self) -> Box<dyn Any>;
}
```

**Why this pattern:** We need to read signal values during rendering without knowing T at the call site. The `AnyReactive` trait lets us store `Signal<f32>` and `Signal<Vec3>` in the same map and read them uniformly during the Effect-driven render pass.

### Pattern 3: Fine-Grained Per-Node Effects

**What:** When a Scene is built, each node that has reactive props gets an Effect that reads those props and marks the node dirty.

**When to use:** This is the core reactivity-to-rendering bridge.

**Example:**
```rust
// During scene construction (inside an Owner scope):
for node in scene_ir.nodes_with_reactive_props() {
    let node_id = node.id;
    let dirty_flags = dirty_flags.clone(); // Rc<RefCell<HashSet<NodeId>>>
    let reactive_props = node.reactive_prop_refs();

    Effect::new(move || {
        // Reading each reactive prop inside this Effect registers deps
        for prop in &reactive_props {
            let _ = prop.get_any(); // triggers subscription
        }
        // Mark this node as needing re-render
        dirty_flags.borrow_mut().insert(node_id);
    });
}
```

**Critical detail:** The Effect runs immediately at creation, which is correct -- it does the initial read to register subscriptions. Subsequent signal changes only trigger re-reads for that specific node's Effect.

### Pattern 4: Consuming-Self Builder with Closure Children

**What:** Builder methods take `self` (not `&mut self`), enabling linear chaining. Child nodes are built inside closures.

**When to use:** The entire DSL surface.

**Example:**
```rust
pub fn scene() -> SceneBuilder {
    SceneBuilder::new()
}

pub struct SceneBuilder {
    camera: Option<CameraConfig>,
    layers: Vec<SceneNode>,
    pipeline: Option<Pipeline>,
}

impl SceneBuilder {
    pub fn camera(mut self, cam: OrbitCamera) -> Self {
        self.camera = Some(CameraConfig::Orbit(cam));
        self
    }

    pub fn layer(mut self, z_order: i16, f: impl FnOnce(LayerBuilder) -> LayerBuilder) -> Self {
        let layer = f(LayerBuilder::new(z_order));
        self.layers.push(layer.into_node());
        self
    }

    pub fn effect(mut self, fx: impl Into<Box<dyn PipelineEffect>>) -> Self {
        self.pipeline
            .get_or_insert_with(Pipeline::new)
            .push(fx.into());
        self
    }

    pub fn build(self) -> Result<Scene, SceneError> {
        // Validate: camera present, no empty scene, no duplicate IDs
        let camera = self.camera.ok_or(SceneError::MissingCamera)?;
        if self.layers.is_empty() {
            return Err(SceneError::EmptyScene);
        }
        // ... validation pass ...
        Ok(Scene { /* ... */ })
    }
}
```

**Why closures for children:** The closure pattern `layer(z, |l| l.cube().position(...))` matches the R3F JSX nesting feel. The closure receives a fresh builder, the user chains child nodes onto it, and the result is consumed by the parent. This prevents nodes from being added to the wrong parent.

### Pattern 5: Scene Wrapper (Validated IR)

**What:** `Scene` is a validated, sealed wrapper around `SceneIr`. Users cannot construct it directly -- only via `SceneBuilder::build()`.

**When to use:** The `Scene` type is what `run()` accepts. It guarantees invariants.

```rust
/// A validated scene ready for rendering.
///
/// Constructed via [`SceneBuilder::build()`]. Cannot be created directly.
pub struct Scene {
    ir: SceneIr,
    camera: CameraConfig,
    pipeline: Option<Pipeline>,
}

impl Scene {
    pub fn ir(&self) -> &SceneIr { &self.ir }
    pub fn camera(&self) -> &CameraConfig { &self.camera }
}
```

### Anti-Patterns to Avoid
- **Global mutable camera:** Camera must be owned by Scene per D-03. Never use a global `static mut` or `thread_local!` camera.
- **Eager full-scene invalidation:** When one signal changes, do NOT re-render all nodes. The Effect-per-node pattern ensures only dirty nodes are re-processed.
- **Box<dyn Any + Send + Sync> for Signal props:** Signal<T> is !Send. Do not try to make it Send. The SceneIr is !Send by design; it lives on the render thread.
- **String-keyed everything:** Use NodeId(u64) for node identification, not String. String keys are for prop names only (where user-facing naming matters).
- **bon/derive_builder for the DSL:** The builder shape is deeply custom (closures for children, signal refs for props). Derive macros would fight the design, not help it.

## Common Pitfalls

### Pitfall 1: PropValue Send/Sync Confusion
**What goes wrong:** Trying to make PropValue `Send + Sync` because the architecture doc mentions it for the JSON path.
**Why it happens:** D-05 mentions `Box<dyn Any + Send + Sync>` but Signal<T> is `!Send + !Sync`.
**How to avoid:** PropValue uses `Box<dyn Any>` (no Send + Sync bound). The Rust builder path holds actual signal references. The JSON path (Phase 3.2) will use `BindIr::Signal(SignalRef)` -- a string reference resolved at build time -- which IS Send. These are different representations for the same concept.
**Warning signs:** Compiler errors about Signal not implementing Send when stored in PropValue.

### Pitfall 2: Effect Runs Immediately at Creation
**What goes wrong:** The per-node Effect fires during scene construction, before the scene is fully built. If the Effect tries to mark a node dirty in a data structure that doesn't exist yet, it panics.
**Why it happens:** `Effect::new(f)` runs `f` immediately (confirmed in existing code). This is by design for subscription registration.
**How to avoid:** The per-node Effects must be created AFTER the Scene is fully constructed and the dirty-flag data structure exists. The `build()` method should: (1) validate the IR, (2) create the Scene struct with dirty flags, (3) then wire up Effects. Or: defer Effect creation to the first `run()` frame.
**Warning signs:** Panic during `scene().build()` with "borrow already mutably borrowed" or similar.

### Pitfall 3: Hello World Line Count Creep
**What goes wrong:** The "simple" example requires importing 8 types, configuring a camera explicitly, setting up an Owner scope, etc.
**Why it happens:** Rust APIs tend toward explicitness.
**How to avoid:** The prelude must export everything: `scene()`, `run()`, `vec3()`, `Signal`, `FrameSpec`, `OrbitCamera`. Provide sensible defaults: `OrbitCamera::default()` should look reasonable. The builder should not require camera if a default is acceptable (though D-03 says camera is required -- so provide `SceneBuilder::default_camera()` or make camera optional with a sensible default).
**Warning signs:** Count lines in the example. If > 25 including `use happyterminals::prelude::*;`, redesign.

### Pitfall 4: Duplicate ID Detection is O(n^2)
**What goes wrong:** Checking for duplicate IDs by comparing every node to every other node during build().
**Why it happens:** Naive implementation.
**How to avoid:** Use a `HashSet<NodeId>` during the validation pass. Insert each ID; if insert returns false, it's a duplicate. O(n) total.
**Warning signs:** Slow build() on scenes with many nodes.

### Pitfall 5: Owner Disposal Timing in TransitionManager
**What goes wrong:** Disposing the old scene's Owner while Effects are still running causes panics inside reactive_graph.
**Why it happens:** Owner::dispose() cascades cleanup to all Effects/Memos. If an Effect is mid-execution during dispose, the reactive graph is in an inconsistent state.
**How to avoid:** TransitionManager.transition_to() should: (1) stop running the old scene's frame loop, (2) dispose the old Owner on the next frame boundary (not mid-frame), (3) start the new scene. The scaffold in Phase 1.4 only needs the type + dispose-on-drop; full transition sequencing is Phase 3.1.
**Warning signs:** Panics mentioning "already borrowed" or "Owner current not set" during transitions.

### Pitfall 6: Type Erasure Downcasting Failures at Runtime
**What goes wrong:** A prop is stored as `Box<dyn Any>` wrapping `f32`, but the renderer tries to downcast it as `Vec3`. Silent None, or panic.
**Why it happens:** Type erasure loses compile-time type checking.
**How to avoid:** (1) Node kind determines expected prop types -- a Cube's "position" is always Vec3. (2) Downcast failures should produce `SceneError::PropTypeMismatch` at build-time validation, not at render time. (3) Consider a `PropValue::get<T>(&self) -> Option<&T>` helper that returns None on mismatch rather than panicking.
**Warning signs:** Runtime panics with "downcast failed" during rendering.

### Example 3: Layer Z-Order Sorting

```rust
impl SceneGraph {
    /// Returns layers sorted by z-order (lowest first = rendered first = furthest back).
    pub fn sorted_layers(&self) -> Vec<&SceneNode> {
        let mut layers: Vec<&SceneNode> = self.ir.children.iter()
            .filter(|n| matches!(n.kind, NodeKind::Layer { .. }))
            .collect();
        layers.sort_by_key(|n| match n.kind {
            NodeKind::Layer { z_order } => z_order,
            _ => 0,
        });
        layers
    }
}
```

## Open Questions

1. **Should OrbitCamera be wrapped or re-exported?**
   - What we know: OrbitCamera is a plain struct with pub fields (azimuth, elevation, distance, target). D-03 says camera is owned by scene.
   - What's unclear: Should Scene own an OrbitCamera directly, or a CameraConfig enum that can hold different camera types later (orbit, free, FPS)?
   - Recommendation: Use a `CameraConfig` enum with `Orbit(OrbitCamera)` variant. Only one variant now, but it's forward-compatible with REND-05 which mentions free and FPS cameras for v2.

2. **Should scene() return SceneBuilder or take a closure?**
   - What we know: D-08 shows `scene().camera(...).layer(...)` -- a returned builder.
   - What's unclear: Whether `scene(|s| s.camera(...))` would be cleaner for Owner scoping.
   - Recommendation: `scene()` returns `SceneBuilder` (matches D-08 syntax). Owner scoping is the caller's responsibility via `create_root`. This is simpler and matches the locked decision.

3. **Text node type for hello world demo?**
   - What we know: Deferred to "consider in Phase 1.5 if needed for demo."
   - What's unclear: A spinning cube without any text label might be underwhelming.
   - Recommendation: Skip text() in 1.4. The spinning cube demo in 1.5 works without it. If needed, add it there.
