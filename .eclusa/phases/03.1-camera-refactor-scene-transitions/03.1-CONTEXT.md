<decisions>
## Implementation Decisions

### Camera Trait Migration (REND-11)
- **Strategy:** Change `Renderer::draw()` and `Renderer::draw_particles()` signatures from `camera: &OrbitCamera` to `camera: &dyn Camera`. Call `camera.view_matrix()` internally. Update all call sites (examples, tests, run_scene, benches).
- **`run_scene` path:** `CameraConfig::view_matrix()` already delegates polymorphically — no change needed in the scene-driven path. The renderer receives the `Mat4` result.
- **Backward compat:** `&OrbitCamera` implements `Camera`, so existing code like `renderer.draw(&mut grid, &mesh, &camera, ...)` where `camera: OrbitCamera` works via auto-ref to `&dyn Camera`.

### Transition Architecture (SCENE-04, SCENE-06)
- **Location:** `happyterminals-scene/src/transition.rs` — extends the existing `TransitionManager` scaffold.
- **Rendering strategy:** Two-buffer blend. Render Scene A into buffer A, Scene B into buffer B, blend per-cell based on transition progress. The `Grid` already wraps `ratatui::Buffer` which is cloneable — two buffers are cheap.
- **Blend function per transition type:**
  - `dissolve`: interpolate cell chars based on progress (scene A chars at low progress, scene B chars at high progress). At 50% mark, switch char source.
  - `slide-left`: scene A occupies columns 0..w*(1-progress), scene B occupies columns w*(1-progress)..w. Clean column-based split.
  - `fade-to-black`: scene A chars fade to spaces as progress increases, then scene B chars fade in from spaces. Two-phase: 0.0-0.5 = A fades out, 0.5-1.0 = B fades in.
- **Duration + easing:** `Duration` field on the transition + easing function `fn(f32) -> f32`. Ship 3 easings: linear, ease-in-out (cubic), ease-out-cubic. Progress = `easing(elapsed / duration)`.
- **Owner disposal:** When transition COMPLETES (progress >= 1.0), the old scene's `Owner` is disposed synchronously (between frames). The existing scaffold's disposal invariant holds.
- **During transition:** Both scenes are live (both owners exist). Scene A's reactive graph may still fire but its effects are rendered into buffer A which is being blended out.

### Named Transition Registry (SCENE-06)
- **Registry:** `HashMap<String, Box<dyn TransitionEffect>>` on the `TransitionManager`. Built-in: "dissolve", "slide-left", "fade-to-black".
- **`TransitionEffect` trait:** `fn blend(&self, a: &Buffer, b: &Buffer, progress: f32, output: &mut Buffer)` — takes two source buffers and writes the blended result.
- **User extensibility:** Users implement `TransitionEffect` and register via `transition_manager.register("my-effect", MyEffect)`.

### Transition Trigger (SCENE-07)
- **Programmatic:** `transition_manager.transition_to(scene_b, owner_b, "dissolve", Duration::from_millis(500))`.
- **InputMap:** Register `"next_scene"` Bool action. Bind to Tab or N in examples. When `JustPressed`, call `transition_to()`.
- **Example:** A transitions example showing 2-3 scenes that cycle via Tab key with different transition effects.

### Claude's Discretion
- Exact easing function implementations (standard cubic bezier curves).
- Whether to add a `TransitionState` enum (Idle, Transitioning { from, to, progress }) or just use Option fields.
- How to handle `run_scene` integration with transitions (likely a new `run_transition_scene` or extend `run_scene` with optional TransitionManager).
- Test strategy for visual blend correctness (snapshot of mid-transition buffer state).

</decisions>

<specifics>
## Specific Ideas

- The transitions example should show 2-3 distinct scenes (e.g., spinning cube, bunny mesh, particle snow) cycling via Tab with different effects (dissolve between 1→2, slide-left between 2→3, fade-to-black between 3→1).
- Mid-transition snapshot tests can verify that at progress=0.5, the output buffer contains cells from both scenes.
- The two-buffer blend is the same pattern as a GPU compositor's ping-pong buffers — familiar to anyone from graphics programming.

</specifics>

<deferred>
## Deferred Ideas

- **Custom easing curves** (user-defined cubic bezier) — ship the 3 built-ins, add custom in M3+ if needed.
- **Transition chaining** (A→B→C in sequence) — single transition first, chaining is a composition feature.
- **Per-cell transition effects** (dissolve with noise pattern, radial wipe) — the `TransitionEffect` trait supports this but we ship only the 3 simple ones.
- **Audio sync** for transitions — 999.x backlog.

</deferred>
