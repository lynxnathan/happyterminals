# Phase 3.1: Camera Refactor + Scene Transitions - Research

**Researched:** 2026-04-16
**Domain:** Renderer camera abstraction, scene transition blending, ratatui Buffer composition
**Confidence:** HIGH

## Summary

This phase has two distinct workstreams: (1) refactoring `Renderer::draw()` and `draw_particles()` from `&OrbitCamera` to `&dyn Camera`, and (2) building the full `TransitionManager` with two-buffer blend and named transition effects. Both are well-scoped by CONTEXT.md decisions and the existing codebase provides strong foundations.

The camera refactor is mechanically straightforward but has a non-obvious pitfall: `draw()` currently accesses `camera.distance`, `camera.elevation`, `camera.azimuth`, and `camera.target` directly for eye position computation, backface culling direction, and `scene_fit_near()`. The `Camera` trait only exposes `view_matrix() -> Mat4`. The refactor must extract the eye position from the inverse view matrix rather than from OrbitCamera-specific fields, or the Camera trait must be extended. The view-matrix-inverse approach is correct and general.

The transition system uses a two-buffer blend strategy identical to GPU ping-pong compositing. ratatui's `Buffer` is `Clone` (already proven in the `run()` loop via `grid.deref().clone()`), and per-cell access via `cell_mut()` with `symbol`, `fg`, `bg`, `modifier` fields provides everything needed for the blend operations.

**Primary recommendation:** Start with the camera trait migration (smaller, foundational), then build transition infrastructure on top. The `scene_fit_near` function must be generalized to accept any camera (extract eye from inverse view matrix), not just `OrbitCamera`.

## Architecture Patterns

### Camera Refactor: Eye Position from View Matrix

The critical insight is that `draw()` currently computes the eye position and camera direction from OrbitCamera-specific fields:

```rust
// CURRENT (OrbitCamera-specific) -- lines 163-169 of renderer/src/lib.rs
let eye = camera.target
    + Vec3::new(
        camera.distance * camera.elevation.cos() * camera.azimuth.sin(),
        camera.distance * camera.elevation.sin(),
        camera.distance * camera.elevation.cos() * camera.azimuth.cos(),
    );
let camera_dir = (camera.target - eye).normalize();
```

After the refactor, the eye must be extracted from the inverse view matrix:

```rust
// AFTER REFACTOR -- extract eye from view matrix
let view = camera.view_matrix();
let inv = view.inverse();
let eye = inv.col(3).truncate();  // Translation column = eye position
let camera_dir = -Vec3::new(inv.col(2).x, inv.col(2).y, inv.col(2).z); // -Z axis = forward
```

This pattern is already used in the camera tests (e.g., `default_camera_eye_at_positive_z` at line 245) -- confirmed working.

### scene_fit_near Generalization

`scene_fit_near(camera: &OrbitCamera, mesh: &Mesh)` uses `camera.distance`. After refactor, it needs the eye position from the view matrix and a target/center point. Two options:

1. **Compute distance from eye to mesh center:** `(eye - mesh_center).length() - radius` -- more general, works for any camera.
2. **Accept `&Mat4` view matrix instead of camera reference** -- the caller already has it.

Option 1 is recommended: `fn scene_fit_near(view: &Mat4, mesh: &Mesh) -> f32`.

### Two-Buffer Blend Architecture

```
Frame N (during transition):
  1. Render Scene A into buf_a (Grid clone)
  2. Render Scene B into buf_b (Grid clone)
  3. TransitionManager::blend(buf_a, buf_b, progress) -> output Buffer
  4. Apply output to frame
  5. Advance progress: elapsed += dt; progress = easing(elapsed / duration)
  6. If progress >= 1.0: dispose Scene A owner, set Scene B as current
```

### TransitionState Enum (Discretion Recommendation)

**Recommend: Use a `TransitionState` enum** over bare Option fields. It makes the state machine explicit and prevents impossible states:

```rust
pub enum TransitionState {
    Idle {
        scene: Scene,
        owner: Owner,
    },
    Transitioning {
        from_scene: Scene,
        from_owner: Owner,
        to_scene: Scene,
        to_owner: Owner,
        effect_name: String,
        elapsed: Duration,
        duration: Duration,
        easing: fn(f32) -> f32,
    },
}
```

This prevents bugs like "transitioning but no from_scene" or "idle but progress is 0.5".

### run_scene Integration (Discretion Recommendation)

**Recommend: Extend `run_scene` with an optional `TransitionManager`** rather than creating a new function. The run loop already has the frame tick structure; transitions just add a blend step between scene rendering and output. A new `run_scenes` (plural) function that takes `TransitionManager` and an `InputMap` is cleaner than adding more parameters to `run_scene`:

```rust
pub async fn run_scenes<F>(
    transition_manager: TransitionManager,
    spec: FrameSpec,
    input_map: InputMap,
    tick_fn: F,
) -> Result<(), Box<dyn std::error::Error>>
```

This avoids bloating the existing `run_scene` signature which already has a different callback shape.

### Anti-Patterns to Avoid
- **Extending Camera trait with eye()/forward():** Adding methods to `Camera` creates a leaky abstraction. The view matrix IS the camera; eye and forward are derivable from it.
- **Rendering both scenes with the same Renderer instance without clearing:** The z-buffer and cell buffers are shared state. Each scene must get a fresh clear or use separate Renderer instances.
- **Blending at the Grid level instead of Buffer level:** Grid has restricted write access (put_str only). Blend operations must use `buffer_mut()` to get `&mut Buffer` or work directly with `Buffer` clones.

## Common Pitfalls

### Pitfall 1: scene_fit_near Breaks for Non-Orbit Cameras
**What goes wrong:** After changing `draw()` to `&dyn Camera`, `scene_fit_near` still expects OrbitCamera fields. The current shim in `walk_and_render` creates a dummy `OrbitCamera::default()` for non-orbit cameras -- this produces wrong near planes.
**Why it happens:** The function was written for the single-camera era. Distance from eye to mesh center varies by camera type.
**How to avoid:** Refactor `scene_fit_near` to compute distance from the view matrix inverse. Extract eye, compute `(eye - mesh_center).length() - radius`.
**Warning signs:** Near-plane clipping artifacts when using FreeLook or FPS cameras.

### Pitfall 2: Backface Culling Direction from Trait Camera
**What goes wrong:** The `camera_dir` for backface culling is currently computed from OrbitCamera's target-eye vector. With `&dyn Camera`, you must extract the forward direction from the view matrix.
**Why it happens:** `camera_dir` was hand-computed from spherical coordinates.
**How to avoid:** Extract forward as `-view_matrix.col(2).truncate()` (the negated Z-axis of the view matrix in RH coordinate systems, since the camera looks down -Z).
**Warning signs:** Missing or inverted faces after the refactor.

### Pitfall 3: Buffer Area Mismatch During Blend
**What goes wrong:** If buf_a and buf_b have different `area` dimensions (e.g., terminal resized mid-transition), the per-cell blend panics or produces garbage.
**Why it happens:** Each Grid is created from `frame.area()` which can change between frames.
**How to avoid:** At blend time, assert `buf_a.area == buf_b.area == output.area`. On resize during transition, either: (a) abort the transition and snap to Scene B, or (b) re-render both scenes into new-sized buffers before blending.
**Warning signs:** Panic in blend function index arithmetic.

### Pitfall 4: Owner Lifetime During Transition
**What goes wrong:** Scene A's Owner is disposed too early (before transition completes) or too late (leaked).
**Why it happens:** The TransitionState enum must hold both owners during the blend. The old owner must be disposed exactly when progress >= 1.0.
**How to avoid:** The TransitionState::Transitioning variant holds `from_owner: Owner`. On completion, `from_owner.dispose()` is called synchronously between frames (not mid-render). Use `Owner::dispose(self)` which takes ownership, preventing double-dispose.
**Warning signs:** Reactive effects from Scene A firing after transition completes (leak), or effects stopping mid-transition (premature dispose).

### Pitfall 5: Renderer Shared State Between Two Scenes
**What goes wrong:** Using the same `Renderer` instance for both Scene A and Scene B in the same frame means the z-buffer from Scene A interferes with Scene B.
**Why it happens:** Renderer has internal z_buffer and cell_chars that persist across calls.
**How to avoid:** Either (a) use two separate Renderer instances (one per scene during transition), or (b) clear the Renderer's buffers between Scene A and Scene B draws. Option (a) is cleaner. Since Renderer is cheap (just Vec allocations), two instances during transition are fine.
**Warning signs:** Scene B's mesh is occluded by Scene A's z-buffer values.

### Easing Functions

```rust
// Standard cubic easing implementations
pub fn linear(t: f32) -> f32 { t }

pub fn ease_in_out(t: f32) -> f32 {
    if t < 0.5 {
        4.0 * t * t * t
    } else {
        1.0 - (-2.0 * t + 2.0).powi(3) / 2.0
    }
}

pub fn ease_out_cubic(t: f32) -> f32 {
    1.0 - (1.0 - t).powi(3)
}
```

### TransitionEffect Trait and Dissolve Example

```rust
use ratatui_core::buffer::Buffer;

pub trait TransitionEffect: Send {
    fn blend(&self, a: &Buffer, b: &Buffer, progress: f32, output: &mut Buffer);
}

pub struct Dissolve;

impl TransitionEffect for Dissolve {
    fn blend(&self, a: &Buffer, b: &Buffer, progress: f32, output: &mut Buffer) {
        let area = output.area;
        for y in area.y..area.y + area.height {
            for x in area.x..area.x + area.width {
                let pos = ratatui_core::layout::Position::new(x, y);
                let source = if progress < 0.5 { a } else { b };
                if let (Some(src_cell), Some(out_cell)) = (source.cell(pos), output.cell_mut(pos)) {
                    out_cell.set_symbol(src_cell.symbol());
                    out_cell.set_style(src_cell.style());
                }
            }
        }
    }
}
```

### SlideLeft Effect

```rust
pub struct SlideLeft;

impl TransitionEffect for SlideLeft {
    fn blend(&self, a: &Buffer, b: &Buffer, progress: f32, output: &mut Buffer) {
        let area = output.area;
        let w = area.width as f32;
        let split_x = area.x + (w * (1.0 - progress)) as u16;
        for y in area.y..area.y + area.height {
            for x in area.x..area.x + area.width {
                let pos = ratatui_core::layout::Position::new(x, y);
                let source = if x < split_x { a } else { b };
                if let (Some(src_cell), Some(out_cell)) = (source.cell(pos), output.cell_mut(pos)) {
                    out_cell.set_symbol(src_cell.symbol());
                    out_cell.set_style(src_cell.style());
                }
            }
        }
    }
}
```

## Open Questions

1. **Should `scene_fit_near` move to a standalone function or become a Camera trait method?**
   - What we know: Currently a private function in renderer/lib.rs. It's renderer-specific (computes near plane for projection).
   - Recommendation: Keep as a private function but change signature to accept `eye: Vec3` and `mesh: &Mesh`. The caller extracts eye from the view matrix and passes it in. This avoids adding projection-specific methods to the Camera trait.

2. **How should resize during transition be handled?**
   - What we know: Terminal resize causes `grid.resize()` which changes buffer dimensions. If buf_a and buf_b have different sizes, blend is undefined.
   - Recommendation: On resize during transition, skip the blend for that frame (render a blank frame like the existing `resize_pending` path), then re-render both scenes at the new size on the next frame.

3. **Send bound on TransitionEffect trait?**
   - What we know: Owner is `!Send + !Sync`. TransitionManager holds Owners. The trait itself only operates on Buffers (no reactive state).
   - Recommendation: `TransitionEffect: Send` is fine since effects only touch Buffer data. The TransitionManager itself will be `!Send` because it holds Owners.
