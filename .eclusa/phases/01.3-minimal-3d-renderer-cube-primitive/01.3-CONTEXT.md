## Phase Boundary

**Goal:** Ship `happyterminals-renderer` with a z-buffer rasterizer, perspective projection with correct cell-aspect handling, an ASCII shading ramp, a built-in `Cube` primitive, and a signal-driven orbit camera. This is the 3D engine for the terminal — bringing spatial depth to text cells. No OBJ loading yet — that's M2.

**In scope:**
- `happyterminals-renderer` crate: z-buffer, projection, rasterizer, shading, camera, Cube primitive
- Perspective projection with cell aspect ratio as public API parameter (default 2:1)
- Reversed-Z + scene-fit near/far planes (no z-fighting)
- Configurable ASCII shading ramp (default 10 levels ` .:-=+*#%@`)
- Built-in `Cube` primitive with vertices/indices/face normals
- Signal-driven orbit camera (azimuth, elevation, distance)
- Zero heap allocations in draw() hot path (criterion bench)
- Snapshot test of static cube + z-fighting regression test

<decisions>
## Implementation Decisions (locked)

### Math foundation
- **D-01:** [auto] Use `glam` for Vec3/Vec4/Mat4/projection math. Rationale: zero-allocation, SIMD-ready, no_std compatible, widely used in Rust game engines (bevy, wgpu). Add `glam = "0.29"` to workspace deps and renderer crate.
- **D-02:** Cell aspect ratio is a `pub` parameter on the projection (default `2.0` — terminal cells are ~2x taller than wide). The projection matrix scales x by `1/aspect` so a cube looks cubic.

### Z-buffer and rasterization
- **D-03:** Per-pixel z-buffer stored as `Vec<f32>` (reused across frames, no allocation). Reversed-Z: near=1.0, far=0.0 for better float precision at distance.
- **D-04:** [auto] Triangle rasterization via half-space (edge function) method — simple, correct, no scanline edge-case bugs. Cube = 12 triangles (2 per face). Backface culling via face normal dot camera direction.
- **D-05:** Scene-fit near/far planes: renderer auto-computes near/far from the bounding sphere of all geometry, so the user never sees z-fighting from bad defaults.

### Shading
- **D-06:** Flat shading per face (face normal dot light direction → shading ramp index). Sufficient for M1's cube demo. Per-vertex interpolation deferred to M2.
- **D-07:** ASCII shading ramp is a `&[char]` slice configurable at runtime. Default: ` .:-=+*#%@` (10 levels, space = darkest, @ = brightest). Swapping the ramp at runtime visibly changes the silhouette shading.
- **D-08:** Single directional light (hardcoded direction, e.g., normalized (1,1,1)). Configurable light is Phase 2.x.

### Camera
- **D-09:** Orbit camera with three `Signal<f32>` parameters: `azimuth`, `elevation`, `distance`. Camera computes view matrix from spherical coordinates → look-at. Changing `azimuth` signal causes the cube to visibly orbit.
- **D-10:** Camera is owned by a `Renderer` struct (not a global). Scene graph ownership formalized in Phase 1.4.

### Renderer → Grid integration
- **D-11:** [auto] `Renderer::draw(grid: &mut Grid, ...)` writes directly into the Grid via `grid.buffer_mut()` for character placement and `grid.put_str()` for styled text. The z-buffer is internal to the Renderer (reusable Vec<f32>). No intermediate buffer — the Grid IS the framebuffer.

### Performance
- **D-12:** Zero heap allocations in `Renderer::draw()` after warmup. Z-buffer is `Vec<f32>` owned by Renderer, resized only on Grid dimension change. Criterion bench verifies this.
- **D-13:** [auto] Reusable projection/view matrices cached in Renderer. Only recomputed when camera signals change (dirty flag or compare).

### Quality
- **D-14:** `#![forbid(unsafe_code)]` initially — SIMD optimization is future work. If the researcher finds that the rasterizer needs unsafe for performance, it can be relaxed with justification.
- **D-15:** Snapshot test: render a cube at a fixed camera pose (azimuth=45°, elevation=30°, distance=5.0) on an 80×24 Grid, compare output buffer cell-by-cell against a stored reference.

### Claude's Discretion
- Exact vertex/index representation for Cube (array of [f32;3] vs glam::Vec3)
- Whether to use a Mesh trait or just concrete Cube struct for M1
- Exact reversed-Z projection matrix formula (standard OpenGL-style reversed vs custom)
- Whether Renderer holds a reference to Grid or takes it as parameter each frame
- Internal module structure (projection.rs, rasterizer.rs, camera.rs, etc.)
