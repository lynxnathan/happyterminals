# Phase 01.3: Minimal 3D Renderer (Cube Primitive) - Research

**Researched:** 2026-04-15
**Domain:** Software 3D rasterization to ASCII terminal grid (z-buffer, perspective projection, flat shading, orbit camera)
**Confidence:** HIGH

## Summary

This phase implements `happyterminals-renderer` as a fresh software rasterizer that projects 3D geometry onto the terminal Grid. The core pipeline is: vertices -> model/view/projection transforms -> clip-space -> NDC -> screen-space (cell coordinates) -> half-space triangle rasterization with per-pixel z-buffer -> flat per-face ASCII shading. The output is written directly into `Grid` cells via `buffer_mut()`.

The math foundation is `glam 0.32.1`, which provides `Mat4::perspective_infinite_reverse_rh()` for reversed-Z projection and `Mat4::look_at_rh()` for the orbit camera's view matrix. The cell aspect ratio (default 2.0) must be baked into the projection to prevent vertical stretching -- this is the single most visible correctness issue for terminal rendering.

**Primary recommendation:** Build the renderer as a set of focused modules (projection, rasterizer, shading, camera, cube primitive) with `Renderer` as the orchestrating struct. Use `glam`'s built-in reversed-Z infinite perspective for depth precision. Write a criterion bench from the start that asserts zero heap allocations in the hot path. Use insta snapshot tests for visual regression.

### Pattern 1: Reversed-Z Infinite Perspective with Cell Aspect Correction

**What:** The projection combines glam's reversed-Z infinite perspective with a manual x-scale for cell aspect ratio.

**When to use:** Every frame (cached, recomputed only when viewport or cell_aspect changes).

**Example:**
```rust
// Source: https://docs.rs/glam/0.32.1/glam/f32/struct.Mat4.html
use glam::Mat4;

pub struct Projection {
    pub fov_y: f32,          // radians, default PI/4
    pub cell_aspect: f32,    // default 2.0 (cells are ~2x taller than wide)
    pub viewport_w: u16,
    pub viewport_h: u16,
}

impl Projection {
    pub fn matrix(&self) -> Mat4 {
        // Pixel aspect ratio = viewport_w / viewport_h
        // But cells are cell_aspect:1 (tall:wide), so effective aspect:
        let pixel_aspect = (self.viewport_w as f32) / (self.viewport_h as f32);
        let effective_aspect = pixel_aspect / self.cell_aspect;

        // glam's reversed-Z infinite perspective: near maps to 1.0, infinity to 0.0
        // z_near = 0.1 (we use scene-fit, but need a minimum)
        let proj = Mat4::perspective_infinite_reverse_rh(
            self.fov_y,
            effective_aspect,
            0.1, // z_near -- scene-fit adjusts this
        );
        proj
    }
}
```

**Key insight on cell aspect:** Terminal cells are ~2x taller than wide. The projection's `aspect_ratio` parameter must account for this. If viewport is 80x24 cells, the geometric aspect is `80 / (24 * 2.0) = 1.667`, not `80/24 = 3.33`. Failing to do this makes the cube look like a wide rectangle instead of a cube.

### Pattern 2: Half-Space Triangle Rasterization (Edge Functions)

**What:** For each triangle, compute the bounding box in screen space, then for each cell in the bbox test all three edge functions. A cell is inside the triangle iff all three edge functions are >= 0 (or all <= 0, depending on winding).

**When to use:** Core rasterization loop, called for each of the 12 cube triangles.

**Example:**
```rust
// Source: scratchapixel.com rasterization practical implementation
// + Fabian Giesen "Triangle rasterization in practice"

/// Edge function: positive if (px,py) is on the left side of edge v0->v1
#[inline]
fn edge_function(v0x: f32, v0y: f32, v1x: f32, v1y: f32, px: f32, py: f32) -> f32 {
    (px - v0x) * (v1y - v0y) - (py - v0y) * (v1x - v0x)
}

/// Rasterize one triangle into grid cells.
/// screen_verts: 3 vertices in screen space (cell coordinates + depth).
/// z_buffer: mutable slice, width*height, reversed-Z (init to 0.0, near=1.0).
/// shade_char: the ASCII character for this face's shading.
fn rasterize_triangle(
    sv: &[(f32, f32, f32); 3],  // (screen_x, screen_y, depth)
    z_buffer: &mut [f32],
    grid_width: usize,
    grid_height: usize,
    shade_char: char,
    output: &mut [char],  // or write to Grid buffer
) {
    // Bounding box (clamped to grid)
    let min_x = sv.iter().map(|v| v.0).fold(f32::MAX, f32::min).max(0.0) as usize;
    let max_x = sv.iter().map(|v| v.0).fold(f32::MIN, f32::max).min((grid_width - 1) as f32) as usize;
    let min_y = sv.iter().map(|v| v.1).fold(f32::MAX, f32::min).max(0.0) as usize;
    let max_y = sv.iter().map(|v| v.1).fold(f32::MIN, f32::max).min((grid_height - 1) as f32) as usize;

    for y in min_y..=max_y {
        for x in min_x..=max_x {
            let px = x as f32 + 0.5;
            let py = y as f32 + 0.5;

            let w0 = edge_function(sv[1].0, sv[1].1, sv[2].0, sv[2].1, px, py);
            let w1 = edge_function(sv[2].0, sv[2].1, sv[0].0, sv[0].1, px, py);
            let w2 = edge_function(sv[0].0, sv[0].1, sv[1].0, sv[1].1, px, py);

            if w0 >= 0.0 && w1 >= 0.0 && w2 >= 0.0 {
                // Barycentric interpolation of depth
                let area = w0 + w1 + w2;
                let depth = (w0 * sv[0].2 + w1 * sv[1].2 + w2 * sv[2].2) / area;

                let idx = y * grid_width + x;
                // Reversed-Z: greater depth = closer. Pass if depth > z_buffer[idx].
                if depth > z_buffer[idx] {
                    z_buffer[idx] = depth;
                    output[idx] = shade_char;
                }
            }
        }
    }
}
```

### Pattern 3: Orbit Camera from Spherical Coordinates

**What:** Convert (azimuth, elevation, distance) to a camera eye position, then use `Mat4::look_at_rh` to produce the view matrix.

**Example:**
```rust
use glam::{Vec3, Mat4};

pub struct OrbitCamera {
    pub azimuth: f32,     // radians, Signal<f32>-driven
    pub elevation: f32,   // radians, Signal<f32>-driven
    pub distance: f32,    // Signal<f32>-driven
    pub target: Vec3,     // default Vec3::ZERO
}

impl OrbitCamera {
    pub fn view_matrix(&self) -> Mat4 {
        let eye = self.target + Vec3::new(
            self.distance * self.elevation.cos() * self.azimuth.sin(),
            self.distance * self.elevation.sin(),
            self.distance * self.elevation.cos() * self.azimuth.cos(),
        );
        Mat4::look_at_rh(eye, self.target, Vec3::Y)
    }
}
```

### Pattern 4: Cube Primitive Definition

**What:** A unit cube centered at origin with 8 vertices, 36 indices (12 triangles), and 6 face normals.

**Example:**
```rust
use glam::Vec3;

pub struct Cube;

impl Cube {
    /// 8 vertices of a unit cube centered at origin.
    pub const VERTICES: [Vec3; 8] = [
        Vec3::new(-0.5, -0.5, -0.5), // 0: left-bottom-back
        Vec3::new( 0.5, -0.5, -0.5), // 1: right-bottom-back
        Vec3::new( 0.5,  0.5, -0.5), // 2: right-top-back
        Vec3::new(-0.5,  0.5, -0.5), // 3: left-top-back
        Vec3::new(-0.5, -0.5,  0.5), // 4: left-bottom-front
        Vec3::new( 0.5, -0.5,  0.5), // 5: right-bottom-front
        Vec3::new( 0.5,  0.5,  0.5), // 6: right-top-front
        Vec3::new(-0.5,  0.5,  0.5), // 7: left-top-front
    ];

    /// 12 triangles (2 per face), CCW winding when viewed from outside.
    /// Each triple indexes into VERTICES.
    pub const INDICES: [[usize; 3]; 12] = [
        // Front face (+Z)
        [4, 5, 6], [4, 6, 7],
        // Back face (-Z)
        [1, 0, 3], [1, 3, 2],
        // Right face (+X)
        [5, 1, 2], [5, 2, 6],
        // Left face (-X)
        [0, 4, 7], [0, 7, 3],
        // Top face (+Y)
        [7, 6, 2], [7, 2, 3],
        // Bottom face (-Y)
        [0, 1, 5], [0, 5, 4],
    ];

    /// Face normals (one per pair of triangles).
    pub const FACE_NORMALS: [Vec3; 6] = [
        Vec3::new( 0.0,  0.0,  1.0), // front
        Vec3::new( 0.0,  0.0, -1.0), // back
        Vec3::new( 1.0,  0.0,  0.0), // right
        Vec3::new(-1.0,  0.0,  0.0), // left
        Vec3::new( 0.0,  1.0,  0.0), // top
        Vec3::new( 0.0, -1.0,  0.0), // bottom
    ];
}
```

### Pattern 5: Vertex Transform Pipeline (Model -> View -> Projection -> NDC -> Screen)

**What:** The full transform chain from world-space vertices to screen-space cell coordinates.

**Example:**
```rust
use glam::{Vec3, Vec4, Mat4};

/// Transform a world-space vertex to screen space (cell coordinates + depth).
/// Returns None if the vertex is behind the near plane (clipped).
#[inline]
fn project_vertex(
    pos: Vec3,
    mvp: &Mat4,
    half_w: f32,
    half_h: f32,
) -> Option<(f32, f32, f32)> {
    let clip = *mvp * Vec4::new(pos.x, pos.y, pos.z, 1.0);

    // Clip against near plane (w <= 0 means behind camera)
    if clip.w <= 0.0 {
        return None;
    }

    // Perspective divide -> NDC
    let ndc_x = clip.x / clip.w;
    let ndc_y = clip.y / clip.w;
    let ndc_z = clip.z / clip.w; // reversed-Z: 1.0=near, 0.0=far

    // NDC [-1,1] -> screen [0, width/height]
    let screen_x = (ndc_x + 1.0) * half_w;
    let screen_y = (1.0 - ndc_y) * half_h; // flip Y (screen Y goes down)

    Some((screen_x, screen_y, ndc_z))
}
```

### Anti-Patterns to Avoid
- **Allocating in draw():** Never use `Vec::new()`, `format!()`, `String::from()`, or any heap allocation inside the per-frame draw path. All buffers must be pre-allocated and reused.
- **Reading signals with `.get()` in the renderer:** Use `.untracked()` to avoid registering reactive dependencies from the render loop. The renderer reads scene state imperatively, never subscribes.
- **Treating cells as square pixels:** Always apply the cell_aspect correction in the projection. Without it, everything looks vertically stretched.
- **Using `Mat4::perspective_rh` with hardcoded near/far:** Use scene-fit or infinite perspective to avoid z-fighting.
- **Double-buffering inside the renderer:** The Grid IS the framebuffer. Do not create an intermediate buffer; write cells directly via `buffer_mut()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Projection matrices | Custom 4x4 matrix math | `glam::Mat4::perspective_infinite_reverse_rh()` + `Mat4::look_at_rh()` | Numerically stable, SIMD-optimized, well-tested. Reversed-Z is built-in. |
| Matrix multiplication | Manual loop over 4x4 | `glam::Mat4` operator overloading (`*`) | Correct, fast, handles edge cases (NaN, denormals). |
| Vector normalization | `let len = (x*x+y*y+z*z).sqrt(); ...` | `Vec3::normalize()` | Handles zero-length vectors, uses fast reciprocal sqrt. |
| Trigonometry for camera | Manual sin/cos chains | `f32::sin_cos()` + glam Vec3 construction | Standard library trig is optimized and correct. |

**Key insight:** glam exists precisely to eliminate hand-rolled linear algebra. The only custom math needed is the edge-function rasterizer and the NdotL shading computation, both of which are trivially small.

## Common Pitfalls

### Pitfall 1: Cell Aspect Ratio Distortion
**What goes wrong:** A cube rendered without aspect correction looks like a tall rectangle or wide bar depending on the direction of error.
**Why it happens:** Terminal cells are ~2:1 (height:width). Standard projection math assumes square pixels.
**How to avoid:** Divide the geometric aspect ratio by `cell_aspect` before passing to the projection. Example: for an 80x24 grid with cell_aspect=2.0, effective aspect = `(80/24) / 2.0 = 1.667`. This is decision D-02.
**Warning signs:** The first cube render looks oblong. No `cell_aspect` parameter in the projection API.

### Pitfall 2: Z-Buffer Initialization for Reversed-Z
**What goes wrong:** Z-buffer initialized to `f32::INFINITY` or `1.0` (standard convention) causes all fragments to fail the depth test with reversed-Z.
**Why it happens:** Standard z-buffers clear to 1.0 (far) and pass on `<`. Reversed-Z clears to 0.0 (far) and passes on `>`.
**How to avoid:** Initialize z-buffer to `0.0` each frame. Depth test: `new_depth > z_buffer[idx]`. This is the reversed-Z convention from D-03.
**Warning signs:** Nothing renders (all z-tests fail). Or everything renders in back-to-front painter order (z-test always passes because init is wrong).

### Pitfall 3: Backface Culling Winding Order
**What goes wrong:** Backface culling removes visible faces or fails to remove invisible ones, causing visual artifacts or performance waste.
**Why it happens:** Culling depends on the winding order of vertices matching the face normal direction. If the cube indices have CCW winding but the culling checks for CW (or vice versa), it inverts.
**How to avoid:** Decide on CCW winding viewed from outside (standard OpenGL convention). Cull when the face normal dot camera-to-face direction is >= 0 (face points away from camera). Verify with a single-face test before wiring up the full cube.
**Warning signs:** Cube appears inside-out (back faces visible, front faces culled). Rotating the cube shows missing faces.

### Pitfall 4: Per-Frame Allocation in draw()
**What goes wrong:** Heap allocations per frame cause jitter, especially at higher frame rates. GC-less languages still pay allocator costs.
**Why it happens:** Easy to accidentally use `format!()`, `Vec::new()`, or collect iterators inside the draw loop.
**How to avoid:** Pre-allocate all buffers in the Renderer struct. Use `Vec::clear()` + reuse, not `Vec::new()`. Criterion bench gates this (D-12).
**Warning signs:** Frame time variance > 2x median. `dhat` profile shows allocations inside draw().

### Pitfall 5: Perspective Divide by Zero
**What goes wrong:** Vertices at or behind the camera (w <= 0) cause division by zero or inverted coordinates.
**Why it happens:** The perspective divide (`x/w, y/w, z/w`) is undefined when w=0 and produces garbage when w<0.
**How to avoid:** Clip vertices with `w <= 0.0` before the perspective divide. For M1 with a single cube, simple vertex clipping (discard triangles with any vertex behind camera) is sufficient. Triangle clipping against the near plane is a Phase 2 concern.
**Warning signs:** NaN or inf values in screen coordinates. Triangles that "flip" when they cross the near plane.

### Pitfall 6: Signal Subscription in Renderer
**What goes wrong:** Renderer calls `signal.get()` which subscribes it to the reactive graph, causing infinite re-render loops or unexpected dependency chains.
**Why it happens:** `Signal::get()` registers the calling context as a subscriber. The render loop is not a reactive context, but if called inside an Effect it would be.
**How to avoid:** Always use `signal.untracked()` to read values in the renderer. This is an established project pattern (REACT-07).
**Warning signs:** Render loop triggers itself. Stack overflow from circular dependencies.

### ASCII Shading Ramp
```rust
// Source: decision D-06, D-07, D-08

pub struct ShadingRamp<'a> {
    pub ramp: &'a [char],
    pub light_dir: glam::Vec3,  // normalized, default (1,1,1).normalize()
}

impl<'a> ShadingRamp<'a> {
    pub const DEFAULT_RAMP: &'static [char] = &[' ', '.', ':', '-', '=', '+', '*', '#', '%', '@'];

    pub fn new() -> Self {
        Self {
            ramp: Self::DEFAULT_RAMP,
            light_dir: glam::Vec3::new(1.0, 1.0, 1.0).normalize(),
        }
    }

    /// Map a face normal to an ASCII character.
    /// normal: world-space face normal (normalized).
    #[inline]
    pub fn shade(&self, normal: glam::Vec3) -> char {
        let ndotl = normal.dot(self.light_dir).clamp(0.0, 1.0);
        let idx = (ndotl * (self.ramp.len() - 1) as f32) as usize;
        self.ramp[idx]
    }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Standard-Z (near=0, far=1) | Reversed-Z (near=1, far=0) | ~2018 (GPU engines), now standard in software too | Dramatically better depth precision at distance; eliminates z-fighting for most practical scenes |
| Scanline rasterization | Half-space / edge function | Modern default for software rasterizers | Simpler to implement correctly, no edge-case bugs from scanline walking, naturally parallelizable |
| Square pixel assumption | Cell aspect ratio correction | Specific to terminal rendering | Without this, all terminal 3D renderers look wrong. Most ASCII art renderers hardcode 2:1. |
| `glam` 0.29 | `glam` 0.32.1 | 2025 | Workspace already on 0.32.1. CONTEXT.md says "0.29" but workspace has the newer version -- use 0.32.1. |

**Deprecated/outdated:**
- `nalgebra` for game-like 3D math: overly complex for this use case, `glam` is the ecosystem standard for game/graphics rendering in Rust.
- `cgmath`: abandoned, explicitly forbidden by project (HYG-09).

## Open Questions

1. **Scene-fit vs infinite perspective**
   - What we know: `glam::Mat4::perspective_infinite_reverse_rh()` uses infinite far plane with reversed-Z. D-05 says "scene-fit near/far planes from bounding sphere."
   - What's unclear: Infinite perspective (no far plane) may be sufficient for M1 since reversed-Z gives excellent precision. Scene-fit adds complexity for computing near from the bounding sphere.
   - Recommendation: Use infinite perspective for M1 (simpler, reversed-Z handles precision). The z_near can still be scene-fit from the bounding sphere minimum distance. Finite far-plane scene-fitting can be added in Phase 2 if needed.

2. **Triangle clipping at near plane**
   - What we know: For M1, the cube is at a fixed distance. Simple vertex rejection (discard triangle if any vertex has w<=0) works.
   - What's unclear: Proper triangle clipping (Sutherland-Hodgman against near plane) is needed for geometry that crosses the near plane.
   - Recommendation: Vertex-level clipping for M1 (discard entire triangle if any vertex behind camera). Document that triangle-level clipping is needed for Phase 2.1 (arbitrary meshes).

3. **Snapshot test format**
   - What we know: insta supports string snapshots. The rendered grid is a 2D array of characters.
   - What's unclear: Best format for the snapshot (raw character grid vs. annotated with coordinates).
   - Recommendation: Dump the Grid as a multi-line string (one line per row, each cell's character). Use `insta::assert_snapshot!`. This gives a human-readable diff when the rendering changes.
