## Decisions Made

### 1. Path A (low-level `run`) over DSL extension

**Decision:** Viewer uses `run(render_fn, spec)` directly instead of `run_scene` + a new `NodeKind::Mesh` variant.

**Rationale:** CONTEXT.md (Claude's Discretion) and RESEARCH §"DSL Integration" both recommend deferring the scene-graph mesh node until DSL-04 designs JSON serialization. The viewer's 3-model carousel state is viewer-specific, not shared scene state — putting it in a scene graph would be a category mistake. Writing the example on the low-level API also proves the Phase 1.1 `run()` contract is actually usable for non-trivial demos, which is useful validation independent of the DSL decision.

### 2. Static `FrameSpec.title` + in-grid row-0 overlay

**Decision:** Window title is `"happyterminals - Model Viewer"` (static, set once); current model name renders as an in-grid string at row 0 via `grid.put_str(0, 0, label, Style::default())`.

**Rationale:** Planning constraint #7 explicitly freezes the backend surface in 2.1 — dynamic title mutation would require re-emitting `SetTitle` inside `run.rs`. The in-grid overlay is visible, doesn't clip the model (row 0 overlays a region the camera rarely paints through), and costs zero new backend surface area. If a future phase wants dynamic titles, that's a library feature, not an example concern.

### 3. Per-model auto-fit camera at `radius * 2.5`

**Decision:** `camera.distance = mesh.bounding_sphere().1 * 2.5` per frame; `elevation = 0.4` (matches spinning-cube default).

**Rationale:** RESEARCH §"Open Questions #4" recommends the 2.5× multiplier. Bunny has ~0.2-unit radius, teapot ~3.0-unit — a 15× scale span. Any fixed `distance` would either make bunny a speck or clip teapot. The `bounding_sphere()` API was explicitly shipped in Wave 1 to unblock this use case, and Wave 2's `scene_fit_near` already uses it for the near plane, so mirroring the pattern here keeps the two distance computations consistent.

### 4. `crossterm` added to `happyterminals` dev-deps

**Decision:** Add `crossterm = { workspace = true }` under `[dev-dependencies]` in `crates/happyterminals/Cargo.toml`.

**Rationale:** `InputEvent::Key { code: KeyCode, modifiers: KeyModifiers }` re-exposes `crossterm::event::KeyCode` by design — the framework chose not to wrap every keycode enum into a parallel type. Pattern-matching on `KeyCode::Left` / `KeyCode::Right` in the example requires the import to be nameable. Adding to dev-deps (not prod deps) keeps the public crate surface unchanged; only examples/tests can see the import.
