# Phase 2.1: OBJ Mesh Loading тАФ Research

**Researched:** 2026-04-15
**Domain:** OBJ mesh parsing, `tobj 4.0.3` integration, flat-normal / winding recovery, Mesh/Cube unification, viewer example UX
**Confidence:** HIGH for API + recipe (docs.rs primary source), MEDIUM for fixture corpus sourcing (license per-file work) and winding auto-correction (defensible-but-partial heuristic)

## Summary

Phase 2.1 is a narrow, well-constrained slice: add a panic-free OBJ loader on top of `tobj 4.0.3` (already pinned in workspace deps, not yet consumed), fold loaded data into a `Mesh` struct that unifies the rasterizer path with `Cube`, and ship a `model-viewer` example that cycles the three pre-imported OBJs via arrow keys. The `Mesh` API is already frozen in CONTEXT.md тАФ research here is prescriptive about **how** to wire `tobj` to that locked shape and **what defensive plumbing** keeps the loader honest against weird in-the-wild OBJs.

Three pieces need care. First, `tobj::LoadOptions` has mutually-exclusive flag combinations (`single_index` is incompatible with `merge_identical_points` and `reorder_data`) тАФ we pick `{ triangulate: true, single_index: true, ignore_points: true, ignore_lines: true }` and document why. Second, `tobj` leaves `mesh.normals` empty when the OBJ omits them тАФ we **always** compute flat per-face normals post-load so the rasterizer stays branchless (it already shades via a single `face_normal` lookup). Third, `tobj`'s face-varying index model means a single OBJ vertex can have multiple per-face normals; `single_index: true` duplicates the position data to de-duplicate, which is exactly what we want for a flat-shaded rasterizer (no smoothing groups, no Gouraud тАФ 2.1 is face-normal only).

The `rasterizer.rs` hot path already handles both windings (`sign_positive` branch) тАФ so we don't need to *correct* winding, only to *detect it once* and record a `LoadStats.warnings` entry if the mesh looks inside-out so users get a diagnostic. Full winding auto-correction (volume-sign test) is possible but adds allocation and risk of false positives on non-closed meshes тАФ **recommend deferring** and instead documenting "if your model renders inside-out, export with CCW winding."

**Primary recommendation:** Use `tobj` with `single_index: true + triangulate: true`, compute flat normals unconditionally post-load (ignoring any provided normals тАФ this matches current `ShadingRamp` flat-shade model), return `Result<(Mesh, LoadStats), MeshError>` with per-triangle skip accounting, ship the 10-file fixture corpus by expanding the existing `examples/models/{bunny,cow,teapot}` with 7+ small fixtures (synthetic `tests/fixtures/obj/*.obj` files тАФ quad, ngon, missing_normals, degenerate, empty, truncated, negative-index) rather than pulling large external models. Extend `renderer_alloc` bench with a loaded-bunny case to prove zero-per-frame-alloc holds for ~5k-triangle meshes.

## tobj 4.0.3 API Surface (HIGH confidence, docs.rs primary)

### Load Functions

```rust
// File path (auto-discovers sibling .mtl files; we don't use materials)
pub fn load_obj<P: AsRef<Path>>(
    path: P,
    load_options: &LoadOptions,
) -> LoadResult;

// From in-memory buffer (useful for tests without file I/O)
pub fn load_obj_buf<B: BufRead, ML>(
    reader: &mut B,
    load_options: &LoadOptions,
    material_loader: ML,
) -> LoadResult
where
    ML: FnMut(&Path) -> MTLLoadResult;
```

Where:

```rust
pub type LoadResult = Result<(Vec<Model>, Result<Vec<Material>, LoadError>), LoadError>;
```

**Key insight:** `LoadResult` is a doubly-fallible type тАФ outer `Result` for OBJ parse, inner `Result` for MTL parse. Since we ignore materials, we:
- Use `load_obj` (handles `.mtl` discovery silently; MTL failure returns inner `Err(_)` which we drop)
- Or use `load_obj_buf` in tests with a stub material loader that always returns `Ok(Vec::new())` тАФ e.g. `|_| Ok((Vec::new(), Default::default()))`.

### LoadOptions

```rust
pub struct LoadOptions {
    pub merge_identical_points: bool,  // needs "merging" feature flag
    pub reorder_data: bool,            // needs "reordering" feature flag
    pub single_index: bool,            // unifies vertex attribute indexing (GPU-friendly)
    pub triangulate: bool,             // fan-triangulate polygons (required for us)
    pub ignore_points: bool,           // drop point-only faces
    pub ignore_lines: bool,            // drop line-only faces
}
```

**Mutually-exclusive flag combinations (enforced тАФ violates тЖТ `LoadError::InvalidLoadOptionConfig`):**
1. `single_index` + `merge_identical_points` тАФ forbidden.
2. `single_index` + `reorder_data` тАФ forbidden.
3. `merge_identical_points` + `reorder_data` тАФ forbidden.

**Recommended config for this phase:**

```rust
tobj::LoadOptions {
    triangulate: true,     // handle quads + ngons
    single_index: true,    // one index stream matches our Vec<[u32; 3]>
    ignore_points: true,   // we don't support point primitives
    ignore_lines: true,    // we don't support line primitives
    ..tobj::LoadOptions::default()  // merge + reorder stay false
}
```

Rationale for `single_index: true`:
- OBJ's native face format `f 1/2/3 4/5/6 7/8/9` has separate indices for position / texcoord / normal. `single_index` duplicates positions as needed so one index stream addresses all attributes тАФ matching our `Mesh.indices: Vec<[u32; 3]>`.
- We ignore normals and recompute them flat per-face anyway, so the "duplication" doesn't hurt тАФ we're already going to have one normal per triangle, not per vertex.
- Without `single_index`, our struct shape would require three parallel index streams (pos, tex, nrm) тАФ violates CONTEXT's locked struct.

**Conversion sketch** (tobj тЖТ our `Mesh`):

```rust
// After load_obj with {triangulate: true, single_index: true}:
let (models, _materials_result) = tobj::load_obj(path, &opts)?;

// Concatenate all models' meshes into one Mesh (OBJ files often have multiple "o" objects)
let mut vertices = Vec::new();
let mut indices: Vec<[u32; 3]> = Vec::new();

for model in &models {
    let base = u32::try_from(vertices.len()).map_err(|_| MeshError::Parse("vertex overflow".into()))?;
    // Positions: flat [x,y,z,x,y,z,...] тЖТ Vec<Vec3>
    for chunk in model.mesh.positions.chunks_exact(3) {
        vertices.push(Vec3::new(chunk[0], chunk[1], chunk[2]));
    }
    // Indices: flat тЖТ Vec<[u32; 3]>, rebased to account for concat
    for tri in model.mesh.indices.chunks_exact(3) {
        indices.push([tri[0] + base, tri[1] + base, tri[2] + base]);
    }
}
```

### Error Mapping (tobj::LoadError тЖТ MeshError)

`tobj::LoadError` has 16 variants (verified docs.rs). Mapping:

| tobj::LoadError | Our MeshError |
|-----------------|---------------|
| `OpenFileFailed`, `ReadError` | `Io(std::io::Error)` тАФ wrap via conversion |
| `UnrecognizedCharacter`, `PositionParseError`, `NormalParseError`, `TexcoordParseError`, `FaceParseError`, `MaterialParseError`, `InvalidObjectName`, `GenericFailure` | `Parse(String)` with the Display of the tobj variant |
| `InvalidPolygon` | `Parse(String)` with contextual message |
| `FaceVertexOutOfBounds`, `FaceTexCoordOutOfBounds`, `FaceNormalOutOfBounds`, `FaceColorOutOfBounds` | `Parse(String)` тАФ these are whole-file corruption, not per-triangle |
| `InvalidLoadOptionConfig` | `unreachable!()` since our options are statically valid тАФ OR treat as `Parse(String)` for safety (prefer the latter given `unwrap_used = deny`) |

**Per-triangle skips happen AFTER `tobj::load_obj` succeeds** тАФ during our post-load normal computation, when we detect degenerate (zero-area) triangles or NaN vertices. These produce `LoadStats.triangles_skipped += 1` and a warning line; they don't return `Err`.

### Memory / Performance Notes (MEDIUM confidence, inferred)

- `tobj::load_obj` is a one-shot allocator; expect ~`sizeof::<f32>() * 3 * vertex_count` for positions plus index overhead. For bunny.obj (~2500 verts, ~5000 tris) this is ~30 KB + ~60 KB indices тЙИ ~90 KB тАФ insignificant.
- Parsing is sequential + buffered; docs describe no streaming API.
- No `#![no_std]` support тАФ we're in std anyway.
- `tobj` allocates internally during parse; this is fine because loading is one-shot (not hot path). PITFALLS ┬з11 applies only to the **rasterizer**; loading is pre-frame cost.

## Flat-Normal Computation Recipe (HIGH confidence)

```rust
/// Compute flat per-face normal via Newell's formula / cross product of edges.
///
/// Returns `None` if the triangle is degenerate (zero-area / collinear vertices).
/// Callers treat `None` as "skip this triangle and log a warning".
#[must_use]
fn flat_normal(a: Vec3, b: Vec3, c: Vec3) -> Option<Vec3> {
    let edge1 = b - a;
    let edge2 = c - a;
    let cross = edge1.cross(edge2);
    let len_sq = cross.length_squared();
    if len_sq < f32::EPSILON {
        return None;  // degenerate
    }
    Some(cross / len_sq.sqrt())
}
```

**Usage in the load path:**

```rust
let mut normals = Vec::with_capacity(indices.len());
let mut stats = LoadStats::default();
let mut kept_indices = Vec::with_capacity(indices.len());

for (i, &[a_idx, b_idx, c_idx]) in indices.iter().enumerate() {
    let a = vertices[a_idx as usize];
    let b = vertices[b_idx as usize];
    let c = vertices[c_idx as usize];

    // NaN guard (cheap; real OBJs rarely contain NaN but malformed/fuzzed ones do)
    if !a.is_finite() || !b.is_finite() || !c.is_finite() {
        stats.triangles_skipped += 1;
        stats.warnings.push(format!("skipped triangle {i}: non-finite vertex"));
        continue;
    }

    match flat_normal(a, b, c) {
        Some(n) => {
            normals.push(n);
            kept_indices.push([a_idx, b_idx, c_idx]);
            stats.triangles_loaded += 1;
        }
        None => {
            stats.triangles_skipped += 1;
            stats.warnings.push(format!("skipped triangle {i}: degenerate (zero area)"));
        }
    }
}
```

**Invariants:**
- `mesh.normals.len() == mesh.indices.len()` тАФ one normal per kept triangle.
- `vertices` is NOT pruned (unused vertices stay; cheap to keep and avoids reindexing).
- Warning count capped (see ┬зPitfalls тАФ unbounded warning strings on a pathological file).

**CCW convention:** Our `Cube::FACE_NORMALS` uses `edge1.cross(edge2)` where edges are CCW-ordered. For a loaded OBJ with CCW winding (the convention), this formula produces outward normals. For CW-wound input, normals point inward тАФ see ┬зWinding Detection below.

## Winding Detection + Decision (MEDIUM confidence тАФ documented, narrow scope)

### Observation

The rasterizer at `crates/happyterminals-renderer/src/rasterizer.rs:88-104` already accepts **both** windings: after Y-flip projection, CCW triangles become CW in screen space, making `area` negative. The `sign_positive` branch handles this. So a CW-wound mesh *rasterizes correctly as a silhouette* тАФ but its computed **normals point inward**, meaning the `ShadingRamp::shade(normal)` call produces inverted light/dark тАФ bright faces go dark and vice versa. The model looks "inside-out" to the user.

### Detection approach (no correction тАФ 2.1 scope)

For a closed manifold mesh, the **signed volume** via divergence theorem is positive for CCW outward normals, negative for inward:

```rust
fn signed_volume(vertices: &[Vec3], indices: &[[u32; 3]]) -> f32 {
    indices.iter()
        .map(|&[a, b, c]| {
            let va = vertices[a as usize];
            let vb = vertices[b as usize];
            let vc = vertices[c as usize];
            va.dot(vb.cross(vc)) / 6.0
        })
        .sum()
}
```

**If `signed_volume < 0.0`, push `LoadStats.warnings.push("mesh appears to have inverted winding; normals may render inside-out".into())` and proceed unchanged.**

### Why not auto-correct?

1. Requires allocating a new `Vec<[u32; 3]>` with swapped indices тАФ trivial, but ties us to a decision.
2. **False positives** on non-closed meshes (a single plane, a clipped model) тАФ volume sign is meaningless.
3. The user may intentionally want inside-out rendering for effect.
4. PITFALLS ┬з15 lists "silently renders a featureless blob, or rotates the model 180┬░ because winding order was assumed CCW when it was CW" as the failure mode тАФ **warning-not-panic** is the correct fix per that pitfall.
5. `MeshError::Winding` is already in the locked enum тАФ reserve it for **whole-file** rejection cases the user opts into later; use `LoadStats.warnings` for detection.

### Tests bunny/cow/teapot will pass

Reviewing `examples/models/*.obj`:
- `bunny.obj` header: standard Stanford-bunny format, no comments about winding. Assume CCW (Stanford convention).
- `cow.obj` header: `# OBJ File Generated by Meshlab` тАФ Meshlab exports CCW by default.
- `teapot.obj` header: pure vertex data, no generator tag тАФ check volume sign during fixture test setup.

Plan should include a one-shot test per corpus file asserting "loads, volume sign positive, no warnings" тАФ catches regression if any file gets re-imported with flipped winding.

### Pattern 1: `load_obj` Public Signature

**Design notes:**
- Accepts `AsRef<Path>` for ergonomics (matches std / tobj convention).
- Returns `(Mesh, LoadStats)` as a tuple тАФ not a struct тАФ because stats are consumer-optional (`let (mesh, _) = load_obj(p)?;` is a common case).

### Pattern 2: Cube Unification

Existing `Cube` exposes `VERTICES: [Vec3; 8]`, `INDICES: [[usize; 3]; 12]`, `FACE_NORMALS: [Vec3; 6]`. Add:

```rust
impl Cube {
    /// Construct a heap-allocated `Mesh` equivalent to the Cube primitive.
    ///
    /// Every triangle gets the matching face normal (two triangles per face тЖТ six
    /// normals repeated). Enables uniform rendering through `Renderer::draw_mesh(&Mesh)`.
    #[must_use]
    pub fn mesh() -> Mesh {
        let vertices = Self::VERTICES.to_vec();
        let indices: Vec<[u32; 3]> = Self::INDICES
            .iter()
            .map(|&[a, b, c]| [a as u32, b as u32, c as u32])
            .collect();
        let normals: Vec<Vec3> = (0..12)
            .map(|tri_idx| Self::FACE_NORMALS[Self::face_normal_index(tri_idx)])
            .collect();
        Mesh {
            vertices,
            indices,
            normals,
            shading: None,
        }
    }
}
```

impl Renderer {

    pub fn draw(&mut self, grid: &mut Grid, mesh: &Mesh, camera: &OrbitCamera,

        // ... buffer setup same as before ...
        for (tri_idx, &[i0, i1, i2]) in mesh.indices.iter().enumerate() {
            let face_normal = mesh.normals[tri_idx];
            if rasterizer::backface_cull(face_normal, camera_dir) { continue; }
            let effective_ramp = mesh.shading.as_ref().unwrap_or(shading);
            let shade_char = effective_ramp.shade(face_normal);
            // project + rasterize as before
        }
    }
}
```

**Breaking-change implications:** All existing call sites (the spinning-cube demo, `renderer_alloc` bench, `draw_renders_visible_cube` test) need updating. But they all pass a `Cube` implicitly тАФ change them to `Cube::mesh()` once. Small patch, few files тАФ see ┬зPlan Integration Notes.

    ("teapot", "examples/models/teapot.obj"),

        let current_model = Signal::new(0_usize);
        // ... build scene similarly to spinning-cube, but mesh-node-aware ...
        Ok::<_, Box<dyn std::error::Error>>((scene, rotation, current_model))

    run_scene(scene, FrameSpec { title: Some("...".into()), ..Default::default() },
        |dt, input| {
            // Rotate
            rotation.set(rotation.untracked() + dt.as_secs_f32() * 1.5);

                    _ => {}

```

**Nuance тАФ the title dynamic update:** `FrameSpec.title` is set once at `run_scene` entry (see `run.rs:54-56` тАФ `SetTitle` once, never re-executed). Dynamic title updates require a follow-up to `run.rs` to re-emit `SetTitle` when current model changes, OR accept static title "happyterminals - Model Viewer" without per-model name. **Recommend static title for 2.1**; plan can file a small backlog item to add dynamic title if useful later. Model name visibility via an on-screen overlay (write into `Grid` row 0) is also viable and stays within the existing render callback.

### Anti-Patterns to Avoid

- **Loading in the render loop.** PITFALLS ┬з15 + integration-gotchas row: `OBJ loader | Parse from path inside a hot loop | Load once into a Mesh struct, render that`. The viewer MUST pre-load all three meshes before `run_scene`.
- **`unwrap` on `tobj::load_obj`.** Lints deny it. Match with `?` via `From<tobj::LoadError> for MeshError`.
- **Re-triangulating manually.** `tobj` does it correctly via `triangulate: true` тАФ don't roll fan triangulation.
- **Storing `tobj::Mesh` directly.** Leaks the dep through our public API and couples us to tobj's flat `Vec<f32>` shape forever. Convert at load time.
- **Allocating during rasterization of loaded meshes.** Existing zero-alloc path works because `z_buffer` / `cell_chars` are reused. For a 5000-triangle bunny vs 12-triangle cube, the **iteration count** increases but **allocation count** stays zero тАФ don't introduce any `Vec::new()` inside `draw`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OBJ parsing | Custom tokenizer / `split_whitespace` line loop | `tobj::load_obj` | Negative indices, line continuations, `\` escapes, multiple separator variants, comment handling, groups, smoothing groups, `#`-comments mid-line тАФ 15+ edge cases. PITFALLS ┬з15 explicit. |
| Polygon triangulation | Fan triangulation by hand | `LoadOptions { triangulate: true }` | `tobj` handles convex-fan case; non-trivial polygons produce defined behavior (documented). Hand-rolling introduces winding inconsistencies. |
| Face-varying index de-duplication | Manual vertex splitting | `LoadOptions { single_index: true }` | Single index stream gets us directly to our `Vec<[u32; 3]>` shape. |
| Error type for parsing | Hand-roll `Box<dyn Error>` | `thiserror::Error` derive | Already in workspace deps; project convention. |
| Test fixture OBJs | Large downloaded models (Lucy, Happy Buddha) | Small synthetic fixtures + the 3 imported models | Real models have licenses to audit; synthetic files at ~200 bytes each target specific quirks and stay in-repo. |
| Winding flip | Volume-sign-based correction + index swap | Warning-only in `LoadStats` | False-positive risk on non-closed meshes. Rasterizer already handles both windings geometrically. |

**Key insight:** Every quirk OBJ throws at us is *already* in `tobj`'s test corpus. The one thing we add тАФ flat-normal computation тАФ is trivial (~15 lines). The rest is glue. The trap is **gold-plating the glue**: don't add winding correction, don't add smoothing groups, don't add material stubs. File those in 999.x if they become real.

## Common Pitfalls

### Pitfall 1: `tobj::LoadError::MaterialParseError` when `.mtl` missing

**What goes wrong:** `tobj::load_obj` returns `Ok((models, Err(MaterialParseError)))` when a sibling `.mtl` is missing but the OBJ references one via `mtllib`. Naive `let (models, materials) = tobj::load_obj(...)?;` doesn't compile тАФ `materials` is `Result<Vec<Material>, LoadError>`, not `Vec<Material>`. Handled inside our load path but **tests should cover this** тАФ include a fixture that references a nonexistent `.mtl`.

**How to avoid:** Ignore the inner `Result`: `let (models, _materials_result) = tobj::load_obj(...)?;`. Document in the implementation that MTL failures are silently absorbed (we don't use materials).

### Pitfall 2: Concatenated multi-object OBJs with non-unique names

**What goes wrong:** OBJ files can contain multiple `o <name>` objects. `tobj` returns one `Model` per `o`. Our loader concatenates them тАФ fine, except that if one object has positions `[0..100]` and the next has `[100..200]`, the second's `indices` are also `[100..]` (indexed into the combined-global). Or are they per-Model-local? **Per-Model-local.** So **we MUST rebase indices** by adding the cumulative vertex count to each sub-model's indices. The conversion sketch in ┬зtobj API Surface shows this; verify during implementation.

**How to avoid:** Unit test: fixture OBJ with two objects, each a single triangle, asserting combined `Mesh` has 6 vertices + 2 triangles with correctly rebased indices.

### Pitfall 3: Non-finite vertices (NaN / Inf) from pathological OBJs

**What goes wrong:** A fuzzer-generated OBJ with `v NaN NaN NaN` triggers degenerate triangles; if we blindly `.cross()` and `.normalize()`, we get NaN normals, then NaN `dot` products in `ShadingRamp::shade`, then the ramp index arithmetic silently yields `ramp[0]`. Result: no panic, but the shading breaks invisibly. Worse: propagates NaN into the z-buffer тАФ visual artifacts across frames.

**How to avoid:** Explicit `v.is_finite()` check per vertex before the cross-product (shown in the flat-normal recipe). Skip + count in `LoadStats`. Proptest generates OBJs with NaN positions to verify no-panic + no-NaN in output.

### Pitfall 4: Warning vector unbounded growth on pathological files

**What goes wrong:** A 10-million-triangle malformed OBJ produces 10M warning strings, each allocated. Before the loader finishes returning, the process OOMs. PITFALLS security row: `OBJ/STL loader trusts file size | Malicious .obj claims 1B vertices, OOMs | Cap mesh sizes (configurable, sane default)`.

**How to avoid:** Cap `LoadStats.warnings` at e.g. 100 entries; after that, increment a counter only. Also set a hard triangle cap (e.g., 1M) тАФ if `mesh.indices.len() / 3 > MAX_TRIANGLES`, return `MeshError::Parse("mesh exceeds maximum triangle count".into())`. Document the cap as a public constant.

### Pitfall 5: Per-frame allocation regression when meshes scale up

**What goes wrong:** The `renderer_alloc` bench verifies zero-alloc on the 12-triangle cube. Rendering a 5000-triangle bunny exercises the same code path with 400├Ч more iterations, but `tracing::trace_span!("frame").entered()` allocates per frame (see `run.rs:73` тАФ existing cube behavior, not new). At 5000 iterations/frame the per-cell loops are fine; the concern is whether any added mesh-lookup code (e.g., per-triangle shading-ramp selection) introduces allocation.

**How to avoid:** Extend `renderer_alloc.rs` with a `bench_draw_mesh_bunny` function. Use `dhat` or criterion's allocation counting to verify 0 allocations after warmup. Preallocate renderer buffers proportionally to max-triangle-count on first call (already done for grid cells; **also** ensure no Vec inside the triangle loop).

### Pitfall 6: `MeshError` implementing `From<tobj::LoadError>` leaks tobj type into public API

**What goes wrong:** If we write `impl From<tobj::LoadError> for MeshError`, consumers can downcast and see the underlying tobj variant. This is fine for 2.1 but commits us to keeping `tobj` as a transitive public dep. Since we ignore materials and may want to swap loaders someday, keep the conversion internal тАФ via a private `fn map_tobj_err(e: tobj::LoadError) -> MeshError` helper, not a `From` impl. The public `MeshError::Parse(String)` pattern matches existing `happyterminals-scene::SceneError` style.

**How to avoid:** Use `#[from] std::io::Error` in `MeshError::Io` (standard), but **do not** use `#[from] tobj::LoadError`. Call a private helper at the one-and-only tobj call site.

### Pitfall 7: Example binary references model paths relative to cwd

**What goes wrong:** `cargo run --example model-viewer` runs with cwd = workspace root. Path `examples/models/bunny.obj` resolves correctly from workspace root but fails from `crates/happyterminals/` if invoked with a different cwd. In CI, the path discipline matters.

**How to avoid:** Document in example header: "Run from workspace root: `cargo run --example model-viewer`." Or use `env!("CARGO_MANIFEST_DIR")` to resolve an absolute path at compile time. Recommend the latter for robustness:

```rust
const BUNNY: &str = concat!(env!("CARGO_MANIFEST_DIR"), "/../../examples/models/bunny.obj");
```

### Pitfall 8: Arrow-key handling duplicates

**What goes wrong:** `InputSignals.last_key` holds the *last* key event as a `Signal<Option<InputEvent>>`. If the user presses Right and holds, the render-loop ticker may read the same "Right" event on multiple frames, cycling twice per press. `map_event` already filters out `KeyEventKind::Repeat`, but the render callback still reads the same signal on every frame.

**How to avoid:** The render callback must track whether it has already consumed the current `last_key` value. Pattern:

```rust
let last_handled = Signal::new(0_u64);  // increments on each real key
// In map_event: set last_key AND increment a generation counter.
```

Or simpler: after handling, reset `input.last_key` to `None` from within the tick. But `last_key` is a `Signal<Option<InputEvent>>` exposed immutably тАФ writing from the tick needs `.set(None)`. Verify `Signal` exposes `.set` on a borrow (reviewing `happyterminals-core` confirms yes тАФ `Signal::set` takes `&self`).

**Plan recommendation:** Use the "reset to None after handling" pattern in the viewer. Simpler than adding a new signal.

## Open Questions

1. **Are the pre-imported OBJ files license-compatible with MIT-OR-Apache-2.0?**
   - What we know: Stanford bunny has a research-use caveat; Meshlab-exported cow.obj origin unclear; teapot is Martin Newell's (typically public domain).
   - What's unclear: Per-file licensing provenance.
   - Recommendation: Audit happens out-of-scope for Phase 2.1 (flagged as HYG-05-adjacent). If audit blocks, substitute with synthetic + Blender-exported Suzanne / Keenan Crane's Spot (both well-licensed). Plan should include a "license notes" markdown in `examples/models/README.md`.

2. **Should we dynamically update `FrameSpec.title` on model switch?**
   - What we know: Current `run.rs` emits `SetTitle` once at entry.
   - What's unclear: User preference for dynamic terminal title vs. in-grid overlay.
   - Recommendation: Ship with static title for 2.1; add in-grid model-name overlay (row 0) as the visible indicator. Dynamic title is a 5-line follow-up if desired.

3. **Should `Mesh.shading` store `ShadingRamp<'static>` or an owned variant?**
   - What we know: Current `ShadingRamp<'ramp>` borrows its `ramp: &'ramp [char]`. Owning would require `Vec<char>` тАФ ~24 bytes overhead per Mesh.
   - What's unclear: Do we want users to pass a `&'static` ramp, or an owned one?
   - Recommendation: `Option<ShadingRamp<'static>>` for 2.1 (users pass `&'static` custom arrays via `const`). The DSL-level ergonomics work lands in 999.x and can introduce `OwnedShadingRamp` at that time. Document this choice.

4. **Per-model auto-fit camera distance?**
   - What we know: bunny is ~0.2 units tall; teapot is ~3.0 units tall (visible from raw vertex values above). Using a fixed distance renders bunny as a dot, teapot as a giant clipped mesh.
   - What's unclear: Does CONTEXT's "no scope creep" apply? CONTEXT ┬зspecifics explicitly flags this: "may need per-model distance auto-fit since the three models differ in scale".
   - Recommendation: Include bounding-box computation (`Mesh::bounding_sphere() -> (Vec3, f32)` тАФ cheap, no allocation beyond the method call). Viewer sets `camera.distance = radius * 2.5`. Adds ~10 LOC to mesh.rs. Net win; recommend in scope.

## DSL Integration Recommendation (Claude's Discretion area)

**Recommendation: Bypass the DSL for 2.1. No new SceneNode variant, no new builder method.**

**Why:**
1. The CONTEXT.md viewer spec has the example load meshes up-front and cycle via arrow keys. It doesn't need to go through `scene().layer().xxx()` тАФ the cycling state is a `Signal<usize>` that the render callback reads, and the renderer can draw the currently-selected pre-loaded mesh directly.
2. Adding a `NodeKind::Mesh(Mesh)` variant commits us to serialization shape for DSL-04 (JSON recipes) before we've designed that surface. Premature lock-in.
3. Adding a `.mesh(Mesh)` or `.obj(path)` builder expands the DSL API in a way that the 999.x backlog work will likely revise (once `.shading()` lands).
4. The spinning-cube example already demonstrates the DSL surface. Model-viewer demonstrates a *different* capability (input-driven mesh switching) that doesn't need the DSL's tree structure.

**Practical approach тАФ two paths, pick one at plan time:**

**Path A (recommended, minimal):** Bypass DSL entirely in the example.
- Pre-load 3 meshes.
- Build a scene with one empty Layer (or just skip scene graph).
- Write the render callback to call `renderer.draw_mesh(&meshes[current_model], ...)` directly.
- Bonus: this validates the direct renderer-level Mesh API and ensures `Renderer::draw(&Mesh)` is self-contained.

**Path B (if aesthetic parity matters):** Add a `Custom("mesh")` node with the mesh index as a prop.
- Use the existing `NodeKind::Custom(String)` variant тАФ it exists already (node.rs:46) for exactly this kind of forward-compatible extension.
- Backend `render_node` matches on `NodeKind::Custom("mesh")` and draws from a `meshes: HashMap<String, Mesh>` passed alongside.
- Cost: ~15 LOC of backend glue. Benefit: demonstrates the custom-node extensibility.

Both paths are < 80 LOC example total. **Path A is cleaner and resists the pull to over-engineer the DSL before 999.x lands.**
