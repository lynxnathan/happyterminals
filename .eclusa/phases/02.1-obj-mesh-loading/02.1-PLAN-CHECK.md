## Nice-to-Have Improvements (advisory, non-blocking)

1. **Wave parallelism hint.** 02.1-03 Task 1 (prelude re-exports + `examples/models/README.md`) only depends on Plan 01 artifacts, not Plan 02. An executor running plans in parallel could start 03 Task 1 alongside 02. The current sequential `depends_on: ["02.1-01", "02.1-02"]` on Plan 03 is correct for 03 Task 2 but leaves latency on the table for Task 1. Not worth splitting into a separate plan; just an observation.
2. **`insta` snapshot of rendered cube.** Must-have #4 in RESEARCH §Validation refers to "Cube::mesh() produces a Mesh whose rendered output is pixel-identical to the current direct-Cube rasterization (insta snapshot comparison)." VALIDATION.md (the per-task map) relaxes this to "12 tris, 8 unique vertices, flat normals match face count." The relaxation is defensible (an insta snapshot on ASCII output is brittle across glam versions and listed as a nice-to-have in RESEARCH §Nice-to-haves) but the planner could add a one-line rationale in 01 Task 2 explaining why the insta comparison was substituted. Non-blocking.
3. **Backend pre-allocated Cube::mesh() lifetime.** 02.1-02 Task 1 correctly hoists `let cube_mesh = Cube::mesh();` above the `run_scene` loop. If a second scene is ever run after the first exits, the mesh is rebuilt — acceptable for current single-scene lifetimes but worth a comment in the code once it lands.
4. **Bunny triangle count `>= 4900`.** Both Plan 01 (unit test) and Plan 03 consumers rely on this threshold. If the bunny.obj file is ever replaced (license audit outcome), the magic number becomes stale. Future mitigation: read once, assert `>= half the observed count`. Advisory.
