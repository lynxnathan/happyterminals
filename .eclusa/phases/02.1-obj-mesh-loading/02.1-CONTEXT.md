<decisions>
## Implementation Decisions

### Mesh API Shape
- Storage: heap-allocated `Vec<Vec3>` + `Vec<[u32; 3]>` — runtime-loaded so can't use const arrays like `Cube::VERTICES`; mirrors glam + CCW winding convention established in `crates/happyterminals-renderer/src/cube.rs`.
- Concrete `struct Mesh` — no multi-backend need; adding a trait would be speculative abstraction.
- Module location: `happyterminals-renderer::mesh::{Mesh, load_obj, MeshError}` — same crate that consumes it; `tobj 4.0` dependency added to renderer `Cargo.toml` only.
- Unification: `Cube::mesh() -> Mesh` helper so the rasterizer takes `&Mesh` uniformly. No Cube-vs-loaded-mesh branching in `rasterizer.rs`.

### Error Handling Strategy
- `MeshError` is a `thiserror`-derived enum: `Io(std::io::Error)`, `Parse(String)`, `Degenerate { file: String, line: Option<u32> }`, `Winding`. Style matches existing `-core`/`-pipeline` errors; `thiserror 2.0` already in `[workspace.dependencies]`.
- Malformed-data policy: per-triangle skip + warning counter. Return type is `Result<(Mesh, LoadStats), MeshError>` where `LoadStats { triangles_loaded, triangles_skipped, warnings: Vec<String> }`. Whole-file issues (IO failure, total parse failure) still return `Err`. Per-triangle issues (degenerate, NaN) skip and count.
- Missing normals: compute flat per-face normals on load via cross product of edges; store on `Mesh`. No runtime branching in the rasterizer.
- Quad triangulation: delegate to `tobj` via `triangulate: true` load config. Fan-triangulation is standard and well-tested upstream.
- 10+ real-world OBJ corpus in `crates/happyterminals-renderer/tests/fixtures/obj/` asserts zero panics (proptest + explicit file list).

### Viewer Example Scope
- Location: `crates/happyterminals/examples/model-viewer/main.rs` — parallels existing `crates/happyterminals/examples/spinning-cube/main.rs`. Invoked via `cargo run --example model-viewer`.
- Cycles bunny/cow/teapot from `examples/models/` via Left/Right arrow keys; current model name shown in the `FrameSpec` title bar.
- Orbit camera (same as spinning-cube). Visual language consistency; no new camera modes in 2.1 (those belong to Phase 2.3).
- No effects pipeline — static mesh + orbit rotation. Keeps the example a pure demonstration of OBJ loading.

### Per-Model Shading (999.x scaffolding)
- Add `shading: Option<ShadingRamp>` field to `Mesh` (scaffolded, default `None` = scene-default ramp applies). Hooks the 999.x backlog incrementally without locking API.
- No DSL-level `.shading()` method in 2.1 — set only via struct literal / builder (`Mesh { shading: Some(ramp), ..default }`). DSL surface waits for 999.x multi-strategy work.
- No named ramp presets (voxcii/dotted/blocks) in 2.1 — single voxcii default + custom via `ShadingRamp::new(chars, light_dir)`. Presets land with 999.x.
- Viewer uses the single default ramp for all three models. When 999.x lands (post-M2 or absorbed later), revisit the viewer to showcase variety.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `crates/happyterminals-renderer/src/cube.rs` — `Cube` primitive with `VERTICES: [Vec3; 8]` + `INDICES: [[usize; 3]; 12]`, CCW winding convention. `Cube::mesh()` helper to be added here returns the same triangle set via the new `Mesh` type.
- `crates/happyterminals-renderer/src/shading.rs` — `ShadingRamp` with configurable `chars: &[char]` + `light_dir` exists and is applied in the rasterizer. Ready to be optionally overridden per-`Mesh`.
- `crates/happyterminals-renderer/src/rasterizer.rs` — takes triangles + projection today. After unification it accepts `&Mesh` (triangles expanded from `mesh.indices` + `mesh.vertices`).
- `crates/happyterminals/examples/spinning-cube/main.rs` — 43-LOC reference for example shape: `scene().layer().build()`, `run_scene(scene, FrameSpec, tick)`. The model-viewer mirrors this with input handling over rotation.
- `examples/models/{bunny,cow,teapot}.obj` — 3 pre-imported OBJs (~200 KB each) awaiting this phase.

### Established Patterns
- `glam::Vec3` is the vector type across the renderer crate (no ad-hoc types).
- CCW winding when viewed from outside (cube.rs convention — preserved for loaded meshes via `tobj`'s winding consistency check + optional flip).
- Errors are `Result`-based on public API; `unwrap_used = "deny"` + `expect_used = "deny"` lints already configured in `crates/happyterminals-renderer/Cargo.toml`.
- `[lints.rust] unsafe_code = "forbid"` in renderer — all OBJ loading code stays safe.
- Workspace deps pinned: `glam 0.32.1`, `thiserror 2.0`. `tobj 4.0.3` must be added to `[workspace.dependencies]` and surfaced via `dep.workspace = true` in renderer.
- `criterion` benches live as `[[bench]]` in renderer Cargo.toml; existing `renderer_alloc` bench sets the per-frame-allocation pattern that OBJ-rendered frames must maintain.

### Integration Points
- `happyterminals-renderer`'s public `lib.rs` re-exports: new additions `mesh::Mesh`, `mesh::MeshError`, `mesh::LoadStats`, `mesh::load_obj`.
- `happyterminals-scene` / `happyterminals-dsl`: the `.cube()` builder pattern is the template for a new `.obj(path)` or `.mesh(mesh)` node — exact DSL extension to be designed in the planner stage.
- `happyterminals` meta crate `prelude`: add `pub use happyterminals_renderer::mesh::{Mesh, load_obj, MeshError};`.

<specifics>
## Specific Ideas

- Model-viewer example: Left/Right arrow keys cycle between bunny → cow → teapot → bunny (wrap). Current model name in title ("happyterminals - Model Viewer: bunny"). Orbit camera with same default pose as spinning-cube (azimuth 0, elevation 0.4, distance 4.0 — may need per-model distance auto-fit since the three models differ in scale).
- Fixture corpus MUST include degenerate cases: missing normals, flipped winding, degenerate triangle (3 collinear points), ngon (5+ vertex face forcing triangulation), empty file, truncated file mid-triangle. Proptest fuzzer over the corpus asserts no panics.
- Per-frame allocation: loading is a one-shot up-front cost, so it's outside the hot path. But the hot path (rasterizing a loaded Mesh) MUST stay `O(triangles)` with zero per-frame allocations — enforced by extending the existing `renderer_alloc` criterion bench to cover a loaded Mesh case.

</specifics>
