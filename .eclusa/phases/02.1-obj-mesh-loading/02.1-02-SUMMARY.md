# Phase 2.1 Plan 02: Renderer::draw(&Mesh) + Bunny Bench Summary

**Refactored `Renderer::draw` to accept `&Mesh` instead of hardcoded `Cube::VERTICES`/`INDICES`/`FACE_NORMALS` — unifying cube rendering and loaded-mesh rendering under a single hot path that preserves REND-09 zero-per-frame-allocation discipline, proven at 12-triangle (cube) and ~5000-triangle (Stanford bunny) scale via criterion bench + capacity-stability proxy tests.**

- `Renderer::draw` now takes `&Mesh` as 2nd positional parameter. Hot loop iterates `mesh.indices` + `mesh.normals` (length-matched by `Mesh` invariant — enforced by `load_obj` and `Cube::mesh()`). No Cube-vs-loaded-mesh branching in the rasterizer.

## Decisions Made

### 1. `scene_fit_near` option (b) — delete constant + make mesh-aware

**Decision:** Removed `CUBE_BOUNDING_RADIUS` constant entirely and rewrote `scene_fit_near(&OrbitCamera, &Mesh) -> f32` to derive radius from `mesh.bounding_sphere().1`.

**Rationale:** The plan offered two variants (§Step 1). Keeping a cube-specific constant side-by-side with a mesh-aware variant would have been dead code after the refactor and contradicted REND-06's "single hot path" premise. Plan 03's model-viewer needs this anyway — bunny (~0.2 unit) and teapot (~3.0 unit) radii differ by 15×, so a cube-only constant would mis-position the near plane for both. Deleting it now avoids a follow-up cleanup in Plan 03.

### 2. Backend pre-builds cube mesh once

**Decision:** `run_scene` constructs `Cube::mesh()` in scope before the tokio `loop`, threading `&cube_mesh` through `walk_and_render → render_node → NodeKind::Cube` branch.

**Rationale:** The alternative (option b in plan) — building `Cube::mesh()` inside `render_node` per frame — heap-allocates vertex/index/normal Vecs every frame per cube node. That's a direct REND-09 violation. Single up-front construction pays the heap cost once at scene startup (negligible). Future multi-cube scenes will share the same `&cube_mesh` borrow.

### 3. Per-mesh shading override via `Option<&ShadingRamp>` borrow

**Decision:** `let effective_ramp: &ShadingRamp<'_> = mesh.shading.as_ref().unwrap_or(shading);`

**Rationale:** `mesh.shading.as_ref()` returns `Option<&ShadingRamp<'static>>`; `unwrap_or(shading)` picks the scene-default `&ShadingRamp<'_>` when no override. Both variants live in borrow form — zero heap alloc, zero clone. The type system handles the lifetime dance (`'static` outlives `'_`). This is the minimum-effort way to surface the 999.x scaffold.

### 4. Defensive index bounds check retained in `draw`

**Decision:** Added explicit `if vi0 >= mesh.vertices.len() || ... { continue; }` guard in the hot loop.

**Rationale:** `Mesh` is a public struct with `pub vertices` / `pub indices` fields — callers can construct hand-rolled meshes bypassing `load_obj`'s validation. A panic on index out-of-bounds would undermine `unsafe_code = forbid` for no gain. The check is a single branch on a register-resident length — negligible overhead vs the rasterization cost of even a single triangle.
