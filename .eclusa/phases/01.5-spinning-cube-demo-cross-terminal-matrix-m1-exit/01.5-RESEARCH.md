## Summary

This is a pure integration phase. All building blocks exist and are tested (164 tests, all passing). The job is wiring them together in `run_scene()`, writing the spinning-cube example, extending the 1-cell bytes test to the full Scene stack, updating the README, and documenting the cross-terminal manual verification matrix.

The critical integration point is `run_scene()`: it must walk `SceneIr` nodes, extract reactive rotation props via `AnyReactive::get_any_untracked()`, apply them to the `OrbitCamera`, call `Renderer::draw()`, then `Pipeline::run_frame()`, then copy `Grid` to ratatui's frame buffer. The existing `run()` function (closure-based) provides the exact loop template -- `run_scene()` replaces the user closure with scene-aware rendering logic.

**Primary recommendation:** Implement `run_scene()` as a sibling function to `run()` in the backend crate, following the identical `tokio::select!` loop pattern but with Scene-driven rendering instead of a user closure. The example should be under 60 LOC (leaving margin for the 100 LOC limit). The bytes test extends the existing `SharedBuf` pattern from `one_cell_change.rs`.

## Architecture Patterns

### run_scene() Integration Path

The core integration is a new `run_scene()` function in `happyterminals-backend-ratatui/src/run.rs` that follows the exact same `tokio::select!` loop as `run()` but replaces the user closure with scene-driven rendering:

```
run_scene(scene: Scene, spec: FrameSpec)
  |
  +-- install_panic_hook()
  +-- TerminalGuard::acquire()
  +-- Extract OrbitCamera from scene.camera()
  +-- Create Renderer::new()
  +-- Create Projection from terminal size
  +-- Create ShadingRamp::default()
  +-- Clone scene's Pipeline (if any)
  |
  +-- tokio::select! loop {
  |     Frame tick:
  |       1. grid.resize(frame.area())
  |       2. Walk scene.ir().nodes() recursively
  |       3. For each NodeKind::Cube node:
  |          a. Read "rotation" prop via get_any_untracked() -> f32
  |          b. Apply to camera.azimuth (or node transform)
  |          c. Update Projection viewport from grid dimensions
  |          d. renderer.draw(&mut grid, &camera, &projection, &shading)
  |       4. If pipeline exists: pipeline.run_frame(&mut grid, dt)
  |       5. *frame.buffer_mut() = grid.deref().clone()
  |     
  |     Event stream:
  |       Same as run() -- map_event, is_quit_event, input_signals
  |   }
```

### Key Wiring Detail: Rotation Signal -> OrbitCamera

The `CubeBuilder::rotation(&signal)` stores a `PropValue::Reactive(Box<dyn AnyReactive>)` in the node's props map under key `"rotation"`. At render time, `run_scene()` reads this:

```rust
// For each Cube node:
if let Some(prop) = node.props.get("rotation") {
    if let Some(angle) = prop.read::<f32>() {  // calls get_any_untracked internally
        camera.azimuth = angle;
    }
}
```

Note: `PropValue::read::<f32>()` calls `get_any()` which subscribes. For the render path we need `get_any_untracked()`. However, `PropValue::read()` uses `get_any()`. There are two options:
1. Add a `read_untracked::<T>()` method to `PropValue` (preferred -- clean separation)
2. Read via `get_any_untracked()` directly on the `Reactive` variant

**Recommendation:** Add `PropValue::read_untracked::<T>()` which calls `get_any_untracked()` instead of `get_any()`. This is critical -- the render loop must NOT create signal subscriptions (REACT-07).

### Signal Update Pattern: dt-driven rotation

Each frame tick, the example increments the rotation signal:

```rust
// In the example's setup:
let rotation = Signal::new(0.0_f32);

// In run_scene()'s frame tick (or the example itself):
// Option A: run_scene drives the signal from dt
// Option B: the example passes a closure that updates signals
```

Per D-01 and D-03: the signal update happens inside `run_scene()`. The frame loop knows `dt` (from `FrameSpec::frame_duration()`). But `run_scene()` doesn't know which signals to update -- that's the example's concern.

**Resolution:** `run_scene()` should accept a `tick_fn: impl FnMut(Duration)` callback that the example uses to update signals each frame. This is the bridge between "scene-aware rendering" and "user-driven signal updates":

```rust
pub async fn run_scene<F>(
    scene: Scene,
    spec: FrameSpec,
    mut tick_fn: F,
) -> Result<(), Box<dyn std::error::Error>>
where
    F: FnMut(Duration, &InputSignals),
```

The example provides: `|dt, _input| rotation.set(rotation.untracked() + dt.as_secs_f32() * ROTATION_SPEED)`

Alternative: `run_scene()` takes only `Scene + FrameSpec`, and signal updates happen via Effects from the reactive system. However, Effects require Owner/scope infrastructure which adds complexity. The tick callback is simpler and sufficient for M1.

### Anti-Patterns to Avoid
- **Subscribing to signals in the render loop:** Use `read_untracked()` / `get_any_untracked()` only. `read()` / `get_any()` creates subscriptions that accumulate.
- **Full grid clone every frame when nothing changed:** The ratatui diff engine handles this -- just always write the full grid. The diff engine compares with the previous frame.
- **Adding Pipeline as a dependency to the backend crate's Cargo.toml without going through Scene:** `run_scene()` takes a `Scene` which owns the `Pipeline`. The backend only needs `happyterminals-scene` and `happyterminals-renderer` as new deps.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ANSI diff output | Custom terminal diff | ratatui's internal diff engine | Already proven by one_cell_change.rs test |
| Terminal cleanup | Custom signal/panic handlers | TerminalGuard RAII + install_panic_hook() | Already exists and tested |
| Scene tree walking | Complex visitor pattern | Simple recursive fn over `SceneIr::nodes()` | Tree is shallow (Layer -> Cube), no need for visitor |
| Frame timing | Manual Instant tracking | tokio::time::interval (existing in run()) | Already working in run() |

## Common Pitfalls

### Pitfall 1: Signal Subscription Leak in Render Loop
**What goes wrong:** Calling `PropValue::read()` (which calls `get_any()`) inside the render loop creates a new signal subscription every frame, leading to memory growth and eventual slowdown.
**Why it happens:** `read()` is the obvious API; `read_untracked()` doesn't exist yet.
**How to avoid:** Add `PropValue::read_untracked()` that calls `get_any_untracked()`. Use exclusively in the render path.
**Warning signs:** Memory growth during 10-minute soak test.

### Pitfall 2: Backend Crate Dependency Creep
**What goes wrong:** `run_scene()` needs to know about `Scene`, `Renderer`, `Pipeline`, `OrbitCamera`, etc. If all those become direct dependencies of the backend crate, the dependency graph gets tangled.
**Why it happens:** Natural tendency to import everything needed.
**How to avoid:** The backend crate needs new deps: `happyterminals-scene`, `happyterminals-renderer`, `happyterminals-pipeline`. This is acceptable for `run_scene()`. Alternatively, put `run_scene()` in the meta crate. But the meta crate is currently a thin re-export layer. **Recommendation:** Add the deps to the backend crate -- it's the runtime crate and should own the run loop.
**Warning signs:** Circular dependencies in `cargo tree`.

### Pitfall 3: Projection Not Updated on Resize
**What goes wrong:** Viewport dimensions in `Projection` go stale after terminal resize, causing stretched/squished rendering.
**Why it happens:** `Projection` is created once with initial terminal size.
**How to avoid:** Recreate Projection from `grid.area` dimensions at the start of each frame tick (after `grid.resize(frame.area())`).

### Pitfall 4: Pipeline Ownership in run_scene()
**What goes wrong:** `Scene` owns `pipeline: Option<Pipeline>` but `Pipeline::run_frame()` takes `&mut self`. If Scene is immutable, can't call run_frame().
**Why it happens:** Scene is designed as a validated, sealed container.
**How to avoid:** Extract the pipeline from Scene at run_scene() setup time: `let mut pipeline = scene.pipeline().cloned();` -- Pipeline is not Clone. Need to add `pub fn take_pipeline(&mut self) -> Option<Pipeline>` to Scene, or destructure Scene into parts. **Recommendation:** Add `Scene::into_parts() -> (SceneIr, CameraConfig, Option<Pipeline>)` to consume the Scene and get mutable ownership of all parts.

### Pitfall 5: Example Line Count Overflow
**What goes wrong:** Example exceeds 100 LOC with verbose imports, comments, error handling.
**Why it happens:** Rust is verbose with imports and error types.
**How to avoid:** Use `happyterminals::prelude::*` (1 import line). Use `#[tokio::main]`. Keep comments minimal. Target 50-60 LOC to leave margin.

## Code Examples

### run_scene() skeleton (verified against existing run.rs pattern)

```rust
// Source: Derived from crates/happyterminals-backend-ratatui/src/run.rs
pub async fn run_scene<F>(
    scene: Scene,
    spec: FrameSpec,
    mut tick_fn: F,
) -> Result<(), Box<dyn std::error::Error>>
where
    F: FnMut(Duration, &InputSignals),
{
    install_panic_hook();
    let _guard = TerminalGuard::acquire()?;

    if let Some(ref title) = spec.title {
        let _ = crossterm::execute!(io::stdout(), SetTitle(title.as_str()));
    }

    let backend = CrosstermBackend::new(io::stdout());
    let mut terminal = Terminal::new(backend)?;
    let mut events = EventStream::new();

    let mut tick = interval(spec.frame_duration());
    tick.set_missed_tick_behavior(MissedTickBehavior::Delay);

    let (w, h) = crossterm::terminal::size()?;
    let input_signals = InputSignals::new(w, h);
    let mut grid = Grid::new(Rect::new(0, 0, w, h));

    // Extract scene parts for mutable access
    let (ir, camera_config, pipeline) = scene.into_parts();
    let CameraConfig::Orbit(mut camera) = camera_config;
    let mut renderer = Renderer::new();
    let shading = ShadingRamp::default();
    let mut pipeline = pipeline;
    let dt = spec.frame_duration();

    loop {
        tokio::select! {
            _ = tick.tick() => {
                tick_fn(dt, &input_signals);

                terminal.draw(|frame| {
                    grid.resize(frame.area());
                    let projection = Projection {
                        viewport_w: grid.area.width,
                        viewport_h: grid.area.height,
                        ..Projection::default()
                    };

                    // Read rotation from scene nodes
                    walk_and_render(&ir, &mut grid, &mut renderer, &mut camera, &projection, &shading);

                    // Apply pipeline effects
                    if let Some(ref mut pipe) = pipeline {
                        pipe.run_frame(&mut grid, dt);
                    }

                    *frame.buffer_mut() = grid.deref().clone();
                })?;
            }
            maybe_event = events.next() => {
                // Same event handling as run()
            }
        }
    }
}

fn walk_and_render(
    ir: &SceneIr,
    grid: &mut Grid,
    renderer: &mut Renderer,
    camera: &mut OrbitCamera,
    projection: &Projection,
    shading: &ShadingRamp<'_>,
) {
    for node in ir.nodes() {
        render_node(node, grid, renderer, camera, projection, shading);
    }
}

fn render_node(
    node: &SceneNode,
    grid: &mut Grid,
    renderer: &mut Renderer,
    camera: &mut OrbitCamera,
    projection: &Projection,
    shading: &ShadingRamp<'_>,
) {
    match &node.kind {
        NodeKind::Cube => {
            // Read reactive rotation prop
            if let Some(prop) = node.props.get("rotation") {
                if let Some(angle) = prop.read_untracked::<f32>() {
                    camera.azimuth = angle;
                }
            }
            renderer.draw(grid, camera, projection, shading);
        }
        NodeKind::Layer { .. } | NodeKind::Group => {
            for child in &node.children {
                render_node(child, grid, renderer, camera, projection, shading);
            }
        }
        NodeKind::Custom(_) => {}
    }
}
```

### spinning-cube example skeleton (~55 LOC target)

```rust
// Source: Derived from examples/static_grid.rs + DSL tests/hello_world.rs
use happyterminals::prelude::*;
use std::time::Duration;

const ROTATION_SPEED: f32 = 1.5; // radians per second

#[tokio::main(flavor = "current_thread")]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let (result, _owner) = create_root(|| {
        let rotation = Signal::new(0.0_f32);
        let rotation_clone = rotation.clone();

        let scene = scene()
            .camera(OrbitCamera {
                azimuth: 0.0,
                elevation: 0.4,
                distance: 4.0,
                ..OrbitCamera::default()
            })
            .layer(0, |l| l.cube().rotation(&rotation))
            .pipeline(Pipeline::new().with(effects::dissolve(Duration::from_secs(0))))
            .build()?;

        Ok((scene, rotation_clone))
    });

    let (scene, rotation) = result?;

    run_scene(
        scene,
        FrameSpec {
            title: Some("happyterminals - Spinning Cube".into()),
            ..FrameSpec::default()
        },
        |dt, _input| {
            let current = rotation.untracked();
            rotation.set(current + dt.as_secs_f32() * ROTATION_SPEED);
        },
    )
    .await
}
```

### Scene-level bytes test skeleton

```rust
// Source: Extends crates/happyterminals-backend-ratatui/tests/one_cell_change.rs
use happyterminals_core::{Grid, Rect, Signal, create_root};
use happyterminals_dsl::prelude::*;
use happyterminals_renderer::{Renderer, Projection, ShadingRamp};
use happyterminals_scene::CameraConfig;
use ratatui::Terminal;
use ratatui_crossterm::CrosstermBackend;
use std::ops::Deref;

// SharedBuf: same as one_cell_change.rs

#[test]
fn scene_one_cell_change_minimal_bytes() {
    let (_result, _owner) = create_root(|| {
        let rotation = Signal::new(0.0_f32);
        let scene = scene()
            .camera(OrbitCamera { elevation: 0.3, ..OrbitCamera::default() })
            .layer(0, |l| l.cube().rotation(&rotation))
            .build()
            .unwrap();

        let (ir, camera_config, _pipeline) = scene.into_parts();
        let CameraConfig::Orbit(mut camera) = camera_config;
        let mut renderer = Renderer::new();
        let projection = Projection { viewport_w: 80, viewport_h: 24, ..Projection::default() };
        let shading = ShadingRamp::default();

        let buf = SharedBuf::new();
        let backend = CrosstermBackend::new(buf.clone());
        let mut terminal = Terminal::new(backend).unwrap();

        // Frame 1: initial render
        terminal.draw(|frame| {
            let mut grid = Grid::new(frame.area());
            // walk and render...
            *frame.buffer_mut() = grid.deref().clone();
        }).unwrap();

        buf.clear();

        // Tiny signal change
        rotation.set(0.001);

        // Frame 2: re-render with nudged rotation
        terminal.draw(|frame| {
            let mut grid = Grid::new(frame.area());
            // walk and render with updated camera.azimuth...
            *frame.buffer_mut() = grid.deref().clone();
        }).unwrap();

        let delta = buf.len();
        assert!(delta <= 50, "Expected <= 50 bytes, got {delta}");
        assert!(delta > 0, "Expected some bytes, got 0");
    });
}
```

## Open Questions

1. **Pipeline mutability through Scene**
   - What we know: `Scene` owns `Option<Pipeline>`, Pipeline::run_frame needs `&mut self`
   - What's unclear: Best API to extract mutable pipeline from Scene
   - Recommendation: Add `Scene::into_parts()` to destructure Scene. This is clean and avoids interior mutability.

2. **Per-node vs camera rotation**
   - What we know: D-03 says "azimuth signal is incremented each frame tick" and "OrbitCamera reads it"
   - What's unclear: The rotation prop is on the Cube node, but it controls camera azimuth (not node transform). This is a simplification for M1.
   - Recommendation: For M1, rotation prop on Cube drives camera.azimuth. Per-node transforms are M2.

3. **tick_fn signature**
   - What we know: run_scene() needs a way for users to update signals each frame
   - What's unclear: Whether tick_fn should also receive `&InputSignals` or just `Duration`
   - Recommendation: `FnMut(Duration, &InputSignals)` -- gives the user both time and input access.

4. **Effect choice for example**
   - What we know: No dedicated `vignette()` effect. Have `dissolve()`, `darken()`, `fade_from()`, etc.
   - Available options: `dissolve(Duration::ZERO)` (instant, effectively a noop), `darken(0.3, Duration::ZERO)` (instant vignette-like), or `coalesce(Duration::from_secs(1))` (visual startup effect)
   - Recommendation: Use `coalesce(Duration::from_secs(1))` -- it provides a visually interesting startup effect (characters reform from dissolved state), demonstrating the pipeline without ongoing per-frame cost after completion.
