# Phase 01.2: Pipeline + tachyonfx Adapter - Research

**Researched:** 2026-04-15
**Domain:** Effect pipeline architecture, tachyonfx integration, trait-object composition
**Confidence:** HIGH

## Summary

Phase 1.2 builds the `happyterminals-pipeline` crate: our `Effect` trait, the `Pipeline` executor, and a `TachyonAdapter` that bridges tachyonfx shaders into our system. The architecture decisions are locked (CONTEXT.md) and well-aligned with the existing codebase.

The critical technical challenge is the **Grid mutability bridge**: Grid's `inner_mut()` is currently `pub(crate)` in happyterminals-core, but TachyonAdapter (in happyterminals-pipeline) needs `&mut Buffer` access to pass to tachyonfx's `Shader::process()`. This requires either promoting `inner_mut()` to `pub` or adding a public `as_buffer_mut()` method. The CONTEXT.md already identifies this issue and notes `Deref` only provides read access.

tachyonfx 0.25's `Shader` trait is well-documented: `process(&mut self, Duration, &mut Buffer, Rect) -> Option<Duration>` where `Duration` is tachyonfx's own type (`{milliseconds: u32}`), not `std::time::Duration`. Conversion is available via `From<std::time::Duration>`. The `Effect` struct wraps `Box<dyn Shader>` and delegates all Shader methods. The `done()` / `running()` lifecycle model maps cleanly to our `EffectState::Running | Done`.

**Primary recommendation:** Promote Grid's buffer access to `pub` (e.g., `pub fn buffer_mut(&mut self) -> &mut Buffer`), implement the Effect trait with `apply(&mut self, grid: &mut Grid, dt: Duration) -> EffectState`, wrap tachyonfx effects via `TachyonAdapter` that converts our `std::time::Duration` dt to tachyonfx's `Duration`, and benchmark the 10-effect pipeline on a 200x60 grid to prove O(cells) overhead.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** `Effect` trait signature: `fn apply(&mut self, grid: &mut Grid, dt: Duration) -> EffectState` where `EffectState` is `Running | Done`. Effects are stateful (own timers, progress counters).
- **D-02:** Our trait keeps the name `Effect`. `tachyonfx::Effect` is re-exported as `Fx` in our public surface. This resolution is committed to code + CHANGELOG before any Pipeline consumer code.
- **D-03:** `Pipeline = Vec<Box<dyn Effect>>` -- trait objects, not generic chains. Runtime-constructable for JSON recipes (Phase 3.2) and Python (Phase 4.x).
- **D-04:** `TachyonAdapter<S: tachyonfx::Shader>` wraps a tachyonfx effect as one of our `Effect` trait objects. Real `Duration` dt is forwarded to tachyonfx's timing system.
- **D-05:** tachyonfx uses `ratatui::buffer::Buffer` directly for its effects. Since our Grid derefs to Buffer, the adapter can pass `grid.deref_mut()` (via inner_mut()) to tachyonfx. The adapter translates our `dt: Duration` into tachyonfx's frame-timing model.
- **D-06:** Pipeline runs effects in sequence (Vec order). Each effect's `apply()` receives the Grid after previous effects have mutated it.
- **D-07:** When an effect returns `EffectState::Done`, it stays in the Vec but is skipped on subsequent frames. Pipeline itself returns `Done` when all effects are `Done`. Effects can be reset via a `reset()` method.
- **D-08:** Whole-grid composition only. No per-object effect loops.
- **D-09:** All errors on public surface are `Result`-based. `clippy::unwrap_used` and `clippy::expect_used` denied crate-wide.
- **D-10:** Invariant: effects mutate Grid only through Pipeline::apply(). Documented in public docstrings.
- **D-11:** Criterion benchmark proves `Pipeline::run_frame()` overhead is O(cells), not O(objects x effects). Benchmark runs the 10 wired effects on a 200x60 Grid for 60 frames.

### Pattern 1: Effect Trait Design

**What:** Our `Effect` trait bridges the lifecycle model (Running/Done) with Grid mutation.

**When to use:** Every visual effect in the system implements this trait.

**Recommended signatures:**

```rust
use std::time::Duration;
use happyterminals_core::Grid;
use ratatui_core::layout::Rect;

/// State of an effect after one frame of processing.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum EffectState {
    /// Effect is still animating; call apply() next frame.
    Running,
    /// Effect has completed; Pipeline will skip it.
    Done,
}

/// A visual effect that mutates a Grid over time.
pub trait Effect: std::fmt::Debug {
    /// Apply this effect for one frame.
    fn apply(&mut self, grid: &mut Grid, dt: Duration) -> EffectState;

    /// Whether this effect has completed.
    fn is_done(&self) -> bool;

    /// Reset effect to initial state for reuse/looping.
    fn reset(&mut self);

    /// Human-readable name for debugging and future JSON round-trip.
    fn name(&self) -> &'static str;
}
```

**Discretion recommendation:** Include `is_done()`, `reset()`, and `name()` as required methods. `is_done()` enables Pipeline to skip without re-calling `apply()`. `reset()` is needed for D-07 (effect reuse). `name()` maps to tachyonfx's `Shader::name()` and future JSON recipes. All three have trivial implementations. `Debug` bound enables diagnostic printing.

### Pattern 2: TachyonAdapter

**What:** Generic adapter wrapping any `tachyonfx::Shader` as our `Effect`.

**Key insight from research:** tachyonfx's `Shader::process()` takes `tachyonfx::Duration` (a `{milliseconds: u32}` struct), not `std::time::Duration`. Conversion is via `From<std::time::Duration>`. The `process()` method returns `Option<Duration>` (overflow time if done) and the `done()` method tracks completion.

```rust
use tachyonfx::Shader;

/// Adapts any tachyonfx Shader into our Effect trait.
#[derive(Debug)]
pub struct TachyonAdapter<S: Shader> {
    shader: S,
    area: Rect,
    completed: bool,
}

impl<S: Shader + 'static> Effect for TachyonAdapter<S> {
    fn apply(&mut self, grid: &mut Grid, dt: Duration) -> EffectState {
        if self.completed {
            return EffectState::Done;
        }
        let tfx_duration = tachyonfx::Duration::from(dt);
        let buf = grid.buffer_mut(); // needs Grid to expose &mut Buffer
        let area = if self.area == Rect::default() { grid.area } else { self.area };
        self.shader.process(tfx_duration, buf, area);
        if self.shader.done() {
            self.completed = true;
            EffectState::Done
        } else {
            EffectState::Running
        }
    }

    fn is_done(&self) -> bool { self.completed }

    fn reset(&mut self) {
        self.shader.reset();
        self.completed = false;
    }

    fn name(&self) -> &'static str { self.shader.name() }
}
```

### Pattern 3: Pipeline Executor

**What:** Sequential effect chain using trait objects.

```rust
pub struct Pipeline {
    effects: Vec<Box<dyn Effect>>,
}

impl Pipeline {
    pub fn new() -> Self { Self { effects: Vec::new() } }

    /// Builder: add an effect to the end of the chain.
    pub fn add(mut self, effect: impl Effect + 'static) -> Self {
        self.effects.push(Box::new(effect));
        self
    }

    /// Process one frame: apply all non-done effects in sequence.
    pub fn run_frame(&mut self, grid: &mut Grid, dt: Duration) -> EffectState {
        let mut all_done = true;
        for effect in &mut self.effects {
            if !effect.is_done() {
                let state = effect.apply(grid, dt);
                if state == EffectState::Running {
                    all_done = false;
                }
            }
        }
        if all_done { EffectState::Done } else { EffectState::Running }
    }

    /// Reset all effects for replay.
    pub fn reset(&mut self) {
        for effect in &mut self.effects {
            effect.reset();
        }
    }
}
```

**Discretion recommendation:** Use the builder pattern (`Pipeline::new().add(fx).add(fx2)`) rather than `from_effects(vec![...])`. The builder is more ergonomic for Rust users and doesn't preclude `from_effects` later. Pipeline should own `Vec<Box<dyn Effect>>` directly (no wrapper struct with metadata) -- metadata like frame count can be added when needed (Phase 1.4+).

### Pattern 4: Grid Buffer Access (critical prerequisite)

**What:** Grid must expose `&mut Buffer` for TachyonAdapter.

**Current state:** `Grid::inner_mut()` is `pub(crate)` in happyterminals-core. The pipeline crate cannot call it.

**Recommendation:** Add a `pub fn buffer_mut(&mut self) -> &mut Buffer` method to Grid. This is the cleanest path:
- `DerefMut` was intentionally not implemented (comment in grid.rs: "NO DerefMut -- writes go through put_str only")
- A named method makes the intent clear: "you're getting raw buffer access for effect processing"
- Document that this is for the pipeline crate's TachyonAdapter, not general use
- The invariant "effects mutate Grid only through Pipeline::apply()" is enforced by convention + docs, not by making Buffer inaccessible

Alternative: implement `DerefMut` -- but this was explicitly rejected in the existing code.

### Anti-Patterns to Avoid
- **Re-instantiating effects each frame:** tachyonfx effects are stateful; creating a new `dissolve()` every frame means animations never progress (Pitfall 16)
- **Hardcoded frame duration:** Pass real `dt` from the frame loop, not `Duration::from_millis(16)`. Pause-friendly.
- **Two types named `Effect` in scope:** Always use `Fx` for `tachyonfx::Effect` and `Effect` for ours. The `Fx` alias must be committed before any consumer code.
- **Per-object effect loops:** Pipeline applies to the whole Grid. No `for obj in objects { for fx in effects { ... } }` pattern (Pitfall 30).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Visual effects (dissolve, fade, sweep, etc.) | Custom cell-by-cell manipulation | tachyonfx 0.25 via TachyonAdapter | 50+ battle-tested effects with timing, easing, cell filtering |
| Effect timing/easing | Custom lerp + timer per effect | tachyonfx's EffectTimer (inside each Shader) | Handles reversals, ping-pong, repeat, easing curves |
| Duration arithmetic | Custom time tracking | tachyonfx::Duration with From<std::time::Duration> | Already handles millisecond precision, overflow detection |
| Benchmarking | Custom timing loops | criterion 0.8 | Statistical rigor, outlier detection, comparison reports |
| Error types | Manual Error impls | thiserror 2.0 | Derive-based, zero boilerplate |

**Key insight:** tachyonfx already handles the hard parts of terminal effects -- cell iteration with filtering, color space conversion, timing with easing curves, spatial patterns. Our adapter is thin glue, not a reimplementation.

## Common Pitfalls

### Pitfall 1: tachyonfx Duration is not std::time::Duration
**What goes wrong:** Passing `std::time::Duration` directly to `Shader::process()` fails to compile or silently truncates.
**Why it happens:** tachyonfx defines its own `Duration { milliseconds: u32 }`. The two types are not interchangeable without conversion.
**How to avoid:** Always convert: `tachyonfx::Duration::from(std_duration)`. The TachyonAdapter does this in one place.
**Warning signs:** Compilation error on `process()` call; effects completing instantly (u32 overflow from nanoseconds).

### Pitfall 2: tachyonfx Shader requires Debug + clone_box()
**What goes wrong:** `TachyonAdapter` cannot be stored as `Box<dyn Effect>` if Effect doesn't have the right bounds.
**Why it happens:** tachyonfx's `Shader` trait requires `Debug` and `clone_box()` (for `Box<dyn Shader>` cloning). Our `Effect` trait should require `Debug` too for consistency. However, `clone_box()` is internal to tachyonfx -- our Effect trait does NOT need Clone.
**How to avoid:** Add `Debug` bound to our `Effect` trait. Do not add `Clone` -- Pipeline uses `&mut` references, never clones effects.
**Warning signs:** Missing Debug impl on TachyonAdapter; confusing error messages about missing trait bounds.

### Pitfall 3: Grid::inner_mut() is pub(crate)
**What goes wrong:** TachyonAdapter in the pipeline crate cannot get `&mut Buffer` from Grid.
**Why it happens:** Grid was designed to restrict mutation to `put_str()`. The pipeline crate is a new consumer that needs raw buffer access.
**How to avoid:** Add `pub fn buffer_mut(&mut self) -> &mut Buffer` to Grid in happyterminals-core before implementing the adapter.
**Warning signs:** Compilation error in adapter.rs when trying to call `grid.inner_mut()`.

### Pitfall 4: tachyonfx ThreadSafetyMarker
**What goes wrong:** If the `sendable` feature is enabled on tachyonfx, `Shader` requires `Send`, which may conflict with non-Send types in the adapter.
**Why it happens:** tachyonfx has a `features.rs` that conditionally adds `Send` bound via `ThreadSafetyMarker`.
**How to avoid:** Do NOT enable the `sendable` feature on tachyonfx. The default (no feature) makes `ThreadSafetyMarker` a no-op blanket trait. Our pipeline is single-threaded anyway.
**Warning signs:** Unexpected `Send` bound errors when wrapping effects.

### Pitfall 5: Effect name clash not resolved before consumer code
**What goes wrong:** A module imports both `happyterminals_pipeline::Effect` and `tachyonfx::Effect`, causing ambiguity errors or silent wrong dispatch.
**Why it happens:** Both crates export a type named `Effect`.
**How to avoid:** D-02 is locked: re-export `tachyonfx::Effect` as `Fx` in pipeline's public surface. Commit this alias in the first task, before any consumer code. Never `pub use tachyonfx::Effect` without renaming.
**Warning signs:** `ambiguous name` compiler errors; wrong `Effect` trait in scope.

### Pitfall 6: Forgetting to pass area to tachyonfx
**What goes wrong:** Effects apply to `Rect::default()` (0x0), producing no visible output.
**Why it happens:** tachyonfx's `Shader::process()` takes an explicit `area: Rect`. If the adapter doesn't pass Grid's area, effects have no region to operate on.
**How to avoid:** TachyonAdapter should default to `grid.area` (the full grid area via Deref) when no custom area is set.
**Warning signs:** Effects "run" (process returns Running) but no cells change.

**Note on "vignette" and "crt":** tachyonfx does not have dedicated `vignette()` or `crt()` functions. These effects are composed from primitives:
- **Vignette:** Use `darken()` with a spatial pattern/filter, or compose `effect_fn()` with custom cell iteration
- **CRT:** Compose `hsl_shift()` + `evolve()` + a scanline pattern via `effect_fn()`

The 10-effect list from CONTEXT.md should be adjusted to use actual tachyonfx function names. The list above provides 10 real, available effects.

### Criterion Benchmark Pattern
```rust
// benches/pipeline_overhead.rs
use criterion::{criterion_group, criterion_main, Criterion};
use std::time::Duration;
use happyterminals_pipeline::{Pipeline, EffectState};
use happyterminals_core::Grid;
use ratatui_core::layout::Rect;

fn bench_pipeline_run_frame(c: &mut Criterion) {
    let area = Rect::new(0, 0, 200, 60); // 12,000 cells
    let mut grid = Grid::new(area);
    let mut pipeline = Pipeline::new()
        // .add(effect1) ... add 10 effects
        ;

    let dt = Duration::from_millis(16); // ~60fps

    c.bench_function("pipeline_10fx_200x60_single_frame", |b| {
        b.iter(|| {
            pipeline.reset();
            pipeline.run_frame(&mut grid, dt);
        })
    });

    // 60-frame sustained run
    c.bench_function("pipeline_10fx_200x60_60_frames", |b| {
        b.iter(|| {
            pipeline.reset();
            for _ in 0..60 {
                pipeline.run_frame(&mut grid, dt);
            }
        })
    });
}

criterion_group!(benches, bench_pipeline_run_frame);
criterion_main!(benches);
```

## Open Questions

1. **ratatui-next-cell feature flag**
   - What we know: tachyonfx 0.25 has a `ratatui-next-cell` feature for ratatui 0.30's `Cell::diff_option` (replacing `Cell::skip`)
   - What's unclear: Whether we need to enable this feature given our Grid wraps ratatui-core 0.1's Buffer
   - Recommendation: Check if `Cell::skip` exists in ratatui-core 0.1 (it does based on grid.rs tests). If tachyonfx 0.25 depends on ratatui-core 0.1 without the feature, we're fine. Enable the feature only if cell mutation tests fail.

2. **Vignette and CRT effects**
   - What we know: CONTEXT.md lists "vignette" and "crt" in the 10 required effects, but tachyonfx has no dedicated functions for these
   - What's unclear: Whether the user expects literal tachyonfx functions or composed effects
   - Recommendation: Replace "vignette" with `darken()` (closest match) and "crt" with `hsl_shift()` (color manipulation). Document in plan that these are composed from tachyonfx primitives. Alternative: use `effect_fn()` to build custom vignette/crt as our own Effect impls.

3. **Grid::buffer_mut() vs promoting inner_mut()**
   - What we know: Grid's `inner_mut()` is `pub(crate)` and has a comment "Backend access to the underlying buffer for blit operations"
   - What's unclear: Whether to rename it and make it pub, or add a second method
   - Recommendation: Add `pub fn buffer_mut(&mut self) -> &mut Buffer` as a new method, leave `inner_mut()` as deprecated `pub(crate)` for now. This is cleaner than changing the existing method's visibility which might confuse the backend crate usage.
