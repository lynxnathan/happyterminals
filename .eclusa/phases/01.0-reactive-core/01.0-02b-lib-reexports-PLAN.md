## Why this plan exists (from VERIFICATION.md BLOCK 1)

VERIFICATION.md's fix list identifies a multi-plan race condition: plans 01.0-02, 01.0-03, and 01.0-04 were all declared `wave: 2` with `depends_on: ["01.0-01"]`, and all three appended different `pub use` lines to `crates/happyterminals-core/src/lib.rs`. In a parallel-wave execution model where each plan runs on a fresh base (each Wave-2 plan starts from Wave-1's output simultaneously), the second and third plans to land would not see the first plan's `lib.rs` edit â€” the three edits would collide, and whichever landed last would overwrite the earlier two (depending on merge semantics).

VERIFICATION.md offered three fix options:
- **(a)** Serialize 02/03/04 across waves â€” gives up all parallelism.
- **(b)** Centralize the crate-root `pub use` block in one dedicated plan â€” THIS PLAN. Preserves parallelism for 01.0-02/03/04.
- **(c)** Per-module re-export idioms â€” rejected because it requires `use ::*` globs at the crate root, which collides with the project's `clippy::pedantic` baseline.

This plan implements option (b). Plans 01.0-02/03/04 have been revised to:
- Remove `crates/happyterminals-core/src/lib.rs` from their `files_modified` lists.
- Delete any task step that appends `pub use` lines to `lib.rs`.
- Add an explicit out-of-scope clause pointing readers here.

## Decisions this plan executes

1. **Crate-root re-exports are a single-writer concern.** One plan owns `lib.rs`; this is that plan. No other Phase 1.0 plan may edit `crates/happyterminals-core/src/lib.rs`. (Plan 01.0-01 is the exception â€” it wrote the initial `lib.rs` skeleton with `#![forbid(unsafe_code)]` and `pub mod ...` declarations. This plan extends that skeleton; it does not replace it.)

2. **Test-util types stay nested.** `ManualClock` and `SeededRng` are accessed via `happyterminals_core::clock::test_util::ManualClock` / `happyterminals_core::rng::test_util::SeededRng`. They are NOT re-exported at the crate root. This is consistent with 01.0-04 Â§context.8 and keeps test-util signalled as "below the line" for production consumers.

3. **Re-export order is alphabetical by module** (signal â†’ memo â†’ batch â†’ effect â†’ owner â†’ clock â†’ rng â†’ error), matching the canonical order in CONTEXT.md Â§"API surface" and the `pub mod` declarations that 01.0-01 installed. Alphabetical-by-module keeps the diff stable across future phases that may add more modules.

4. **This plan's verify step MUST confirm that no sibling plan touched `lib.rs`.** The `<verify>` block asserts that the file's content since 01.0-01's landing was unchanged until this plan's edit â€” a minimal git-log check. If a sibling plan has accidentally edited `lib.rs`, this plan's verify fails and the orchestrator must resolve the conflict before this plan can land.
