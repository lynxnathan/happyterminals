<domain>
## Phase Boundary

**Goal:** Deliver a stable, documented, fine-grained reactive runtime in `happyterminals-core` ‚Äî the foundation every later phase consumes. Wrap Leptos's `reactive_graph 0.2.13` behind our own `Signal` / `Memo` / `Effect` / `Owner` / `Root` / `batch` / `untracked` / `SignalSetter` types so the public API is ours, not Leptos's, and a migration away from `reactive_graph` later is non-breaking.

**In scope:**
- `happyterminals-core` crate only (other crates stay empty stubs).
- Core reactive types wrapping `reactive_graph 0.2.13`: `Signal<T>`, `Memo<T>`, `Effect`.
- Owner/scope tree: `create_root`, `Owner`, `on_cleanup`.
- `batch(|| { ... })` API.
- Two-phase propagation + runtime cycle detection with clear panic message ("signal cycle A‚ÜíB‚ÜíA").
- `Signal::untracked()` / `Memo::untracked()` read APIs.
- Single-threaded runtime + `SignalSetter` channel for cross-thread writes (drained on the render thread).
- `Clock` trait (Production: `SystemClock` ‚Äî `std::time::Instant`) and `Rng` trait (Production: wrapping `rand::thread_rng`), both injectable for deterministic tests.
- Test suite: diamond-dependency test, cleanup/disposal test, 10k-scene-transition RSS ceiling test, cycle-detection test, cross-thread SignalSetter test, untracked-read test, batch semantics test.
- Public API documentation: every public item has `///` docs; Send/Sync rules stated in module-level doc.
- `REACT-01 ‚Ä¶ REACT-10` requirements satisfied.

**Out of scope (future phases):**
- Grid / Pipeline / renderer / scene graph ‚Äî Phases 1.1+
- Actual tachyonfx integration ‚Äî Phase 1.2
- Any Python bindings ‚Äî Phase 4.x
- `no_std` support ‚Äî not planned for v1
- Async runtime selection (tokio vs smol) ‚Äî deferred to Phase 1.1 (backend perimeter is where it matters)
- Full observer-graph inspector / debugging tool ‚Äî backlog

</domain>

### API surface (user-facing)
Minimal surface ‚Äî everything else is internal. Keep the public API as small as possible so future migration away from reactive_graph is cheap.

```rust
pub struct Signal<T>;
impl<T: 'static> Signal<T> {
    pub fn new(value: T) -> Self;
    pub fn get(&self) -> T where T: Clone;          // tracks subscription
    pub fn set(&self, value: T);                    // triggers dependents
    pub fn update(&self, f: impl FnOnce(&mut T));   // in-place update
    pub fn untracked(&self) -> T where T: Clone;    // read without subscribing
    pub fn setter(&self) -> SignalSetter<T>;        // Send handle
}

pub struct SignalSetter<T>;    // Send; queues updates
impl<T: 'static + Send> SignalSetter<T> {
    pub fn set(&self, value: T);
    pub fn update(&self, f: impl FnOnce(&mut T) + Send + 'static);
}

pub struct Memo<T>;
impl<T: 'static + Clone + PartialEq> Memo<T> {
    pub fn new(f: impl Fn() -> T + 'static) -> Self;
    pub fn get(&self) -> T;
    pub fn untracked(&self) -> T;
}

pub struct Effect;
impl Effect {
    pub fn new(f: impl FnMut() + 'static) -> Self;  // runs once immediately, then on dep changes
    // Effect is dropped when its Owner is disposed.
}

pub struct Owner;
impl Owner {
    pub fn run_in<R>(&self, f: impl FnOnce() -> R) -> R;
    pub fn dispose(self);
}

pub fn create_root<R>(f: impl FnOnce() -> R) -> (R, Owner);
pub fn on_cleanup(f: impl FnOnce() + 'static);  // must be called inside an Owner scope
pub fn batch<R>(f: impl FnOnce() -> R) -> R;

// Clock + Rng
pub trait Clock: 'static {
    fn now(&self) -> std::time::Instant;
}
pub struct SystemClock;
impl Clock for SystemClock { /* std::time::Instant::now */ }

pub trait Rng: 'static {
    fn gen_u64(&mut self) -> u64;
    fn gen_f32(&mut self) -> f32;
}
pub struct ThreadRng;  // wraps rand::thread_rng
```

### Memo equality-skip (Q1 decision)
- **Decision:** `Memo<T>: PartialEq` bound IS required. Equality-skip behavior is always-on. Rationale: the downstream Effect cost (scene mutation + dirty-flag set) is higher than PartialEq comparison cost for typical scene values (numbers, short strings, small structs); opt-in equality-skip would make the Python bindings surface harder to document.
- Bench harness (criterion) added in this phase to measure the PartialEq overhead on common value types (`f32`, `Vec3`, small struct). Fail CI if PartialEq check takes > 1¬µs on the benchmark types ‚Äî unlikely for these, but guards against someone using a big `PartialEq` impl.
- Documented in module-level doc + every Memo docstring.

### Send/Sync model (PITFALLS ¬ß6)
- Runtime is **single-threaded**. `Signal<T>`, `Memo<T>`, `Effect`, `Owner` are `!Send` + `!Sync` (phantom data enforces this).
- Cross-thread writes go through `SignalSetter<T>: Send`. Calling `.set(x)` queues the value into an `mpsc` drained on the owning thread at the start of each frame tick (frame scheduling lives in the backend crate ‚Äî Phase 1.1 ‚Äî but the drain hook is in core).
- Documented prominently in the crate's module-level doc. Every `Signal` / `Memo` / `Effect` docstring says "Single-threaded. Use `.setter()` for cross-thread writes."
- This is the project's "reactive threading tattoo."

### Cycle detection (PITFALLS ¬ß5)
- Runtime tracks the active observer stack during propagation. If an observer's dependency is itself being recomputed higher in the stack, we have a cycle.
- **Behavior:** runtime panics on cycles. The exact message is defined by `reactive_graph 0.2.13`'s `ImmediateEffect::new_mut_scoped` recursion check (which is panic-on-recursion). Adding a `panic_hook`-installed custom message with labelled owner chain (`"signal cycle detected: A -> B -> A"`) is deferred ‚Äî see [Research ¬ß"Integration test: cycle detection"]. The test suite asserts *a* panic occurs, not the exact message wording. Labelled panic messages may be added in a future phase via a `std::panic::catch_unwind` + label-stack wrapper in `runtime.rs`.
- Not a `Result` ‚Äî cycles are programmer errors.
- Test: a signal whose effect writes back to itself must panic within the first propagation tick (exact message is reactive_graph-defined).

**Decision date:** 2026-04-14 (Plan 01.0-01). Rationale: research found no way to customize reactive_graph's internal panic message without a panic-hook wrapper, which adds code for little user benefit in v0.0.0.

### Two-phase propagation (REACT-06)
- Phase 1 (mark dirty): traverse dependency graph, set dirty flags on every dependent Memo/Effect.
- Phase 2 (recompute in topo order): walk dirty nodes in topological order, recomputing Memo values and running Effects.
- `reactive_graph` handles this internally; our wrapper just ensures the semantics are preserved and documented. Our job is the **public surface**, not the internal graph.

### batch() semantics (REACT-05)
- `batch(|| { sig_a.set(1); sig_b.set(2); })` defers propagation until the closure returns. Dependents run once with the combined new state.
- `reactive_graph` provides this; wrap it as `happyterminals_core::batch`.

### Owner tree + on_cleanup (REACT-04)
- `create_root(|| body)` returns `(body_result, Owner)`. Dropping or `.dispose()`-ing the Owner cascades cleanup to every Effect, Memo, and `on_cleanup` registered inside.
- `on_cleanup(f)` registers a fn to run when the current owner is disposed. Panics (in debug) if called outside an owner scope.
- Test: 10k scene transitions (create_root ‚Üí lots of Memos + Effects ‚Üí dispose ‚Üí create_root again) keeps RSS flat within a documented ceiling (e.g., < 10MB growth over 10k cycles). Use `/proc/self/status` or `libc::getrusage` on Linux; skip test on non-Linux with `#[cfg(target_os = "linux")]`.

### Clock + Rng injection (REACT-09)
- Production path uses `SystemClock::default()` / `ThreadRng::default()`.
- Tests construct `ManualClock { now: Cell<Instant> }` and `SeededRng { state: u64 }` to make snapshot tests deterministic.
- Clock/Rng are passed as arguments to functions that need them ‚Äî NOT global state. This keeps the core crate free of global time/rng.
- Scene code (Phase 1.4+) will accept `&dyn Clock` / `&mut dyn Rng` as params where time/randomness is relevant (e.g., ease functions, particle lifetimes ‚Äî later phases).

### Error handling
- Public errors via `thiserror`-derived `CoreError { .. }` where applicable.
- `on_cleanup` outside an owner scope: **panic in debug** (`debug_assert!`), **no-op in release** with a `tracing::warn!`. Documented tradeoff.
- Cycle detection: always panic (cycles are bugs).

### Testing strategy (REACT-10)
- Unit tests inline in each module for isolated logic.
- Integration tests in `crates/happyterminals-core/tests/`:
  - `diamond.rs` ‚Äî A signal feeding two Memos that both feed one Effect runs the Effect exactly once per `A.set()`.
  - `disposal.rs` ‚Äî dropping an Owner disposes all its Effects (including via `on_cleanup`).
  - `transitions_10k.rs` ‚Äî 10k create_root‚Üídispose cycles, assert RSS stays under ceiling (Linux only).
  - `cycle.rs` ‚Äî self-cycling signal panics with the expected message.
  - `signal_setter.rs` ‚Äî spawn a thread that holds a `SignalSetter`, write to it, observe value updates after `drain()` on the main thread.
  - `untracked.rs` ‚Äî reading a signal via `.untracked()` inside an Effect does NOT create a subscription.
  - `batch.rs` ‚Äî `batch` coalesces multiple sets into a single propagation pass.
- Property tests (`proptest`):
  - `set()` then `get()` on a signal always returns the last-set value (trivial but sanity).
  - Any sequence of nested `batch`es is equivalent to one flat batch wrapping all the sets.
- Criterion bench:
  - `Memo<f32>` equality-skip micro-benchmark ‚Äî establish the < 1¬µs overhead baseline.
- Doc-tests on every public item where feasible.

### Clippy + unsafe policy
- `#![forbid(unsafe_code)]` at `lib.rs` top.
- `#![deny(clippy::unwrap_used, clippy::expect_used, clippy::panic)]` ‚Äî panics only via the explicit cycle-detection path (`unreachable!` / `std::panic::panic_any`) and `debug_assert!`.
- All `.workspace = true` lints from Phase 0 apply.

### Claude's Discretion
- Exact phantom-data type to enforce `!Send`/`!Sync` (`PhantomData<*const ()>` is canonical; use it).
- Exact label/debug-name plumbing for cycle detection messages ‚Äî implementer chooses whether to use `#[track_caller]` + `Location::caller()` or explicit `.label("name")` methods (recommend: both, with `.label()` overriding the caller-location default).
- Exact allocation strategy for observer graph ‚Äî `reactive_graph` handles it internally; we don't re-invent.
- Whether to expose a `create_rw_signal` split (Leptos style) ‚Äî recommend NOT for now; keep `Signal::new` unified.

## Specific Concerns

### Current state
- `crates/happyterminals-core/src/lib.rs` is an empty doc-comment scaffold (Phase 0 cleaned it up).
- `Cargo.toml` has zero `[dependencies]` ‚Äî Phase 0 stripped them.
- `[workspace.dependencies]` already pins `reactive_graph 0.2.13`, `any_spawner 0.3`, `thiserror 2.0`, `insta 1.47`, `proptest 1.11`, `criterion 0.8`.
- Need to ADD `rand = "0.9"` to `[workspace.dependencies]` during this phase (currently not pinned).
- `slotmap` may or may not be needed ‚Äî plan starts without, add only if `reactive_graph`'s surface forces us.

### Public API stability commitment
- Everything in this phase's public surface ships under `happyterminals-core 0.0.0` (pre-release). Semver discipline kicks in at `0.1.0` (Phase 3.5 publish). Until then, breaking changes are allowed ‚Äî but we aim to avoid them to keep Phase 1.1+ stable.

### Reactive API naming departures from Leptos
- We use `Signal::new()` / `.get()` / `.set()` / `.update()` / `.untracked()` / `.setter()` ‚Äî NOT Leptos's `create_signal` / `ReadSignal` + `WriteSignal` split. This is the main intentional departure.
- `Memo::new()` ‚Äî NOT `create_memo`.
- `Effect::new()` ‚Äî NOT `create_effect`.
- `create_root` / `Owner` ‚Äî we keep Leptos's names because `reactive_graph` exposes them directly.
- `batch` / `on_cleanup` ‚Äî same.

### Explicitly declined alternatives
- **Not building our own reactive graph from scratch.** Wrapping `reactive_graph` is the explicit STACK ¬ß1.4 recommendation. Rationale: correctness of two-phase propagation + cycle detection is tricky; re-proving it is a 3-month tangent.
- **Not exposing `reactive_graph` types in our public API.** Future migration (or replacement with our own impl) must be non-breaking for downstream crates. Wrapper types MUST hide the underlying graph.

</specifics>
