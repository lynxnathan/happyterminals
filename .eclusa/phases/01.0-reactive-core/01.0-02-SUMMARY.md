# Phase 1.0 Plan 02: Signal + SignalSetter + Memo + batch Summary

Implemented the data read/write surface of the reactive core — `Signal<T>`, `SignalSetter<T>`, `Memo<T>`, and `batch()` — by wrapping reactive_graph 0.2.13 with PhantomData-enforced `!Send + !Sync` markers, an mpsc keep-alive queue for cross-thread writes, and a SendWrapper-based closure shim in `runtime::wrap_local_fn`. Gate (`cargo check` / `cargo clippy -D warnings` / `cargo fmt --check`) is green.

## What landed

### signal.rs — `Signal<T>` + `SignalSetter<T>`

- `Signal<T>` wraps `RwSignal<T, LocalStorage>` (thread-pinned, `T: 'static` only — no `Send + Sync` leak).
- Fields: `inner` (the `RwSignal`, `Copy`), `rx` (`Rc<RefCell<mpsc::Receiver<PendingOp<T>>>>`, shared across clones), `_keepalive_tx` (owner-thread Sender, never read — just held for keep-alive), `tx_factory` (cloneable sender minted into `SignalSetter`s), `_not_send: PhantomData<*const ()>`.
- `Signal<T>: !Send + !Sync` enforced by `PhantomData<*const ()>`. Raw pointers are neither `Send` nor `Sync`.
- `Clone` is **manual** because `mpsc::Receiver` is `!Clone`; the `Rc<RefCell<_>>` lets every clone share one queue. The clone also clones `_keepalive_tx` so the keep-alive invariant survives (the allow-lint on `used_underscore_binding` is annotated in-source with the rationale).
- Methods: `new`, `get`, `set`, `update`, `untracked`, `setter`, `drain_setter_queue` — signatures match the plan's `<interfaces>` block exactly.
- `#[track_caller]` on `Signal::new` per CONTEXT §"Claude's Discretion".
- `SignalSetter<T>` is our own type (`T: Send + 'static`, `Send + !Sync`), mpsc-backed. Its `set` / `update` box closures into `PendingOp<T> = Box<dyn FnOnce(&RwSignal<T, LocalStorage>) + Send + 'static>` and drop `send` errors — a dropped Signal is a no-op by design (Pitfall 5 recovery).
- Zero `.unwrap()` / `.expect()` / `panic!()` in the file.

### memo.rs — `Memo<T>`

- `Memo<T>` wraps `reactive_graph::computed::Memo<T>` (the arena-stored, `SyncStorage`-default variant — the only `::new` constructor reactive_graph 0.2.13 exposes at the `Memo` (not `ArcMemo`) level).
- `!Send + !Sync` via `PhantomData<*const ()>`.
- `T: Clone + PartialEq + Send + Sync + 'static` (see Deviation §D1).
- User's compute closure routes through `runtime::wrap_local_fn` — users can still pass non-`Send` closures (capturing `Rc<Signal<_>>` etc.); the outer-`T` `Send + Sync` bound is the only leak.
- Methods: `new` (`#[track_caller]`), `get`, `untracked` — match `<interfaces>`.
- `Clone` is manual (consistent with `Signal`; `RgMemo<T>` is `Copy` so the clone is trivial).

### batch.rs — `batch()`

- Direct forward: `pub fn batch<R>(f: impl FnOnce() -> R) -> R { rg_batch(f) }` where `rg_batch = reactive_graph::effect::batch`.
- No `!Send` marker — it's a free function over a non-capturing closure.

### runtime.rs — `wrap_local_fn` helper

- Sibling of `wrap_local_fnmut`. Wraps `F: Fn() -> T + 'static` (non-`Send`) into `Fn(Option<&T>) -> T + Send + Sync + 'static` by stuffing `F` in a `SendWrapper`. The `Option<&T>` arg matches `reactive_graph::computed::Memo::new`'s `Fn(Option<&T>) -> T` signature (we ignore the prev-value hint — not exposed in our public `Memo::new`).
- Simpler than `wrap_local_fnmut`: no `RefCell` needed because `Fn` is shared-ref callable.
- Spike C (`__spike_memo_accepts_wrapped_fn`) constructs a `reactive_graph::computed::Memo<i32>` via `wrap_local_fn(|| 42i32)` — compiles, proving the bound flows.

## Deviations from Plan

### §D1. `Memo<T>` widens `T` to `Send + Sync + Clone + PartialEq + 'static`

**Found during:** Task 3 (Memo::new compile check — also confirmed upstream by inspecting reactive_graph 0.2.13 source).

**Issue:** CONTEXT.md §"API surface" specifies `impl<T: 'static + Clone + PartialEq> Memo<T>` — no `Send + Sync` on `T`. The plan's §decisions.2 anticipated this and documented the fallback.

**Why the fallback fires:** `reactive_graph::computed::Memo<T, S = SyncStorage>` is bounded `where S: Storage<T>`; `SyncStorage: Storage<T>` requires `T: Send + Sync + 'static`. The `wrap_local_fn` trick only relaxes the **closure**'s `Send + Sync` bound — it cannot relax the outer `T`'s bound because the `T` threads through the default `SyncStorage` at the struct level. `reactive_graph` only provides a non-`Send+Sync` path via the lower-level `ArcMemo<T, LocalStorage>` (through `FromLocal`), which would require a re-implementation of the `Memo::new` ergonomics and is out of scope for v0.0.0.

**Fix:** Widened `Memo<T>` to `T: Clone + PartialEq + Send + Sync + 'static`. Documented in the module doc under "Known deviation from CONTEXT.md §API surface".

**Impact:** Rules out `Memo<Rc<_>>` / `Memo<RefCell<_>>`. Users wanting shared ownership inside a memo should use `Memo<Arc<_>>` — same ergonomics, same `PartialEq` skip on pointer equality. No other Phase 1.0 surface is affected; `Signal<T>` and `Effect` (01.0-03) are unaffected by this bound.

**Files modified:** `crates/happyterminals-core/src/memo.rs` (bound on `impl` block + the struct itself + its `Clone` impl).

**Rule:** Rule 1 + Rule 2 (correctness / required to compile). The plan's §decisions.2 explicitly pre-authorized this path.

### §D2. `Signal<T>` `Clone` is a manual impl, not `#[derive(Clone)]`

**Found during:** Task 1 (first compile attempt used `#[derive(Clone)]`).

**Issue:** `#[derive(Clone)]` on `Signal<T>` requires every field to be `Clone`. `mpsc::Receiver<PendingOp<T>>` is **not** `Clone` (mpsc has a single-consumer model). Our use of `Rc<RefCell<Receiver<_>>>` makes the `rx` field cloneable, but derive can't see through the wrapper idiom when `T` stays generic — and more importantly, `RwSignal<T, LocalStorage>` is `Copy` (not `Clone`) and derive was unhappy threading the bound.

**Fix:** Wrote a manual `Clone` impl that (a) copies `inner`, (b) clones the `Rc` for `rx`, (c) clones `_keepalive_tx` (preserving the keep-alive invariant on the clone), (d) clones `tx_factory`, (e) re-instantiates `_not_send: PhantomData`.

**Impact:** Zero runtime behavior change vs derive; documented clippy allow (`used_underscore_binding`) on the `.clone()` of `_keepalive_tx` with a code-local rationale.

**Files modified:** `crates/happyterminals-core/src/signal.rs`.

**Rule:** Rule 3 (blocking: needed to compile). Plan's Task 1 deviation path §10 explicitly anticipated a manual `Clone`.

### §D3. `wrap_local_fn` uses `SendWrapper<F>` directly, no `RefCell<F>`

**Found during:** Task 2 (implementing the helper).

**Issue:** The plan's first sketch wrapped the `Fn` in `SendWrapper::new(RefCell::new(f))` then called `.borrow_mut()` — mirroring `wrap_local_fnmut`. But `Fn: Fn` is callable via `&F`, so `RefCell` and `borrow_mut` are redundant (and trigger "unused `mut`" / unnecessary-indirection smells under clippy).

**Fix:** Used the plan's fallback shape: `let cell = SendWrapper::new(f); move |_prev| (*cell)()`. `SendWrapper<F>: Deref<Target = F>` lets us call `(*cell)()` with just a shared borrow.

**Impact:** Simpler code; spike C (`__spike_memo_accepts_wrapped_fn`) compiles end-to-end with `reactive_graph::Memo::new` on the output.

**Files modified:** `crates/happyterminals-core/src/runtime.rs`.

**Rule:** Rule 1 (simplified a needlessly complex implementation). Plan's Task 2 explicitly provided this as the fallback path.

### §D4. `#[must_use]` added to read-only getters under `-D warnings`

**Found during:** Task 5 (final clippy gate).

**Issue:** Workspace lints include `clippy::pedantic`-ish checks; `clippy::must_use_candidate` triggered `-D warnings` failures on `Signal::get`, `Signal::untracked`, `Signal::setter`, `Memo::get`, `Memo::untracked`. Also `clippy::doc_markdown` flagged `reactive_graph` (unbacticked) in two doc comments.

**Fix:** Added `#[must_use]` to the five read-only getters. Backticked `reactive_graph` in module-doc and `SignalSetter` doc. Added an in-source `#[allow(clippy::used_underscore_binding, reason = "...")]` to the `Clone` impl for the `_keepalive_tx` clone (legitimate: the underscore documents "held, never polled", but we still must clone to preserve keep-alive on the clone).

**Impact:** No behavior change. `#[must_use]` is advisory; readers ignoring the return value now get a lint, which matches the intent (getters have no side effects worth calling for).

**Files modified:** `crates/happyterminals-core/src/signal.rs`, `crates/happyterminals-core/src/memo.rs`.

**Rule:** Rule 3 (blocking: `-D warnings` fails the gate otherwise). Under the plan's "no `.unwrap()` / `.expect()`" principle, satisfying the broader clippy gate falls in-scope.

## Plan §output questions — answered

- **Did `Memo<T>` need the `Send + Sync` bound fallback?** **Yes** (§D1). This is decision-relevant for future Python bindings: any `PyObject`-wrapped state inside a `Memo` must be `Send + Sync`, which for `PyO3` means bare `Py<T>` (which is `Send + Sync`) — the bound aligns with the grain of the bindings work rather than against it. Tracked as open question for potential future `LocalStorage` Memo variant if an `Rc`/`RefCell` use case arises.
- **Final `Signal<T>` `Clone` strategy:** manual impl (§D2). Reason: preserving the `_keepalive_tx` clone semantics plus the `Rc<RefCell<Receiver<_>>>` idiom made manual clearer than derive.
- **API-surface deviations from CONTEXT.md §"API surface":** exactly one — `Memo<T>`'s `Send + Sync` widening (§D1). `Signal<T>`, `SignalSetter<T>`, and `batch` all match CONTEXT.md verbatim.
- **Confirmed `!Send` / `!Sync` on `Signal<T>` and `Memo<T>`:** yes, via `PhantomData<*const ()>`. (`SignalSetter<T>` is deliberately `Send`, per the design — it's the cross-thread handle.) Compile-time `assert_not_impl_any!` confirmation is planned for 01.0-05's test suite.
