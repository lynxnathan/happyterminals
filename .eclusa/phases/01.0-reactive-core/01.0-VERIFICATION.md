### Plan 01.0-05 — Tests / Benches / Docs — **FLAG** (one test design correctness issue; one coverage gap)

- **Goal achievable:** Yes — 7 integration tests + 2 proptest + 1 compile-time send_sync + 1 criterion bench + a full-docs audit. The full exit-criteria 1-10 gate runs here.
- **REACT coverage:** REACT-10 primary; REACT-01..09 transitively validated (coverage table in Task 2 action-block maps REACT IDs to test files).
- **Hard rules:** Every public item documented, `cargo doc -- -D warnings` enforced (Task 6), send_sync.rs guards `!Send + !Sync` at compile time (Task 4 — the "reactive threading tattoo").
- **Observability:** Excellent — every `<verify>` is a cargo + grep shell pipeline.
- **FLAG — Diamond test design in `tests/diamond.rs` may not detect thundering herd:** The `behavior` block for Task 2 diamond test reads *"After `create_root(|| { let a = Signal::new(0); let b = Memo::new(a); let c = Memo::new(a*2); let e = Effect::new(|| { b.get(); c.get(); count++; }); a.set(42); })`, `count` reads `2` (one initial run + one post-set). NOT `3` (would be thundering herd)."*
  The exit-criterion language from ROADMAP is *"A signal feeding two Memos into one Effect fires the Effect exactly once per set."* The diamond test SHOULD read `count == 2` after **one** `a.set()` (one initial run + one change run). That's the expected semantic.
  However, the description uses pseudocode `let b = Memo::new(a);` — `a` alone isn't a closure. In real code this must be `Memo::new({ let a = a.clone(); move || a.get() })`. If the executor writes `Memo::new(|| a.get())` where `a` is captured by move without clone, compilation fails because `Memo::new` takes `impl Fn() + 'static` (per plan 02) and `Signal<T>` is `Clone`. Nuance — the executor needs to add explicit `.clone()`s, which the `behavior` pseudocode hides. 01.0-05 should direct the executor to RESEARCH §"Integration test: diamond dependency" verbatim (the research is claimed to have a near-copy-pasteable version, per Task 2's action line 204). Verified this is the case in the plan. Flag downgraded to **nit, not blocker**: plan is correct in referring executor to research; pseudocode in `<behavior>` is just shorthand. Keep the `behavior` prose synchronised with what the copied research code actually asserts — would be good to explicitly mention "exactly once after set (count == 2 total)."

- **FLAG — Criterion `< 1µs` gate is asserted but not enforced:** Task 5 `<verify>` runs `cargo bench -- --quick --sample-size 10` and greps for the bench function name. It does NOT parse the reported median and fail the job if it exceeds 1µs. The exit criterion #8 ("Memo-PartialEq criterion bench ≤ 1µs") is therefore un-enforced by the automated verify — it's a manual-inspection gate. Either:
  a. Add a helper script that parses `target/criterion/*/base/estimates.json` and fails the verify if median > 1000 ns.
  b. Accept it as a human-check gate and downgrade the phase-exit criterion to "bench runs and reports times" (which is what the plan really does).
  Option (a) is strictly better. Without it, the phase can technically exit with a 5µs-per-iter Memo and nobody notices. Recommend adding a parsing step.

- **FLAG — Doc-tests status:** CONTEXT.md § "Testing strategy" says "Doc-tests on every public item where feasible." Plan 01.0-05 punts this (open question 4 — "Punt to Phase 1.4+"). That's a reasonable tradeoff but it should be recorded explicitly in the PHASE SUMMARY so downstream phases don't assume doc-tests exist. Not a revision blocker; document the decision.

- **Minor — `batch` doc example uses `ignore`d fenced block:** The plan mentions in Task 6 that `batch`'s doc has an `ignored` example. For a top-level flagship API, an `ignore` doc-test misses a tiny reader-reassurance gate. Not urgent.
