## Decisions this plan executes

1. **`Signal<T>` uses `RwSignal::new_local`** (not `RwSignal::new`). Matches our `!Send` policy; `new_local` stores in a thread-local arena without bounding `T: Send + Sync`.

2. **`Memo<T>` API bound deviation from CONTEXT.md** ‚Äî documented in RESEARCH ¬ß"Complete `Memo<T>` wrapper" note. CONTEXT.md says `T: Clone + PartialEq + 'static`; `reactive_graph::Memo::new` requires `T: Send + Sync + Clone + PartialEq + 'static` and a `Send + Sync` closure. **Decision: wrap the closure in `runtime::wrap_local_fn`** (same SendWrapper trick as `Effect`; add a second helper in `runtime.rs` for `Fn` instead of `FnMut`). **This keeps the `T: PartialEq + Clone + 'static` bound ONLY** (no `Send+Sync` leak). If the wrap trick fails for `Memo` (the `T: Send + Sync` bound on `reactive_graph::Memo<T>` itself is inescapable without a custom storage flag), **relax our `Memo<T>` bound to `T: Send + Sync + Clone + PartialEq + 'static`** and document the deviation in the module doc. This fallback is a known, acceptable narrowing of the API surface for v0.0.0 ‚Äî most values users put in Memos (numbers, small structs, Arcs) are already `Send + Sync`.

3. **`SignalSetter<T>` is OUR type**, implemented via `std::sync::mpsc`. We do NOT re-export `reactive_graph::wrappers::write::SignalSetter` ‚Äî the name collision warning is already documented in `lib.rs`'s module doc (per 01.0-01).

4. **`Signal<T>: Clone`** ‚Äî users will clone signals into multiple closures (see RESEARCH test examples: `let s_clone = s.clone();`). `RwSignal<T, LocalStorage>` is `Copy` in reactive_graph, so `Signal<T>` can be trivially `Clone`. Implement `#[derive(Clone)]` (or manual impl if derive stumbles on the mpsc fields ‚Äî then use `Rc<...>` wrapping for the non-Clone mpsc receiver).

5. **`Memo<T>: Clone`** ‚Äî same rationale. `reactive_graph::Memo<T>` is `Copy`; our wrapper is `Clone`.

6. **`untracked` free function is NOT exposed.** CONTEXT.md ¬ß"API surface" lists only `.untracked()` methods on `Signal` / `Memo`. A free `untrack(f)` function would duplicate; keep it private to the crate (internal use in 01.0-03's `batch`-like helpers only if needed).

7. **Crate-root `pub use` re-exports are OUT OF SCOPE for this plan.** A separate Wave-3 plan (`01.0-02b-lib-reexports`) owns `crates/happyterminals-core/src/lib.rs` and collates all re-exports from 01.0-02, 01.0-03, 01.0-04. This plan's types are reachable in Wave 2 via the `pub mod signal;` / `pub mod memo;` / `pub mod batch;` declarations that 01.0-01 already landed ‚Äî e.g., `happyterminals_core::signal::Signal`. 01.0-02b promotes them to the crate root (`happyterminals_core::Signal`). Do NOT append `pub use` lines to `lib.rs` from this plan.
