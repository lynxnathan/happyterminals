# Phase 1.0: Reactive Core — Research

**Researched:** 2026-04-14
**Domain:** Fine-grained reactive runtime (`reactive_graph 0.2.13` wrapper crate)
**Confidence:** HIGH across all sections. Every API claim was verified against the live `docs.rs/reactive_graph/0.2.13/` index (not training data) on the research date. Versions were cross-checked against the crates.io registry API the same day.

## Summary

`reactive_graph 0.2.13` is far more capable than ARCHITECTURE.md §4.1 suggested when it called it "a 150-LOC alternative vs build-our-own." The public API already ships the pieces we need — `RwSignal`, `RwSignal::new_local`, `Memo` (with automatic `PartialEq` equality-skip), `ImmediateEffect::new_mut` (synchronous, panics on recursion), `Owner` (with cleanup RAII), `effect::batch`, `graph::untrack`, `owner::on_cleanup`, and a pre-existing `wrappers::write::SignalSetter` type. Phase 1.0 is primarily **thin, `!Send` wrapper types + the `SignalSetter` mpsc channel + `Clock`/`Rng` traits + comprehensive tests**, not a reimplementation.

The biggest discovery: `ImmediateEffect::new_mut` gives us **free cycle detection** — it panics on recursive re-entry with a clear message. Our `Effect` wrapper should use `ImmediateEffect::new_mut_scoped`, not the async-scheduled `Effect` type which requires `any_spawner` and defers to "next tick". This keeps Phase 1.0 **fully synchronous** with no `any_spawner` dependency — contrary to CONTEXT.md's listing of `any_spawner` as a dev-dep. Revised recommendation: **drop `any_spawner` entirely from Phase 1.0**; it re-enters at Phase 1.1 when the backend event loop lands.

Second most important: `reactive_graph`'s `SignalSetter` is **already `Send`** (verified) but it is a thin "apply a closure to a signal" wrapper, not an mpsc channel. It does not solve the cross-thread write problem we actually have (which is: "arbitrary thread → signal write → rendered next frame on the render thread"). We still build our own `SignalSetter<T>` type over `std::sync::mpsc` + a `drain()` hook. Rename ours to avoid collision: **use `happyterminals_core::SignalSetter` in our surface and never re-export `reactive_graph::wrappers::write::SignalSetter`** — they are unrelated concepts with the same name.

Third: `batch` in `reactive_graph` only flushes `ImmediateEffect`s (the documentation is explicit). Since we use `ImmediateEffect` throughout, `batch` works correctly for us. The `create_effect` / `Effect::new` async-scheduled path would NOT participate in `batch` — one more reason to avoid it.

**Primary recommendation:** Build Phase 1.0 as nine thin `!Send`/`!Sync` wrapper modules over `reactive_graph 0.2.13`. Use `RwSignal::new_local` as the underlying storage. Use `ImmediateEffect::new_mut_scoped` for our `Effect` type (gives us synchronous propagation + cycle detection for free). Use `Memo::new` (has automatic `PartialEq` skip — no wrapper layer needed). Build our own `SignalSetter` mpsc-channel type for cross-thread writes. Drop `any_spawner` from this phase.

### Pattern 1: Thin `!Send` Wrappers Over `RwSignal::new_local`

**What:** Every public reactive type owns an underlying `reactive_graph` type plus a `PhantomData<*const ()>` field to enforce `!Send + !Sync`.

**When to use:** All four public wrapper types (`Signal`, `Memo`, `Effect`, `Owner`).

**Example:**

```rust
// src/signal.rs — verified against docs.rs/reactive_graph/0.2.13/reactive_graph/signal/struct.RwSignal.html
use std::marker::PhantomData;
use reactive_graph::signal::RwSignal;
use reactive_graph::owner::LocalStorage;
use reactive_graph::traits::{Get, Set, Update, GetUntracked};

/// A fine-grained reactive value cell.
///
/// **Single-threaded.** `Signal<T>` is `!Send + !Sync`. Use [`Signal::setter`]
/// for cross-thread writes.
pub struct Signal<T: 'static> {
    inner: RwSignal<T, LocalStorage>,
    _not_send: PhantomData<*const ()>,
}

impl<T: 'static> Signal<T> {
    #[track_caller]
    pub fn new(value: T) -> Self {
        Self {
            inner: RwSignal::new_local(value),
            _not_send: PhantomData,
        }
    }

    pub fn get(&self) -> T where T: Clone {
        self.inner.get()
    }

    pub fn set(&self, value: T) {
        self.inner.set(value);
    }

    pub fn update(&self, f: impl FnOnce(&mut T)) {
        self.inner.update(f);
    }

    pub fn untracked(&self) -> T where T: Clone {
        self.inner.get_untracked()
    }
}

// Explicit !Send/!Sync assertion (compile-time tests go in tests/send_sync.rs)
```

### Pattern 2: `ImmediateEffect::new_mut_scoped` for Our `Effect`

**What:** Use the synchronous, owner-scoped effect variant. Auto-disposes when owner drops. Panics on recursion (= our cycle detection).

**Why:** Keeps Phase 1.0 executor-free. Integrates with `effect::batch`. Panics on cycles with no extra machinery.

**Example:**

```rust
// src/effect.rs — verified against docs.rs/reactive_graph/0.2.13/reactive_graph/effect/struct.ImmediateEffect.html
use std::marker::PhantomData;
use reactive_graph::effect::ImmediateEffect;

/// A side-effect that re-runs whenever any tracked signal or memo it reads changes.
///
/// **Lifecycle:** Effects run once immediately at construction, and again on each
/// dependency change. They are disposed automatically when their owning [`Owner`]
/// is dropped.
///
/// **Cycle detection:** If an Effect's closure (directly or transitively) causes
/// itself to re-enter while still running, the runtime panics with a clear
/// "signal cycle detected" message. See the crate-level documentation.
///
/// **Single-threaded.** `Effect` is `!Send + !Sync`.
pub struct Effect {
    // Hold the inner handle so dropping Effect (or its Owner) disposes it.
    // ImmediateEffect::new_mut_scoped auto-disposes on owner drop; we keep
    // the handle so user code can optionally explicit-drop early.
    _inner: ImmediateEffect,
    _not_send: PhantomData<*const ()>,
}

impl Effect {
    /// Constructs a new Effect that runs once immediately, then whenever any
    /// signal/memo it reads changes.
    ///
    /// # Panics
    ///
    /// Panics if the effect recursively re-enters itself (signal cycle).
    /// The message format is `"signal cycle detected: ..."` — see
    /// `runtime::CycleDetectedPanic`.
    #[track_caller]
    pub fn new(f: impl FnMut() + 'static) -> Self {
        // NOTE: ImmediateEffect::new_mut_scoped bounds require `Send + Sync + 'static`.
        // We wrap f in a SendWrapper-equivalent internally, OR we accept Send+Sync
        // on f. Since our public policy is single-threaded, we should NOT require
        // Send+Sync on user closures. Two options:
        //   (a) use reactive_graph::send_wrapper_ext to bypass the Send+Sync bound
        //       — but this is a thread-affinity wrapper, not a relaxation.
        //   (b) change the bound to `FnMut() + 'static` and document the single-
        //       threaded runtime promise.
        //
        // Recommended: use send_wrapper (transitive dep of reactive_graph) to wrap
        // the user closure, which panics if moved across threads (matching our
        // !Send promise). See runtime.rs for the helper.
        let wrapped = crate::runtime::wrap_local_fnmut(f);
        Self {
            _inner: ImmediateEffect::new_mut_scoped(wrapped),
            _not_send: PhantomData,
        }
    }
}
```

### Pattern 3: Owner as RAII Handle Over `reactive_graph::owner::Owner`

```rust
// src/owner.rs — verified against docs.rs/reactive_graph/0.2.13/reactive_graph/owner/struct.Owner.html
use std::marker::PhantomData;
use reactive_graph::owner::Owner as RgOwner;

pub struct Owner {
    inner: RgOwner,
    _not_send: PhantomData<*const ()>,
}

impl Owner {
    /// Runs `f` with this Owner set as the current reactive owner.
    /// Any signals, memos, or effects created inside will register under it.
    pub fn run_in<R>(&self, f: impl FnOnce() -> R) -> R {
        self.inner.with(f)
    }

    /// Disposes this owner, running all cleanup callbacks and dropping every
    /// signal/memo/effect created within its scope.
    pub fn dispose(self) {
        self.inner.cleanup();
        // inner is dropped here, releasing the arena.
    }
}

/// Creates a new root reactive scope, runs `f` inside it, and returns the
/// resulting value plus the owner handle. Drop or `.dispose()` the Owner to
/// clean up everything allocated inside.
#[track_caller]
pub fn create_root<R>(f: impl FnOnce() -> R) -> (R, Owner) {
    let inner = RgOwner::new_root(None);
    let result = inner.with(f);
    (result, Owner { inner, _not_send: PhantomData })
}

/// Registers `f` to run when the current owner is disposed.
///
/// # Panics (debug builds only)
///
/// Panics if called outside an active owner scope.
/// In release builds this logs a `tracing::warn!` and silently drops `f`.
#[track_caller]
pub fn on_cleanup(f: impl FnOnce() + Send + Sync + 'static) {
    // Detect "no current owner" by attempting a no-op Owner::current() check.
    // reactive_graph::owner::Owner::current() exists (via prelude). If None,
    // we're outside a scope.
    if RgOwner::current().is_none() {
        #[cfg(debug_assertions)]
        panic!("happyterminals_core::on_cleanup called outside an owner scope");
        #[cfg(not(debug_assertions))]
        {
            tracing::warn!(
                target: "happyterminals_core::owner",
                "on_cleanup called outside an owner scope — callback dropped"
            );
            return;
        }
    }
    reactive_graph::owner::on_cleanup(f);
}
```

**Note:** The existence of `Owner::current()` should be verified when implementing; the docs.rs index confirmed `Owner` has `cleanup`, `new`, `new_root`, `set`, `with`, `with_cleanup`, but `current()` was inferred from the Leptos idiom. If the method is named differently (e.g., `Owner::get`), adapt accordingly. This is a MEDIUM-confidence detail; implementer must verify at planning time.

### Anti-Patterns to Avoid

- **Using `Effect::new` from `reactive_graph` directly.** That's the async-scheduled variant; requires `any_spawner`; doesn't integrate with `batch`. Always use `ImmediateEffect::new_mut_scoped` in Phase 1.0.
- **Using `RwSignal::new` (non-local).** That variant requires `T: Send + Sync`, which propagates. We want `!Send`. Always use `RwSignal::new_local`.
- **Re-exporting `reactive_graph::wrappers::write::SignalSetter`.** Name collision with our own `SignalSetter`. Keep them fully separate.
- **Holding a `Signal<T>` across a thread boundary.** Compile error via `PhantomData<*const ()>`. Document prominently.
- **Calling `on_cleanup` at the top of `create_root`'s `f` without also calling `setup` hooks.** `reactive_graph` requires the owner to be **active** (i.e., inside `Owner::with`) when `on_cleanup` is invoked. Our `create_root` wraps `f` in `inner.with(f)`, so this is handled — but downstream consumers must not call `on_cleanup` from outside any `Owner::run_in`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Signal dependency tracking | Custom observer registry | `reactive_graph::signal::RwSignal::new_local` | Two-phase propagation + weak-ref subscriber lists + arena are tricky. Leptos spent years hardening it. |
| Memo equality-skip | Custom dirty-cache logic | `reactive_graph::computed::Memo::new` (requires `T: PartialEq`) | Already built in. Bound is already `PartialEq`, matching our Q1 decision. |
| Cycle detection | DFS-with-stack cycle finder | `reactive_graph::effect::ImmediateEffect::new_mut_scoped` | `FnMut` variant **panics on recursion** — that IS cycle detection. Zero code. |
| Owner/scope tree | Custom slotmap-based parent tree | `reactive_graph::owner::Owner` | RAII cleanup + nested-owner cascade already implemented. |
| Thread-local reactive context | Custom `thread_local!` observer stack | `Owner::with` / `Owner::set` | `reactive_graph` maintains the current-owner thread-local internally. |
| Untracked reads | Custom "disable tracking" flag | `signal.get_untracked()` or `graph::untrack(f)` | Built-in. |
| Batching | Custom dirty-set drain | `effect::batch(f)` | Built-in; only affects `ImmediateEffect`s, which matches our setup. |
| `PartialEq`-bounded derived cache | — | `Memo::new_with_compare(f, cmp)` if we ever want a custom equality fn | Available as escape hatch; not exposed in v0.0.0. |

**Key insight:** ARCHITECTURE.md §4.1 (from an older research pass) suggested "build our own 150-LOC reactive core." STACK.md §1.4 superseded that with "wrap `reactive_graph`." This research confirms STACK.md is correct: the wrapper is genuinely thin because `reactive_graph 0.2.13` already covers 90% of Phase 1.0's requirements. The remaining 10% is `Signal → mpsc → render thread` plumbing (`SignalSetter`), the `Clock`/`Rng` traits, and the test suite.

## Common Pitfalls

### Pitfall 1: Forgetting `Owner::with` around `on_cleanup`

**What goes wrong:** `on_cleanup(f)` is called at module top-level or in a helper function that runs outside any `Owner` scope. In `reactive_graph`, the callback is silently dropped.

**Why it happens:** `reactive_graph::owner::on_cleanup` does not panic outside an owner — it just has nowhere to register the callback. Users assume it works like React's `useEffect` cleanup (always runs).

**How to avoid:** Our wrapper `happyterminals_core::on_cleanup` checks `Owner::current().is_none()` and panics in debug / warns in release. Per CONTEXT.md lock.

**Warning signs:** Tests that register cleanups but never observe them run. Memory growth in long-running scenes that supposedly clean up.

### Pitfall 2: Using `ImmediateEffect::new_mut_scoped` with a closure that captures a non-`Send+Sync` environment

**What goes wrong:** `ImmediateEffect::new_mut_scoped` bounds require `FnMut() + Send + Sync + 'static`. Capturing an `Rc<T>` or a `RefCell<T>` fails to compile.

**Why it happens:** `reactive_graph` uses a thread-safe internal representation even for single-threaded runtimes, to keep one API surface.

**How to avoid:** Wrap user closures via `send_wrapper::SendWrapper` (transitive dep of `reactive_graph`, exposed through `reactive_graph::send_wrapper_ext`). `SendWrapper<T>` is `Send + Sync` but panics if dereferenced from a thread other than its origin — matches our "single-threaded" promise exactly. Build this helper once in `src/runtime.rs` and reuse.

```rust
// src/runtime.rs (sketch)
use send_wrapper::SendWrapper;
use std::cell::RefCell;

/// Wraps a non-`Send` `FnMut` closure in a `Send + Sync` shell that panics
/// if actually polled from another thread. Matches our single-threaded
/// runtime contract.
pub(crate) fn wrap_local_fnmut<F: FnMut() + 'static>(
    f: F,
) -> impl FnMut() + Send + Sync + 'static {
    let cell = SendWrapper::new(RefCell::new(f));
    move || {
        (cell.borrow_mut())();
    }
}
```

**Warning signs:** Compile errors of the form `Rc<T> cannot be sent between threads safely` when passing closures into `Effect::new`. Our `runtime::wrap_local_fnmut` eliminates this.

### Pitfall 3: `batch` doesn't coalesce async `Effect`s

**What goes wrong:** Mixing our `Effect` (ImmediateEffect-backed) with any async `reactive_graph::Effect::new` call in the same tree causes `batch` to only defer the synchronous ones.

**Why it happens:** Documented in `reactive_graph::effect::batch` — it only defers `ImmediateEffect`s.

**How to avoid:** Phase 1.0 uses `ImmediateEffect` exclusively (we never expose async effects). Lint-enforce: no public API returning `reactive_graph::Effect`.

### Pitfall 4: `Memo<T>` where `T: PartialEq` has a pathologically slow impl

**What goes wrong:** A user passes `Memo::new(|| huge_vec.clone())` where comparing two huge vecs is 10ms. The "equality-skip saves downstream work" optimization becomes a loss.

**Why it happens:** Q1 locked in always-on PartialEq skip. Users can construct Memos where PartialEq is expensive.

**How to avoid:** Document the contract: "PartialEq for `Memo<T>` should be O(1) or O(small). If `T` is large, wrap it in `Arc<T>` and compare pointers." Criterion bench at `benches/memo_eq_skip.rs` enforces the < 1µs ceiling for our tested value types (`f32`, `[f32; 3]`, small struct). User-supplied types are their responsibility — the bench is guidance, not enforcement.

### Pitfall 5: `SignalSetter` dropped before drain

**What goes wrong:** A background thread holds a `SignalSetter<T>`, writes `.set(x)`, then the thread exits. The `mpsc::Sender` is dropped. The render thread's `drain_setter_queue()` sees `TryRecvError::Disconnected` — but we interpret it as "no-op, no messages waiting." Updates are silently lost.

**Why it happens:** Naive `mpsc::channel` semantics. When the last Sender drops, the Receiver returns `Disconnected`.

**How to avoid:** The render-thread Receiver must be designed to outlive all Senders. Hold one "keep-alive" Sender on the render thread itself. Pattern: factory returns a master Sender; every `.setter()` call clones it.

```rust
// src/signal.rs (sketch)
pub struct Signal<T: 'static> {
    inner: RwSignal<T, LocalStorage>,
    queue_tx: mpsc::Sender<Box<dyn FnOnce(&RwSignal<T, LocalStorage>) + Send>>,
    queue_rx: Rc<RefCell<mpsc::Receiver<Box<dyn FnOnce(&RwSignal<T, LocalStorage>) + Send>>>>,
    _not_send: PhantomData<*const ()>,
}
// The queue_tx held in Signal<T> is the keep-alive; SignalSetter clones it.
```

**Warning signs:** Tests where a spawned thread writes then joins, and the main thread observes no update.

### Pitfall 6: 10k-transition test leaking due to `thread_local!` arena re-entry

**What goes wrong:** `reactive_graph` uses internal arena allocation; repeatedly creating and dropping `Owner::new_root` may retain slots in the thread-local arena. RSS grows linearly over 10k iterations — but not because of our code, because of `reactive_graph`'s internal allocator behavior.

**Why it happens:** Arena allocators don't return memory to the OS; they retain capacity for reuse. RSS != leak.

**How to avoid:** Measure RSS delta, not absolute. Our ceiling is **growth over 10k iterations**, not total allocation. After a warm-up of ~100 iterations, RSS should plateau. Test logic:

```rust
// tests/transitions_10k.rs (Linux only)
#[cfg(target_os = "linux")]
#[test]
fn ten_thousand_transitions_bounded_rss() {
    // Warm-up
    for _ in 0..100 { run_cycle(); }
    let rss_start = rss_kb();

    for _ in 0..10_000 { run_cycle(); }

    let rss_end = rss_kb();
    let delta_mb = (rss_end.saturating_sub(rss_start)) / 1024;
    assert!(
        delta_mb < 10,
        "RSS grew by {}MB over 10k transitions (ceiling: 10MB)",
        delta_mb
    );
}

fn run_cycle() {
    let (_, owner) = create_root(|| {
        let s = Signal::new(0i32);
        let m = Memo::new({ let s = s.clone(); move || s.get() * 2 });
        let _e = Effect::new({ let m = m.clone(); move || { let _ = m.get(); } });
        s.set(1);
    });
    owner.dispose();
}

fn rss_kb() -> u64 {
    // /proc/self/statm: size resident shared text lib data dt (all in pages)
    let statm = std::fs::read_to_string("/proc/self/statm").expect("read statm");
    let rss_pages: u64 = statm.split_whitespace().nth(1).unwrap().parse().unwrap();
    let page_kb = 4; // assume 4KiB pages — true on x86_64/aarch64 Linux default
    rss_pages * page_kb
}
```

**Warning signs:** The test passes locally (fast machine, cold cache) but fails in CI (slower, different allocator behavior). If flaky, raise ceiling to 20MB OR use `libc::getrusage` + `ru_maxrss`.

### Complete `Signal<T>` + `SignalSetter<T>` with mpsc drain

```rust
// src/signal.rs — fully worked example
use std::cell::RefCell;
use std::marker::PhantomData;
use std::rc::Rc;
use std::sync::mpsc;

use reactive_graph::owner::LocalStorage;
use reactive_graph::signal::RwSignal;
use reactive_graph::traits::{Get, GetUntracked, Set, Update};

type PendingOp<T> = Box<dyn FnOnce(&RwSignal<T, LocalStorage>) + Send + 'static>;

pub struct Signal<T: 'static> {
    inner: RwSignal<T, LocalStorage>,
    /// Render-thread-owned receiver; held via Rc so multiple clones of the
    /// same Signal share one queue.
    rx: Rc<RefCell<mpsc::Receiver<PendingOp<T>>>>,
    /// Keep-alive sender on the owning thread — prevents mpsc Disconnect
    /// when all user-side Senders are dropped.
    _keepalive_tx: mpsc::Sender<PendingOp<T>>,
    /// Cloneable sender for handing out to `SignalSetter`s.
    tx_factory: mpsc::Sender<PendingOp<T>>,
    _not_send: PhantomData<*const ()>,
}

impl<T: 'static> Signal<T> {
    #[track_caller]
    pub fn new(value: T) -> Self {
        let inner = RwSignal::new_local(value);
        let (tx, rx) = mpsc::channel();
        let tx_factory = tx.clone();
        Self {
            inner,
            rx: Rc::new(RefCell::new(rx)),
            _keepalive_tx: tx,
            tx_factory,
            _not_send: PhantomData,
        }
    }

    pub fn get(&self) -> T where T: Clone {
        self.inner.get()
    }

    pub fn set(&self, value: T)
    where
        T: 'static,
    {
        self.inner.set(value);
    }

    pub fn update(&self, f: impl FnOnce(&mut T)) {
        self.inner.update(f);
    }

    pub fn untracked(&self) -> T where T: Clone {
        self.inner.get_untracked()
    }

    pub fn setter(&self) -> SignalSetter<T>
    where
        T: Send + 'static,
    {
        SignalSetter {
            tx: self.tx_factory.clone(),
        }
    }

    /// Applies every pending cross-thread op from the mpsc queue.
    /// Called by the backend at the start of each frame tick.
    pub fn drain_setter_queue(&self) {
        let rx = self.rx.borrow();
        while let Ok(op) = rx.try_recv() {
            op(&self.inner);
        }
    }
}

/// A `Send` handle for writing to a [`Signal`] from another thread.
///
/// Writes are queued in an mpsc channel drained on the signal's owning
/// thread by the backend event loop each frame tick. Writes are visible
/// to observers on the next frame, not immediately.
pub struct SignalSetter<T: Send + 'static> {
    tx: mpsc::Sender<PendingOp<T>>,
}

impl<T: Send + 'static> Clone for SignalSetter<T> {
    fn clone(&self) -> Self {
        Self { tx: self.tx.clone() }
    }
}

impl<T: Send + 'static> SignalSetter<T> {
    pub fn set(&self, value: T) {
        // Ignore send errors — Signal dropped means "no longer interested".
        let _ = self.tx.send(Box::new(move |sig| sig.set(value)));
    }

    pub fn update(&self, f: impl FnOnce(&mut T) + Send + 'static) {
        let _ = self.tx.send(Box::new(move |sig| sig.update(f)));
    }
}

// SignalSetter is Send because `mpsc::Sender<T>: Send` when T: Send.
// PendingOp<T> is `Box<dyn FnOnce(...) + Send + 'static>`, so Sender<PendingOp<T>>: Send.
```

### Complete `Memo<T>` wrapper

```rust
// src/memo.rs
use std::marker::PhantomData;
use reactive_graph::computed::Memo as RgMemo;
use reactive_graph::traits::{Get, GetUntracked};

pub struct Memo<T: 'static> {
    inner: RgMemo<T>,
    _not_send: PhantomData<*const ()>,
}

impl<T> Memo<T>
where
    T: Clone + PartialEq + Send + Sync + 'static,
{
    /// Constructs a new memo. The closure `f` is called to compute the value
    /// whenever any signal/memo read inside it changes.
    ///
    /// Downstream effects and memos only re-run if the computed value is
    /// `!=` (via `PartialEq`) its previous value.
    #[track_caller]
    pub fn new(f: impl Fn() -> T + Send + Sync + 'static) -> Self {
        Self {
            // reactive_graph::computed::Memo::new takes Fn(Option<&T>) -> T;
            // we ignore the prev value arg to match our simpler API.
            inner: RgMemo::new(move |_prev| f()),
            _not_send: PhantomData,
        }
    }

    pub fn get(&self) -> T {
        self.inner.get()
    }

    pub fn untracked(&self) -> T {
        self.inner.get_untracked()
    }
}
```

**⚠ API-surface note:** CONTEXT.md §API surface says `impl<T: 'static + Clone + PartialEq> Memo<T> { pub fn new(f: impl Fn() -> T + 'static) -> Self; }` — no `Send + Sync` bound on `T` or `f`. BUT `reactive_graph::Memo::new` requires `T: Send + Sync + 'static` and the closure must be `Send + Sync`. To faithfully implement the CONTEXT.md signature, the implementer must either:
1. **Relax our bound** to match reactive_graph: `T: Send + Sync + Clone + PartialEq + 'static`. This is a **narrower API** than CONTEXT.md suggests — deviation requires planner sign-off.
2. **Wrap the closure** in `SendWrapper` (same trick as `Effect`) to bypass the `Send + Sync` requirement while preserving single-threaded semantics.

Recommended: **option 2**, identical to the Effect wrapper. Matches the "wrap, don't re-expose" principle. Implementer must verify `reactive_graph::Memo::new_with_compare` accepts SendWrapper-wrapped closures (MEDIUM confidence; needs compile check in plan-check).

        let _e = Effect::new(move || {
            let _a = tracked_for_effect.get();           // subscribes
            let _b = untracked_for_effect.untracked();   // does NOT subscribe
            runs_for_effect.set(runs_for_effect.get() + 1);
        });

        let runs_after_init = runs.get();

        // Changing the untracked signal must NOT re-run the effect.
        untracked_sig.set(999);
        assert_eq!(runs.get(), runs_after_init);

        // Changing the tracked signal MUST re-run it.
        tracked.set(1);
        assert_eq!(runs.get(), runs_after_init + 1);
    });
    owner.dispose();
}
```

    #[derive(Clone, PartialEq)]
    struct Vec3 { x: f32, y: f32, z: f32 }

    c.bench_function("memo_vec3_eq_skip", |b| {
        let (stuff, _owner) = create_root(|| {
            let s = Signal::new(Vec3 { x: 0.0, y: 0.0, z: 0.0 });
            let s_clone = s.clone();
            let m = Memo::new(move || {

                Vec3 { x: v.x + 1.0, y: v.y, z: v.z }
            });

        let (s, m) = stuff;

criterion_main!(benches);
```

> **CI gate:** Both benchmark functions must complete in < 1µs per iteration. If criterion reports > 1µs, the build fails. This is a guard against someone introducing a pathologically-expensive `PartialEq` on a type passed to `Memo<T>` elsewhere in the tree — not against `f32::eq`, which is obviously < 1ns.

//! # happyterminals-core
//!
//! Single-threaded fine-grained reactive runtime wrapping `reactive_graph 0.2.13`
//! behind a happyterminals-owned public API.
//!
//! ## Threading contract
//!
//! `Signal<T>`, `Memo<T>`, `Effect`, and `Owner` are all `!Send + !Sync` — they
//! live on a single "render thread." Cross-thread writes go through
//! [`SignalSetter`], which queues updates into an mpsc channel that the render
//! thread drains once per frame tick via `Signal::drain_setter_queue`.
//!
//! ## PartialEq on Memo
//!
//! `Memo<T>` requires `T: PartialEq`. The memo only notifies its dependents

pub mod clock;

## Open Questions

1. **Does `reactive_graph::owner::Owner::current()` exist?**
   - **What we know:** The docs.rs index lists `Owner::{new, new_root, set, with, with_cleanup, cleanup}`. `current()` was not explicitly listed but is idiomatic in Leptos.
   - **What's unclear:** The exact spelling for "get the current active owner, if any" — it may be a trait method on `Owner` or a free function.
   - **Recommendation:** Implementer verifies by grepping `reactive_graph 0.2.13` source at `reactive_graph/src/owner/mod.rs`. If `current()` doesn't exist, track our own "am I in an owner scope?" flag via a `thread_local!` counter in `runtime.rs` incremented/decremented by `Owner::with`.

2. **Does wrapping `ImmediateEffect::new_mut_scoped` in our `Effect::new` allow the user closure to be non-`Send+Sync`?**
   - **What we know:** `ImmediateEffect::new_mut_scoped` requires `FnMut() + Send + Sync + 'static`. `reactive_graph` ships a `send_wrapper_ext` module that re-exports `SendWrapper`.
   - **What's unclear:** Whether `SendWrapper<RefCell<FnMut>>` flows through `new_mut_scoped`'s bounds cleanly (`SendWrapper<T>` is `Send + Sync` by design, but `RefCell<dyn FnMut>` is not `Send`). Needs a compile check.
   - **Recommendation:** Wave 1 spikes this in `runtime::wrap_local_fnmut`. If it doesn't compile, the fallback is: require user closures to be `Send + Sync` explicitly (narrower API than CONTEXT.md §API surface). Document the decision in the phase exit review.

3. **Does `reactive_graph::effect::batch` also defer `Memo` recomputations, or just `ImmediateEffect` runs?**
   - **What we know:** The doc literally says "defers execution of `ImmediateEffect`s." Memos are pull-based (computed on read) in SolidJS-style graphs, so they may not need deferral — but a memo read inside an effect that runs during batch's drain could still see stale state.
   - **What's unclear:** Whether mid-batch reads via `m.get()` return the pre-batch value or the correct new value.
   - **Recommendation:** Add to the property-test suite: batch with multiple signals feeding a memo that an effect reads, assert final value consistency. Run during wave 2 and adjust if surprising.

4. **CONTEXT.md says `any_spawner` is a dev-dep; research recommends dropping it entirely.**
   - **What we know:** With `ImmediateEffect::new_mut_scoped`, no executor is needed for Phase 1.0 tests.
   - **What's unclear:** Whether the planner considers dropping `any_spawner` a CONTEXT.md change that requires user sign-off.
   - **Recommendation:** Flag as a decision point in the plan-check step. Either drop it (cleaner) or keep as unused workspace dep (faithful to CONTEXT.md). Both work.
